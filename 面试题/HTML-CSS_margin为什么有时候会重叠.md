# 题目
为什么 CSS 里的 `margin` 有时候会发生重叠（margin collapse）？哪些场景最容易出现？

# 回答要点
- margin 重叠通常发生在垂直方向的块级元素之间。
- 常见场景：
  - 相邻兄弟块级元素上下 margin
  - 父元素和第一个/最后一个子元素之间
  - 空块级元素自身上下 margin
- 重叠规则直觉理解：
  - 不是简单相加
  - 通常表现为取较大的那个值（复杂情况下还涉及正负值规则）
- 为什么会这样：
  - 这是 CSS 规范对块级垂直间距的布局处理方式
  - 目的是避免垂直间距层层相加过度膨胀
- 常见解决方式：
  - 创建 BFC
  - 增加边框/内边距
  - 改变布局方式（如 flex/grid）
- 面试里可补充：
  - 只在特定文档流块级场景下发生，不是所有 margin 都会重叠

# 追问
- 为什么左右 margin 不会像上下 margin 一样重叠？
- 父子 margin 重叠为什么会看起来像“把父元素顶开了”？
- `overflow: hidden` 或 `flow-root` 为什么能阻止 margin 重叠？
- flex 布局里为什么通常不会出现这类问题？

# 参考答案
CSS 里的 margin 重叠，通常指的是垂直方向上的 margin collapse。也就是说，两个块级元素在正常文档流里上下相邻时，它们的 `margin-top` 和 `margin-bottom` 有时不会像很多人直觉里那样直接相加，而是会发生合并。

最常见的场景有三类。

第一类是相邻兄弟元素。比如上一个块有 `margin-bottom`，下一个块有 `margin-top`，这两个垂直间距可能不会叠加，而是合并成一个。

第二类是父子元素。比如父元素没有边框、没有 padding，而第一个子元素有 `margin-top`，这个 margin 可能会“穿出来”，看起来像把父元素整体往下推了一样。

第三类是空块级元素本身。如果一个块没有内容、没有边框、没有 padding，它自己的上下 margin 也可能发生重叠。

为什么会这样？这是 CSS 规范在普通块级文档流中的一种垂直间距处理规则。你可以理解为浏览器为了避免块与块之间的垂直间距层层叠加得太夸张，所以在某些特定场景下选择合并处理。

要解决这个问题，常见方法包括：让元素形成 BFC，比如设置 `overflow: hidden` 或 `display: flow-root`；或者通过增加 `padding`、`border` 打断这种合并条件；再或者直接改用 Flex / Grid 布局，因为它们通常不会走这套传统块级 margin collapse 规则。

面试里如果总结，可以说：margin 重叠主要发生在普通文档流中的垂直块级布局里，尤其是兄弟块、父子块和空块；它不是 bug，而是 CSS 对垂直间距的一种规范性合并行为。

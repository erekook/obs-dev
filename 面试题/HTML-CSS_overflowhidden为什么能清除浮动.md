# 题目
为什么给父元素设置 `overflow: hidden`，很多时候就能“清除浮动”或解决父元素高度塌陷？

# 回答要点
- 本质原因不是“overflow 专门清除浮动”，而是它常会触发 BFC。
- 创建 BFC 后的典型效果：
  - 独立布局上下文
  - 计算高度时会包含内部浮动元素
- 所以父元素设置 `overflow: hidden/auto/scroll` 后：
  - 父元素高度能被浮动子元素撑开
  - 看起来就像“清除了浮动影响”
- 常见补充：
  - `overflow: hidden` 可能带来内容裁剪副作用
  - 更语义化的现代写法常用 `display: flow-root`
- 面试里可补充：
  - 这题本质是 BFC 机制题，不只是记忆技巧题

# 追问
- 为什么浮动元素会导致父元素高度塌陷？
- `overflow: hidden` 和 `clearfix` 的思路有什么不同？
- `display: flow-root` 为什么更推荐？
- 用 `overflow: hidden` 解决布局问题可能带来哪些坑？

# 参考答案
很多人会记住一个经验：“父元素加 `overflow: hidden` 可以清除浮动”，但更准确地说，它之所以能起作用，不是因为 `overflow` 天生就是清浮动属性，而是因为它通常会让这个元素形成一个 BFC（块级格式化上下文）。

浮动元素脱离了普通文档流，所以如果父元素内部全是浮动子元素，父元素在普通情况下计算高度时，可能不会把这些浮动子元素完整算进去，于是就会出现所谓“高度塌陷”。

而当父元素形成 BFC 后，它在布局和高度计算上会变成一个相对独立的区域。这个区域在计算高度时，会把内部浮动元素考虑进去，所以父元素就能被浮动子元素撑开。

这就是为什么给父元素加上 `overflow: hidden`、`auto` 或 `scroll` 后，浮动导致的高度塌陷问题经常就被“解决”了。表面看像是清除了浮动，实质上是 BFC 改变了父元素的布局计算方式。

不过 `overflow: hidden` 有一个常见副作用，就是它可能把超出边界的内容裁掉。所以在现代开发里，如果你只是单纯想创建一个新的格式化上下文，很多时候更推荐使用 `display: flow-root`，语义更明确，也不容易引入额外裁剪问题。

面试里如果总结，可以说：`overflow: hidden` 能“清除浮动”，本质是因为它触发了 BFC，让父元素在计算高度时把浮动子元素包含进去，而不是因为它直接消灭了浮动本身。

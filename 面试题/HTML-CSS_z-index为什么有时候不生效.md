# 题目
为什么给元素设置了 `z-index`，它有时候看起来还是“不生效”？常见原因有哪些？

# 回答要点
- `z-index` 不是“写了就一定能叠上去”，它依赖层叠上下文和定位条件。
- 常见原因：
  - 元素没有形成可参与 `z-index` 的定位环境
  - 不同层叠上下文之间不能直接按数字大小简单比较
  - 父元素层级本身更低
  - 某些属性创建了新的层叠上下文（如 `position` + `z-index`、`transform`、`opacity` 等）
- 关键认知：
  - `z-index` 比较通常发生在同一层叠上下文里
  - 子元素再大，也不一定能突破父上下文
- 面试里可补充：
  - 这类问题本质上常常不是“数值不够大”，而是“上下文不对”

# 追问
- 什么是层叠上下文？
- 为什么子元素 `z-index: 9999` 也可能盖不过外面的别的元素？
- `transform` 为什么常常让层级关系变得意外？
- 排查 z-index 问题时，应该优先看什么？

# 参考答案
很多人遇到 `z-index` 不生效时，第一反应是“数字不够大”，于是从 10 改到 9999，再改到 999999。但很多时候问题根本不在数字，而在层叠上下文。

`z-index` 不是一个全局无脑比大小的系统。它通常只在同一个层叠上下文里有直接可比性。如果两个元素处在不同的层叠上下文中，它们的层级关系首先受各自父级上下文影响，而不是单纯看谁的 `z-index` 数值更大。

这也是为什么一个子元素即使写了 `z-index: 9999`，也可能还是盖不过另一个外部元素——因为它所在的整个父上下文，本身就排在更低的一层。

另外，`z-index` 还和元素的定位环境有关。像 `position`、`transform`、`opacity` 等属性，在一些情况下会创建新的层叠上下文，导致你以为的层级比较关系被切开了。

所以遇到 `z-index` 不生效时，真正要排查的通常不是“数字够不够大”，而是：
1. 这个元素所在的父级有没有创建新的层叠上下文；
2. 目标元素和被遮挡元素是不是处在同一个上下文；
3. 父级本身的层级是不是已经被压在下面。

从实际经验看，很多 z-index 问题，最后都不是通过“继续加大数字”解决的，而是通过调整 DOM 结构、去掉不必要的上下文、或者把元素移到更合适的层级位置上解决的。

面试里如果总结，可以说：`z-index` 不生效，常见根因不是数值小，而是层叠上下文不同；理解“同一上下文里比较、父级上下文限制子级”的规则，比背数字更重要。

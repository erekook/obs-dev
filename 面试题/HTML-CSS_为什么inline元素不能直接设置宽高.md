# 题目
为什么很多 `inline` 元素默认不能直接设置宽高？它和 `block`、`inline-block` 在布局上的核心区别是什么？

# 回答要点
- `inline` 元素主要参与行内排版，不按独立盒子那样占一整行。
- 默认特性：
  - 宽高更多由内容和行高体系决定
  - `width` / `height` 往往不按我们期望生效
- `block`：
  - 独占一行
  - 更适合做结构布局
  - 可直接控制宽高
- `inline-block`：
  - 保留行内排列特性
  - 又能像盒子一样设置宽高
- 面试里可补充：
  - 这不是“浏览器限制你”，而是不同 display 模式的布局语义不同

# 追问
- 为什么 `padding` 和 `margin` 在 inline 元素上表现也会比较特殊？
- 行高为什么会影响 inline 元素的视觉高度？
- 什么时候适合用 inline-block，什么时候更适合 flex？
- `span` 改成 `display: block` 后本质发生了什么？

# 参考答案
很多 inline 元素默认不能像 block 元素那样直接设置宽高，根本原因不是“这个标签不支持”，而是它的布局语义就不是按独立盒子来参与页面排版的。

`inline` 元素主要是为了行内文本流设计的。也就是说，它更像一句话里的一个片段，跟文字一起在一行里排布。对于这类元素，浏览器更关注的是字体、行高、基线、内容本身，而不是把它当成一个独立占位的大盒子来控制。

所以你给很多 inline 元素写 `width`、`height`，往往不会得到和 block 元素一样的效果。它的可见尺寸更多是被内容本身和所在行的排版规则决定的。

而 `block` 元素则不同。它会独占一行，更像一个完整的布局盒子，所以宽高、外边距、内边距这些盒模型控制方式会更直接、更符合直觉。

`inline-block` 可以理解成两者之间的折中：它仍然能和其他元素排在同一行，但又具备像盒子一样设置宽高的能力。所以很多早期横向布局、按钮排布、标签项布局都会用 `inline-block`。

不过在现代前端里，如果是组件级布局，很多场景已经更倾向于用 Flex，因为它在对齐、间距和响应式控制上更自然。

面试里如果总结，可以说：inline 元素默认不适合直接控制宽高，是因为它属于行内排版模型，尺寸更多受内容和行高体系影响；而 block 和 inline-block 则更像明确的盒子结构，适合做宽高和布局控制。

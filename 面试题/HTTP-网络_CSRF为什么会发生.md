# 题目
为什么会发生 CSRF（跨站请求伪造）？它利用了浏览器的什么行为，常见防御手段有哪些？

# 回答要点
- CSRF 的本质：
  - 攻击者诱导已登录用户在不知情情况下发起请求
- 它能发生的关键条件：
  - 用户已在目标站点登录
  - 浏览器会自动携带该站点相关 Cookie
  - 服务端只凭 Cookie 就认可请求身份
- 利用了浏览器的行为：
  - 跨站请求时仍可能自动带上目标站点 Cookie
- 常见防御手段：
  - CSRF Token
  - SameSite Cookie
  - 校验 Origin / Referer
  - 关键操作二次确认
- 面试里可补充：
  - CSRF 不是“读数据”，更多是“冒用身份发请求”
  - 它和 XSS 关注点不同

# 追问
- 为什么说 Cookie 自动携带是 CSRF 的基础条件？
- SameSite 为什么能缓解 CSRF？
- 为什么只校验登录态不够？
- 如果系统不用 Cookie 改用 Token，CSRF 风险会怎样变化？

# 参考答案
CSRF 会发生，最核心的原因是：浏览器会自动帮用户携带目标站点的 Cookie，而服务端如果只依赖这个 Cookie 来识别身份，就可能把“用户自己发的请求”和“攻击页面诱导发出的请求”混为一谈。

一个典型场景是这样的：用户已经登录了银行网站，这时候浏览器里保存着该站点的登录 Cookie。用户随后又打开了一个恶意网站，这个网站偷偷构造了一个指向银行转账接口的请求。因为浏览器对目标站点仍会自动带上银行的 Cookie，所以请求到了银行服务器后，看起来就像是“这个已登录用户自己发起的”。

所以 CSRF 的本质不是盗取用户信息，而是利用用户现有登录状态，伪造一次“看起来合法”的操作请求。

它利用的关键浏览器行为，就是 Cookie 的自动携带机制。浏览器是为了方便会话维持而这样设计的，但如果服务端没有额外校验请求来源和合法性，这个机制就可能被滥用。

常见防御手段里，最经典的是 CSRF Token。也就是除了 Cookie 之外，请求里还必须带一个攻击者无法提前伪造的随机令牌，服务端校验通过才接受请求。

另外，SameSite Cookie 也是非常重要的防线。它可以限制浏览器在跨站场景下是否自动携带 Cookie，从源头上降低 CSRF 风险。服务端还可以校验 `Origin` 或 `Referer`，确认请求是不是来自可信页面。

对于转账、改密码这类关键操作，还可以加验证码、二次确认、短信确认等额外步骤，进一步降低风险。

面试里如果总结，可以说：CSRF 之所以能发生，是因为浏览器会自动带 Cookie，而服务端又只凭 Cookie 识别身份；防御关键就是增加“请求来源和意图的额外校验”，而不是只看用户是不是登录了。

# 题目
请你讲一下 Cookie、Session、Token 的区别。它们分别适合解决什么问题？在前后端分离项目里为什么很多场景更常见的是 Token 方案？

# 回答要点
- Cookie 是浏览器存储在本地的小型键值数据，会随同域请求自动携带。
- Session 通常是服务端保存的会话状态，客户端一般只保存 Session 标识。
- Token 是服务端签发给客户端的一段凭证，客户端后续请求时主动携带。
- 三者不是完全对立关系：
  - Cookie 更偏存储和传输载体
  - Session 更偏服务端会话机制
  - Token 更偏认证凭证方案
- Cookie + Session：
  - 服务端维护状态
  - 适合传统服务端渲染项目
  - 扩容时可能涉及共享 Session 问题
- Token：
  - 更适合前后端分离、移动端、多端接入
  - 服务端可更无状态
  - 常通过 `Authorization` 头传输
- 安全点：
  - Cookie 有 CSRF 风险
  - Token 如果存储不当会有 XSS 风险
  - 需要结合 HTTPS、过期时间、刷新机制等一起考虑

# 追问
- JWT 一定是无状态的吗？它有什么优缺点？
- Token 放 localStorage、sessionStorage、Cookie 各有什么风险？
- 如何防御 CSRF 和 XSS？
- 单点登录场景下，这三者通常如何配合？

# 参考答案
Cookie、Session、Token 都和登录认证有关，但它们并不是同一个层面的概念。

Cookie 本质上是浏览器端的一种存储机制，服务器可以通过响应头把一小段数据写到浏览器里，之后浏览器访问同域接口时会自动携带这些 Cookie。所以 Cookie 更像一种“保存并自动发送数据的容器”。

Session 通常是服务端的会话机制。用户登录后，服务端在内存、Redis 或数据库里保存一份登录状态，然后给客户端一个 Session ID。客户端下次请求时把这个 ID 带回来，服务端再根据它找到对应会话信息。很多传统 Web 项目会把 Session ID 放在 Cookie 里传输。

Token 则更像一种认证凭证。登录成功后，服务端把一个 Token 发给客户端，客户端后续请求时主动放到请求头里，比如 `Authorization: Bearer xxx`。服务端收到后校验 Token，确认用户身份。

在前后端分离项目里，Token 更常见，原因是它对多端接入更友好，Web、App、小程序都可以统一处理，而且服务端不一定要保存复杂会话状态，更适合分布式部署。

不过 Token 也不是没有问题。如果把 Token 存在 `localStorage`，一旦发生 XSS，可能被恶意脚本窃取；如果放在 Cookie 里自动发送，又要额外考虑 CSRF 风险。所以安全上要结合 HTTPS、过期时间、刷新机制、同站策略等一起设计。

总结来说，Cookie 是浏览器存储和传输机制，Session 是服务端会话方案，Token 是客户端携带的认证凭证。面试时如果能先把三者的层次区分清楚，再讲适用场景和安全差异，基本就比较完整了。

# 题目
请你讲一下 HTTP 里的 GET 和 POST 有什么区别。除了“GET 放 URL、POST 放 body”之外，还有哪些更本质的差异？

# 回答要点
- GET 和 POST 都是 HTTP 请求方法，但语义和使用场景不同。
- GET：
  - 更偏获取资源
  - 应该是安全方法（不修改资源）
  - 幂等
  - 更容易被缓存、收藏、分享
- POST：
  - 更偏提交数据、创建资源或触发处理
  - 通常不是幂等
  - 默认不如 GET 那样天然适合缓存
- 不能只说“GET 没有 body”或“POST 更安全”这类不严谨结论。
- 实际区别维度：
  - 语义
  - 幂等性
  - 缓存行为
  - 参数传递习惯
  - 浏览器历史记录和书签行为
- 面试里可补充：
  - HTTPS 下两者传输过程都可加密
  - 是否安全不能只看方法，还要看业务设计和传输环境

# 追问
- 什么是幂等？为什么 GET 通常要求幂等？
- POST 一定不能缓存吗？
- 如果一个接口用 GET 却会修改数据，会有什么问题？
- PUT 和 POST 的区别又是什么？

# 参考答案
GET 和 POST 的区别，最核心的不是“参数放哪”，而是它们在 HTTP 语义上的定位不同。

GET 通常表示“获取资源”。按照规范，它应该是安全的，也就是不应该对服务器资源产生副作用。同时，GET 还应该是幂等的，也就是说，多次请求相同资源，结果应该和一次请求一致，不应因为重复请求而改变服务器状态。

POST 通常表示“提交数据”或者“触发服务端进行某种处理”，比如创建资源、提交表单、上传内容等。它一般不要求幂等，所以同一个 POST 请求发两次，可能会创建两条记录，产生不同结果。

很多人会把区别简单理解成“GET 参数放 URL，POST 参数放 body”，但这只是常见用法，不是最本质的定义。真正重要的是语义、幂等性和缓存行为。

从缓存角度看，GET 更容易被浏览器和中间缓存利用，也更适合做书签、分享链接，因为参数直接体现在 URL 上。POST 默认不如 GET 那样天然适合这些场景。

安全性上也有一个常见误区，就是“POST 比 GET 更安全”。其实如果不用 HTTPS，GET 和 POST 的内容在传输过程中都可能被看到；如果用了 HTTPS，它们都可以被加密。所以安全与否主要取决于传输层和业务设计，而不是单纯取决于方法名。

面试里如果总结，可以说：GET 更强调读取、幂等和可缓存，POST 更强调提交、处理和副作用；参数位置只是表象，语义才是核心。

# 题目
请你讲一下 HTTP/1.1、HTTP/2、HTTP/3 的主要区别。它们分别解决了什么问题？

# 回答要点
- 三者都是 HTTP 协议演进版本，优化重点逐步从文本协议到多路复用，再到传输层改进。
- HTTP/1.1 特点：
  - 文本协议
  - 默认长连接
  - 管线化理论存在但实践少
  - 存在队头阻塞问题
  - 通常需要雪碧图、资源合并、域名分片等优化手段
- HTTP/2 特点：
  - 二进制分帧
  - 多路复用
  - 头部压缩
  - 服务器推送（现已较少作为重点）
  - 减少了 HTTP 层的队头阻塞
- HTTP/3 特点：
  - 基于 QUIC，底层使用 UDP
  - 把传输层握手和可靠性机制做了升级
  - 更好解决 TCP 层队头阻塞
  - 连接迁移能力更强，弱网场景表现更好
- 面试里可补充：
  - HTTP/2 并没有彻底解决所有队头阻塞，只是缓解了应用层问题
  - HTTP/3 重点在传输层优化

# 追问
- 为什么 HTTP/2 多路复用了，仍然会有队头阻塞问题？
- QUIC 为什么要基于 UDP 实现？
- 服务器推送为什么后来不算 HTTP/2 的核心卖点了？
- 在现代前端优化里，HTTP/2 时代为什么不再那么强调资源合并？

# 参考答案
HTTP/1.1、HTTP/2、HTTP/3 的区别，可以按“协议组织方式”和“解决的瓶颈”来理解。

HTTP/1.1 是最经典、最常见的版本。它本质上还是文本协议，虽然支持长连接，减少了频繁建立 TCP 连接的成本，但同一个连接上的请求处理能力有限，容易出现队头阻塞问题。为了弥补这些问题，前端早期常常会做资源合并、雪碧图、域名分片等优化。

HTTP/2 的核心升级是二进制分帧和多路复用。它允许多个请求和响应在同一个连接里并发传输，不再像 HTTP/1.1 那样一个请求卡住后面一串请求。另外它还引入了头部压缩，减少重复请求头带来的开销。

不过 HTTP/2 虽然解决了 HTTP 层面很多问题，但它底层仍然基于 TCP。只要 TCP 某个包丢失，后续数据流在传输层还是会受到影响，这也是为什么说 HTTP/2 没有彻底消除所有队头阻塞。

HTTP/3 的关键变化是它不再基于 TCP，而是基于 QUIC，而 QUIC 运行在 UDP 之上。这样做的好处是协议设计更灵活，可以把连接建立、拥塞控制、重传机制等能力做得更现代化，也更容易减少 TCP 层面的阻塞影响。在弱网、移动网络切换等场景下，HTTP/3 往往表现更好。

如果总结每个版本的重点：HTTP/1.1 解决基本传输问题但并发效率有限；HTTP/2 重点优化应用层并发和传输效率；HTTP/3 则进一步把优化深入到传输层，重点解决 TCP 带来的瓶颈。

面试里如果用一句话概括，可以说：HTTP/1.1 是文本时代，HTTP/2 是多路复用时代，HTTP/3 是基于 QUIC 的传输层升级时代。

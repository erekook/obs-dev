# 题目
为什么说 HTTPS 比 HTTP 更安全？它主要解决了哪些安全问题？

# 回答要点
- HTTP 本身是明文传输，数据容易被窃听和篡改。
- HTTPS = HTTP + TLS/SSL。
- HTTPS 主要解决三类问题：
  - 机密性：数据加密，降低被窃听风险
  - 完整性：防止传输途中被篡改
  - 身份认证：通过证书校验服务器身份，降低假冒网站风险
- 常见过程：
  - TLS 握手
  - 证书校验
  - 密钥协商
  - 后续使用会话密钥加密传输
- 面试里可补充：
  - HTTPS 也不是绝对安全，还要结合证书管理、服务端安全、前端安全策略一起看
  - 性能开销相比以前已小很多

# 追问
- TLS 握手大致做了什么？
- 为什么 HTTPS 不能防所有 XSS/CSRF？
- 对称加密和非对称加密在 HTTPS 中分别扮演什么角色？
- 如果证书失效或被中间人替换，会发生什么？

# 参考答案
之所以说 HTTPS 比 HTTP 更安全，核心原因是 HTTP 默认是明文传输，而 HTTPS 在 HTTP 外面增加了一层 TLS 安全协议。

HTTP 明文传输意味着，数据在网络中经过的任何节点，如果有能力拦截，就可能看到里面的内容，比如账号密码、Cookie、接口返回数据等。更糟的是，攻击者还可能篡改传输内容，比如插入恶意脚本、替换下载资源、伪造页面。

HTTPS 主要解决三个问题。

第一是机密性。通过加密，别人即使截获了传输内容，也很难直接读懂。

第二是完整性。HTTPS 会校验数据在传输过程中有没有被改动，如果中间人偷偷篡改内容，通信双方能发现异常。

第三是身份认证。浏览器会校验服务器证书，确认你访问的确实是目标网站，而不是一个冒充它的假服务器。这能显著降低中间人攻击和钓鱼伪装的风险。

从过程上看，浏览器和服务器在建立 HTTPS 连接时，会先进行 TLS 握手。握手过程中会验证证书、协商加密算法，并生成后续通信所需的会话密钥。之后真正的数据传输一般使用效率更高的对称加密完成。

不过 HTTPS 也不是万能的。它主要保护的是“传输链路上的安全”，并不能直接解决网站本身的业务漏洞，比如 XSS、CSRF、弱密码、服务端权限问题等。所以它是基础安全能力，但不是全部安全能力。

面试里如果总结，可以说：HTTPS 比 HTTP 更安全，是因为它通过 TLS 提供了加密、完整性校验和身份认证，让数据在传输过程中更不容易被偷看、篡改或伪造。

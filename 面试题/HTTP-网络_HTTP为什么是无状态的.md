# 题目
为什么说 HTTP 是无状态协议？这种“无状态”具体指什么，它带来了什么影响？

# 回答要点
- HTTP 无状态的含义：
  - 协议本身不自动记住前一次请求的上下文
  - 每次请求默认都是独立的
- 这不等于：
  - 网站不能登录
  - 前后端不能维护会话
- 影响：
  - 协议简单、易扩展
  - 服务器处理单次请求更独立
  - 但需要额外机制维持用户状态
- 常见状态维持方式：
  - Cookie
  - Session
  - Token
- 面试里可补充：
  - “无状态”是协议层特性，不等于业务层不能有状态
  - REST 风格设计也强调无状态交互

# 追问
- 如果 HTTP 是无状态的，为什么网站还能记住用户登录？
- 无状态为什么有利于扩展和负载均衡？
- Session 是不是让 HTTP 变成有状态了？
- Token 方案为什么常被说更适合分布式？

# 参考答案
说 HTTP 是无状态协议，意思并不是“HTTP 不能做登录”或者“服务器完全不能记住用户”，而是说：HTTP 协议本身不会自动保存请求与请求之间的上下文关系。

也就是说，客户端发来一个请求，服务器处理完并返回结果后，从协议层看，这次交互就结束了。下一次请求再来时，HTTP 本身并不会天然知道“这还是不是刚才那个用户”、“上一次做到了哪一步”。

这种设计的好处是简单。服务器在处理单个请求时，不需要协议层自动维护复杂会话上下文，请求之间天然更独立，这对扩展、缓存、负载均衡都更友好。

但它也带来一个现实问题：很多业务明明需要状态，比如登录态、购物车、下单流程、权限判断。为了解决这个问题，业务层就会额外引入状态维持机制，比如 Cookie + Session，或者 Token。

比如用户登录成功后，服务端给浏览器一个 Cookie，后续请求浏览器自动带上它，服务端据此识别用户。这不是 HTTP 协议自己“记住了”用户，而是应用层额外建立了一套会话识别机制。

所以“HTTP 无状态”应该理解成：协议不自动持久记忆请求上下文，但应用可以在它之上构建有状态业务。

面试里如果总结，可以说：HTTP 无状态是指每次请求在协议层默认彼此独立，服务器不会自动记住上一次请求；它带来了简洁和易扩展性，但业务上通常需要借助 Cookie、Session、Token 等机制来补足状态管理。

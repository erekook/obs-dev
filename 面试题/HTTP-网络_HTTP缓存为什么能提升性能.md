# 题目
为什么 HTTP 缓存能显著提升前端性能？它优化的到底是哪几个环节？

# 回答要点
- HTTP 缓存的核心是复用已有资源，减少不必要的网络请求。
- 主要优化点：
  - 减少网络传输时间
  - 降低服务器压力
  - 减少带宽消耗
  - 提升页面加载速度和响应速度
- 命中强缓存：
  - 连请求都不用发
  - 直接从本地缓存读取
- 命中协商缓存：
  - 请求体不用重新下载
  - 只做轻量确认，返回 304
- 对用户体验的影响：
  - 页面打开更快
  - 来回切换页面更顺畅
  - 静态资源复用率更高
- 面试里可补充：
  - 缓存并不是越久越好，关键是更新策略设计
  - 通常配合 hash 资源和 HTML 协商缓存一起用

# 追问
- 强缓存和协商缓存在性能收益上差别大吗？
- 为什么缓存不仅省时间，也省服务器资源？
- 如果缓存策略配错了，会带来什么问题？
- 为什么 HTML 通常不像静态资源那样长期强缓存？

# 参考答案
HTTP 缓存能显著提升前端性能，核心原因很简单，就是把“已经拿过的资源”复用起来，避免每次都重新走完整网络请求和下载流程。

如果没有缓存，浏览器每次访问页面都得重新去服务器请求 HTML、JS、CSS、图片等资源。这个过程不仅有网络往返时间，还有服务器处理、数据传输和浏览器再次接收解析的成本。

而有了缓存之后，性能优化主要发生在两个层面。

第一层是强缓存。如果资源还在有效期内，浏览器连请求都不用发，直接从本地内存或磁盘里读取资源。这种收益最大，因为网络往返和服务器处理都完全省掉了。

第二层是协商缓存。即使浏览器需要去问一下服务器资源有没有变，也不一定要重新下载整个文件。服务端只要返回 `304 Not Modified`，浏览器就继续使用本地副本。这虽然比强缓存多了一次请求，但相比重新下载完整资源，仍然能节省大量流量和时间。

所以 HTTP 缓存优化的不只是“下载速度”，它还同时优化了网络传输、服务端处理压力、带宽占用和用户等待时间。对用户来说，最直观的感受就是页面打开更快，来回切换页面时也更顺畅。

当然，缓存也不是越激进越好。如果策略设计不当，用户可能一直拿到旧资源，出现“代码已经更新但页面没变”的问题。所以真正好的缓存策略，通常是：带 hash 的静态资源长期缓存，HTML 文件短缓存或协商缓存，确保性能和更新都兼顾。

面试里如果总结，可以说：HTTP 缓存通过减少甚至消除重复网络请求，优化了请求时间、传输成本和服务器压力，因此是前端性能优化里最基础也最有效的手段之一。

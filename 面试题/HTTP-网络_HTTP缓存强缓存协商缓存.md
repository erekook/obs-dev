# 题目
请你讲一下 HTTP 缓存。什么是强缓存和协商缓存？它们分别依赖哪些响应头，命中后浏览器的表现有什么区别？

# 回答要点
- HTTP 缓存的目标是减少重复请求、降低延迟、减轻服务器压力。
- 强缓存：浏览器直接使用本地缓存，不向服务器发起请求。
- 协商缓存：浏览器会带上缓存标识向服务器确认资源是否变化，由服务器决定是否可继续使用缓存。
- 强缓存常见响应头：
  - `Cache-Control`
  - `Expires`
- 协商缓存常见响应头：
  - `ETag` / `If-None-Match`
  - `Last-Modified` / `If-Modified-Since`
- 命中表现：
  - 强缓存命中：通常直接从磁盘或内存读取，状态可能显示为 `from memory cache` / `from disk cache`
  - 协商缓存命中：会发请求，服务器返回 `304 Not Modified`
- 优先级：
  - `Cache-Control` 优先于 `Expires`
  - `ETag` 通常比 `Last-Modified` 更精确
- 常见策略：
  - 静态资源配合文件 hash，使用长期强缓存
  - HTML 文档通常不做强缓存，更多依赖协商缓存或 no-cache

# 追问
- `no-cache` 和 `no-store` 的区别是什么？
- 为什么带 hash 的静态资源适合长期缓存？
- `ETag` 和 `Last-Modified` 各自有什么优缺点？
- CDN 缓存和浏览器缓存有什么不同？

# 参考答案
HTTP 缓存的核心目的是复用已经获取过的资源，减少不必要的网络传输。前端性能优化里，它是最基础也最重要的一部分。

强缓存是指浏览器在缓存未过期时，直接使用本地缓存副本，不会向服务器发送请求。它主要依赖 `Cache-Control` 和 `Expires`。其中 `Cache-Control` 更常用，也更精确，比如 `max-age=3600` 表示资源在 3600 秒内都可以直接使用。`Expires` 是一个绝对时间，受客户端本地时间影响，所以优先级低于 `Cache-Control`。

协商缓存是指浏览器本地虽然有缓存，但不能直接确定是否还能继续使用，于是会发一个请求到服务器进行确认。如果资源没变，服务器返回 `304 Not Modified`，浏览器继续使用本地缓存；如果资源变了，服务器返回新的资源内容。协商缓存主要依赖两组头：`Last-Modified` 搭配 `If-Modified-Since`，以及 `ETag` 搭配 `If-None-Match`。

`Last-Modified` 记录资源最后修改时间，实现简单，但精度通常只到秒，而且如果文件内容没变但修改时间变了，也可能造成误判。`ETag` 是资源内容的唯一标识，通常更准确，但生成成本可能略高。

在实际项目里，最常见的做法是：JS、CSS、图片这类带内容 hash 的静态资源使用长期强缓存，因为文件名一旦变化就说明内容变了；而 HTML 文件通常不做长期强缓存，否则用户可能拿不到最新入口文件，所以更适合用 `no-cache` 配合协商缓存。

面试里如果想回答得更完整，可以再补一句：强缓存是“连请求都不发”，协商缓存是“发请求问一下还能不能用”，这是两者最本质的区别。

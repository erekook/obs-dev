# 题目
请你讲一下 WebSocket 和 HTTP 有什么区别。它为什么适合做实时通信场景？

# 回答要点
- HTTP 和 WebSocket 都基于应用层通信，但交互模型不同。
- HTTP 特点：
  - 请求-响应模型
  - 一般由客户端主动发起
  - 服务端被动返回结果
- WebSocket 特点：
  - 在 HTTP 握手基础上升级协议
  - 建立连接后可全双工通信
  - 客户端和服务端都可主动发消息
  - 适合低延迟实时推送
- 为什么适合实时场景：
  - 避免频繁轮询
  - 减少重复请求头开销
  - 服务端主动推送更及时
- 常见场景：
  - 聊天室
  - 实时报价
  - 在线协作
  - 消息通知
  - 游戏状态同步
- 面试里可补充：
  - WebSocket 不是用来替代所有 HTTP 请求的
  - 普通 CRUD 场景仍然常用 HTTP

# 追问
- WebSocket 建连前为什么还要先走 HTTP 握手？
- 长轮询和 WebSocket 的区别是什么？
- WebSocket 断线重连一般怎么设计？
- SSE 和 WebSocket 又有什么差异？

# 参考答案
HTTP 和 WebSocket 的核心区别，在于通信模式不同。

HTTP 是典型的请求-响应模型。客户端发起请求，服务端返回结果，一次请求响应结束后，这次交互基本也就结束了。虽然现代 HTTP 有长连接和复用机制，但从语义上看，它依然更适合“客户端问，服务端答”这种模式。

WebSocket 则不一样。它通常先通过一次 HTTP 请求完成握手，然后把协议升级成 WebSocket。连接建立之后，客户端和服务端之间会保持一条持续连接，双方都可以主动向对方发送消息，这就是所谓的全双工通信。

也正因为这样，WebSocket 特别适合实时性要求高的场景。比如聊天室、在线协作文档、股票行情、即时通知、多人游戏等。如果这些场景还用普通 HTTP 轮询实现，就会有很多无效请求和延迟问题。

相比轮询，WebSocket 的优势在于不需要客户端一直反复问“有没有新消息”，而是服务端一有新数据就可以主动推送过来。同时连接建立后也不需要每次都重复带完整请求头，所以在高频消息场景里通常更高效。

不过 WebSocket 也不是用来取代所有 HTTP 的。像页面初始化、获取详情、提交表单、普通 CRUD 接口，这些场景 HTTP 依然更简单直接。WebSocket 更适合持续连接、双向通信、低延迟推送的需求。

面试里如果总结，可以说：HTTP 更适合请求-响应型接口，WebSocket 更适合长连接、双向实时通信；前者擅长常规业务请求，后者擅长实时消息场景。

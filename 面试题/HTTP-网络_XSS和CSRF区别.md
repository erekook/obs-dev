# 题目
请你讲一下 XSS 和 CSRF 的区别。它们都和前端安全有关，但攻击方式和防御思路有什么不同？

# 回答要点
- 两者都是常见 Web 安全问题，但本质不同。
- XSS：
  - 本质是把恶意脚本注入页面并执行
  - 重点是“浏览器执行了不该执行的脚本”
- CSRF：
  - 本质是冒用已登录用户身份发起请求
  - 重点是“浏览器带着合法身份去发了不该发的请求”
- 核心区别：
  - XSS 攻击代码执行
  - CSRF 攻击请求伪造
- 常见防御：
  - XSS：转义、CSP、输入输出过滤、避免危险 API
  - CSRF：Token、SameSite、Origin/Referer 校验
- 面试里可补充：
  - 两者可能联动，XSS 甚至可能绕过部分 CSRF 防护

# 追问
- 为什么说 XSS 的危害通常更广？
- CSRF 为什么依赖浏览器自动带 Cookie？
- CSP 在防 XSS 里起什么作用？
- 如果一个系统已经不用 Cookie，只用 Authorization Token，CSRF 风险会怎样变化？

# 参考答案
XSS 和 CSRF 都是 Web 安全里的高频问题，但它们攻击的方式完全不同，所以防御思路也不一样。

XSS（跨站脚本攻击）的核心，是攻击者把恶意脚本注入到页面里，让浏览器把它当成正常页面代码执行。也就是说，问题的重点在于“页面执行了不该执行的脚本”。一旦成功，攻击者可能读取页面数据、窃取 token、篡改页面、冒充用户做很多事。

CSRF（跨站请求伪造）的核心，则不是执行脚本，而是冒用用户当前的登录身份，诱导浏览器向目标站点发送一个看起来合法的请求。它利用的是浏览器自动携带 Cookie 的机制，所以重点在于“浏览器带着用户身份发了不该发的请求”。

所以可以把区别记成一句话：XSS 是“脚本被执行了”，CSRF 是“请求被伪造了”。

防御方式也因此不同。XSS 更强调输入输出安全，比如对用户输入做转义、避免直接拼接 HTML、启用 CSP、减少危险 API 使用。CSRF 更强调请求来源校验和身份校验，比如加 CSRF Token、使用 SameSite Cookie、校验 Origin/Referer。

面试里一个加分点是提到：XSS 和 CSRF 并不是完全独立的。有些情况下，一旦页面已经被 XSS 攻破，攻击者甚至能直接读取或构造原本用于防御 CSRF 的数据，所以很多团队会认为 XSS 的整体危害范围更大。

面试里如果总结，可以说：XSS 攻击的是浏览器脚本执行环境，CSRF 攻击的是浏览器自动携带身份发请求的机制；前者重点防注入执行，后者重点防伪造请求。

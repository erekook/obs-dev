# 题目
为什么“幂等性”这个听起来像后端的概念，前端也必须关心？前端场景里它通常体现在哪些地方？

# 回答要点
- 幂等性的核心：
  - 同一个操作重复执行，多次结果应与一次结果一致或可控
- 前端为什么要关心：
  - 用户可能重复点击
  - 网络重试可能重复发请求
  - 页面刷新/回退可能再次触发操作
  - 异步回调可能重复落地状态
- 常见前端场景：
  - 表单重复提交
  - 支付/下单按钮连点
  - 请求失败自动重试
  - 列表加载更多重复触发
- 常见前端手段：
  - 按钮防重复点击
  - 请求去重
  - 幂等 key
  - 状态机控制流程只进入一次
- 面试里可补充：
  - 幂等性不是只靠后端，前后端要一起兜底

# 追问
- 防抖和幂等性是一回事吗？
- 为什么支付类操作对幂等性要求特别高？
- 如果前端已经禁用按钮，为什么后端仍然要做幂等保护？
- 请求重试和幂等设计之间是什么关系？

# 参考答案
很多人一听“幂等性”，会觉得这是后端或者接口设计的话题，但前端其实同样必须关心。因为前端正是用户操作和网络请求的第一入口，很多重复操作问题最早就是从前端发生的。

幂等性的核心可以简单理解成：同一个操作重复执行多次，结果应该和执行一次一样，或者至少是可控的，不会造成额外副作用。

前端为什么要关心？因为真实用户不会像测试脚本那样“点一次就等”。他们会连点按钮、刷新页面、回退重试；网络层也可能因为超时、弱网、自动重试导致同一个请求被多次发送。如果前端对这些情况完全不设防，就很容易出现重复下单、重复提交、列表重复插入、状态错乱等问题。

典型场景包括：提交表单时用户连续点两次提交；支付按钮因为页面卡顿被多次点击；加载更多在滚动过程中被重复触发；请求失败后自动重试又把之前的结果重复落库。

所以前端通常也要做一层幂等保护，比如提交后立即禁用按钮、对同类请求做去重、给关键操作带唯一标识、用状态机控制某个流程只能进入一次等。

当然，前端做这些并不能替代后端幂等。因为前端再谨慎，也挡不住刷新、抓包重放、网络重试、多个端同时操作等情况。所以更合理的理解是：前端负责尽量减少重复触发，后端负责最终兜底保证结果正确。

面试里如果总结，可以说：幂等性虽然常被当成后端概念，但前端同样要关心，因为重复点击、弱网重试和页面重入都可能让同一操作被执行多次；前端和后端应该一起保证关键流程结果可控。

# 题目
为什么前端打包后的静态资源文件名常常要带 hash？它主要解决了什么问题？

# 回答要点
- 给静态资源加 hash，核心是为了解决缓存更新问题。
- 典型场景：
  - JS/CSS 做了强缓存
  - 文件内容更新后，希望用户一定拿到新版本
- 原理：
  - 内容变了，文件名里的 hash 也变
  - 浏览器会把它当作一个新资源重新请求
- 带来的好处：
  - 可放心对静态资源设置长期缓存
  - 避免用户拿到旧文件
  - 减少手动清缓存成本
- 常见搭配：
  - HTML 不做长时间强缓存
  - JS/CSS/图片等静态资源带 hash + 长缓存
- 面试里可补充：
  - 这是一种“缓存失效设计”
  - hash 通常基于内容，而不是随机值

# 追问
- 为什么 HTML 通常不适合像静态资源那样长期强缓存？
- query 参数版本号和文件名 hash 有什么差异？
- 如果只改了一张图片，为什么只想让那张图的 URL 变化？
- hash 是构建时生成的还是运行时生成的？

# 参考答案
静态资源文件名带 hash，最主要是为了让缓存既能“用得久”，又能“更新得准”。

前端项目里，JS、CSS、图片这类静态资源通常希望尽可能缓存得久一点。因为它们下载一次后，如果后续版本没变，就没必要每次都重新请求，这样能明显提升加载速度。

但问题在于，如果资源内容更新了，而 URL 还完全一样，浏览器就可能继续使用旧缓存，导致用户拿到过期的脚本或样式。这样就容易出现“代码明明发版了，用户页面却没更新”的情况。

给文件名加 hash，就是为了解决这个问题。通常这个 hash 是根据文件内容生成的。只要内容变了，hash 就会变，文件名也随之变化。浏览器看到新的 URL，就会把它当成一个全新的资源重新请求。

这样一来，没变的资源可以继续长期缓存，变了的资源又能精准失效，不需要让所有缓存全部作废。比如只改了一行 CSS，就只会让对应 CSS 文件的 hash 变化，其他没改的资源仍然可以复用旧缓存。

实际工程里，通常会搭配一种常见策略：HTML 文件本身缓存时间较短，或者走协商缓存；而 JS、CSS、图片等带 hash 的资源则设置较长的强缓存。因为 HTML 负责引用最新的资源入口，一旦 HTML 更新，浏览器就能拿到新的资源地址。

面试里如果总结，可以说：静态资源带 hash，本质上是通过“内容变了就换 URL”的方式实现精准缓存失效，让长期缓存和版本更新两件事同时成立。

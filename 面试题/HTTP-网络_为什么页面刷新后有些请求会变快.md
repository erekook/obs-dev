# 题目
为什么一个页面第一次打开时请求很慢，但刷新之后很多资源和接口看起来会快很多？

# 回答要点
- 刷新后变快，通常与缓存和连接复用有关。
- 常见原因：
  - 浏览器缓存命中（内存 / 磁盘 / HTTP 缓存）
  - DNS 缓存
  - TCP / TLS 连接复用
  - 代码和数据已在内存中部分准备好
- 静态资源常见情况：
  - 第二次请求直接命中强缓存
  - 或协商缓存返回 304
- 接口层面也可能更快：
  - 连接复用
  - 服务端缓存命中
  - 浏览器预热和本地状态已建立
- 面试里可补充：
  - “刷新变快”不一定说明接口优化了，可能只是链路复用了已有状态
  - 普通刷新、强制刷新表现可能不同

# 追问
- 为什么强缓存命中后速度提升最明显？
- DNS 和 TLS 连接复用为什么也会影响二次访问速度？
- 强制刷新为什么有时又会变慢？
- 接口层和静态资源层的“变快”原因有什么不同？

# 参考答案
页面第一次打开慢、刷新后变快，是非常常见的现象。它通常不代表页面“突然优化了”，而是因为浏览器和网络链路已经把很多准备工作做过一遍了。

最直接的原因是缓存。第一次打开页面时，JS、CSS、图片等静态资源都要完整下载；等资源拿到后，浏览器会把它们缓存在内存、磁盘或通过 HTTP 缓存机制保留下来。刷新页面时，如果命中了强缓存，浏览器甚至连请求都不用发，直接本地读取；即使没命中强缓存，也可能只需要发一个轻量确认请求，拿到 304 后继续用本地资源。

第二个原因是连接复用。第一次访问时，浏览器通常要做 DNS 解析、TCP 建连、TLS 握手等准备工作；后续刷新时，这些结果可能还在缓存或复用窗口内，所以再次请求会省掉一部分前置耗时。

第三个原因是服务端和应用状态。比如某些接口第一次请求时服务端还没命中缓存，第二次可能已经有缓存结果；或者前端本地已经建立了一些上下文状态，再次访问时不需要完全从零开始。

所以“刷新后变快”通常是浏览器缓存、连接复用和链路预热共同作用的结果。它说明的是“重复访问成本降低了”，而不一定说明资源本身更小、接口本身更快了。

另外，不同刷新方式也会有差异。普通刷新通常会尽量利用现有缓存，而强制刷新则可能跳过部分缓存，因此看起来又会慢一些。

面试里如果总结，可以说：页面刷新后变快，主要是因为浏览器和网络链路复用了缓存、DNS、连接和部分运行状态，本质上是“已有准备结果被重复利用”，而不是页面突然换了一套更快的实现。

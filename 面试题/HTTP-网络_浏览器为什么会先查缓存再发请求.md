# 题目
浏览器为什么会先查缓存，再决定要不要发网络请求？这个过程对页面性能有什么意义？

# 回答要点
- 浏览器优先查缓存的核心目的：
  - 减少不必要的网络请求
  - 提升加载速度
  - 降低服务器和带宽压力
- 常见缓存层：
  - 内存缓存
  - 磁盘缓存
  - HTTP 强缓存/协商缓存机制
- 浏览器流程大致：
  - 先看本地是否已有可用资源副本
  - 如果强缓存有效，直接使用
  - 如果需要协商，就带标识去服务端确认
  - 确认未变则继续使用本地资源
- 对性能的意义：
  - 少一次请求或少一次完整响应下载
  - 缩短首屏和二次访问时间
  - 提升页面切换和重复访问体验
- 面试里可补充：
  - 缓存查询本身也是浏览器加载链路的重要一环
  - 性能优化常常靠“少请求”和“少传输”同时实现

# 追问
- 为什么强缓存命中收益比协商缓存更高？
- 浏览器为什么不能每次都直接问服务器最新版本？
- 缓存查找失败后还会经历哪些网络步骤？
- 如果资源更新了，缓存策略应该怎么配合？

# 参考答案
浏览器之所以会先查缓存，再决定要不要发请求，本质上是为了避免每次都把已经拿过、而且没变的资源重新走一遍完整网络流程。

一次网络请求不只是“发出去再拿回来”这么简单，里面通常还会涉及 DNS、建连、TLS、服务器处理、响应传输等多个步骤。即使资源本身很小，这些额外开销也是真实存在的。

所以浏览器会先检查本地缓存，看这个资源之前有没有拿过，以及当前是否还能继续使用。如果命中了强缓存，就连请求都不用发，直接从本地读取，这对性能帮助最大。

如果强缓存已经过期，但浏览器还有一份旧副本，就可能进入协商缓存流程，也就是带上资源标识去问服务端“这个资源有没有变”。如果服务端返回没变，浏览器仍然继续使用本地那份资源，只不过多了一次确认请求。

这样做的意义很大。第一，用户重复访问页面时会更快；第二，页面内多个静态资源不用每次都完整下载；第三，服务器和带宽压力也会显著降低。

所以浏览器“先查缓存”其实不是一个小细节，而是整个页面加载优化链路里非常重要的一环。前端性能里很多看起来很快的页面，本质上就是因为浏览器大量复用了已经缓存好的资源。

面试里如果总结，可以说：浏览器先查缓存，是为了优先复用已有资源副本，尽量把“网络请求”降级成“本地读取”或“轻量确认”，从而显著减少加载时间和传输成本。

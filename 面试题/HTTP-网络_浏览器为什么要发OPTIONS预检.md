# 题目
浏览器为什么有时候会先发一个 `OPTIONS` 预检请求？它的作用是什么，什么情况下会触发？

# 回答要点
- `OPTIONS` 预检通常和 CORS 跨域请求有关。
- 它的核心作用：
  - 浏览器先问服务端“这个跨域请求你允不允许”
  - 避免直接发送可能有风险的跨域请求
- 常见触发条件：
  - 请求方法不是简单方法（如 PUT、DELETE）
  - 带了某些自定义请求头
  - `Content-Type` 不是简单类型（如 `application/json` 常会触发）
- 预检过程：
  - 浏览器先发 `OPTIONS`
  - 服务端返回允许的方法、头、源等信息
  - 浏览器确认通过后才发真正请求
- 面试里可补充：
  - 简单请求不一定触发预检
  - 预检本质上是浏览器的安全机制，不是后端“多发一次”

# 追问
- 什么是简单请求？
- 为什么 `application/json` 往往会触发预检？
- 服务端需要返回哪些 CORS 头来通过预检？
- 预检请求能否缓存？

# 参考答案
浏览器有时候会先发一个 `OPTIONS` 请求，通常是因为当前跨域请求不属于“简单请求”，所以浏览器需要先做一次预检，确认服务端是否允许这次跨域操作。

它的核心作用可以理解成一句话：浏览器先替前端问一句，“我要发这个跨域请求，方法、请求头这些条件你接不接受？”如果服务端明确表示允许，浏览器才会继续发送真正的业务请求。

这样设计的原因是安全。因为有些跨域请求可能比普通 GET/POST 更敏感，比如带了自定义请求头、使用了 PUT/DELETE，或者提交的是某些特殊格式的数据。如果浏览器不先确认就直接发出去，风险会更高。

常见会触发预检的情况包括：请求方法不是简单方法，比如 PUT、DELETE；请求头里带了自定义头；或者 `Content-Type` 不是简单类型，比如很多前端常用的 `application/json`。

预检流程通常是这样的：浏览器先发一个 `OPTIONS` 请求，请求里会带上源信息、打算使用的方法、打算带的请求头；服务端如果允许，就在响应头里返回对应的 CORS 许可信息，比如允许哪些源、哪些方法、哪些头。浏览器检查通过后，才真正发送业务请求。

所以这个 `OPTIONS` 请求不是“多余的一次请求”，而是浏览器为了跨域安全自动增加的一道检查流程。

面试里如果总结，可以说：`OPTIONS` 预检是浏览器在某些非简单跨域请求前，先向服务端确认权限的一步安全检查，目的是在真正发请求前完成跨域能力验证。

# 题目
请你讲一下什么是跨域。为什么浏览器会有跨域限制？前端常见的跨域解决方案有哪些？

# 回答要点
- 跨域本质上是浏览器的同源策略限制。
- 同源的判断：
  - 协议相同
  - 域名相同
  - 端口相同
  - 任意一个不同都可能构成跨域
- 为什么有跨域限制：
  - 浏览器安全机制
  - 防止恶意网站读取其他站点敏感数据
- 常见解决方案：
  - CORS
  - 反向代理 / 开发代理
  - JSONP（了解，较老）
  - `postMessage`（跨窗口通信）
  - Nginx 网关统一转发
- CORS 重点：
  - 本质是服务端通过响应头授权
  - 简单请求与预检请求
  - 常见头如 `Access-Control-Allow-Origin`
- 面试里可补充：
  - 跨域限制主要发生在浏览器中
  - 服务端之间通信通常没有这个限制

# 追问
- 什么是简单请求和预检请求？
- 为什么带上自定义头或某些 Content-Type 会触发 OPTIONS 预检？
- CORS 为什么不能随便配 `*`？
- 开发环境代理和生产环境网关方案各自有什么优缺点？

# 参考答案
跨域本质上是浏览器基于同源策略做出的限制。所谓同源，通常要求协议、域名、端口三者都相同，只要其中一个不同，浏览器就会认为这不是同一个源。

浏览器之所以有跨域限制，核心原因是安全。如果没有这个限制，一个恶意网站就可能在用户登录状态下，偷偷读取另一个网站上的敏感数据，比如用户信息、订单信息、银行账户相关内容。

需要注意的是，跨域问题主要发生在浏览器环境里。前端页面通过浏览器去请求另一个源的接口时，会受到限制；但服务端和服务端之间的请求通常没有这个同源限制。

最常见的解决方案是 CORS，也就是跨域资源共享。它不是前端“绕过”限制，而是服务端通过响应头告诉浏览器：这个源的请求我允许访问。比如最常见的头就是 `Access-Control-Allow-Origin`。

如果前端在开发环境里遇到跨域，通常会使用本地开发代理，比如 Vite、Webpack Dev Server 或 Node 中间层，把浏览器请求先发给同源代理，再由代理转发到真实后端。生产环境里则更常见通过 Nginx 或 API 网关做统一转发。

除了这些，还有一些补充方案。比如 JSONP 是早期利用 `script` 标签不受同源限制来获取数据，但只支持 GET，现在更多是历史知识点。`postMessage` 则适用于不同窗口、iframe 之间的安全通信，不是通用接口跨域方案。

面试里如果要总结一句话，可以说：跨域不是接口不能访问，而是浏览器基于同源策略限制前端读取响应；常见解决方式是服务端配 CORS，或者通过代理和网关把跨域请求变成同源请求。

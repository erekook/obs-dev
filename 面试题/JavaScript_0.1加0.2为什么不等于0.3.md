# 题目
为什么 JavaScript 里 `0.1 + 0.2 !== 0.3`？这个问题的根本原因是什么？实际开发里通常怎么处理？

# 回答要点
- JavaScript 中 Number 类型基于 IEEE 754 双精度浮点数。
- 十进制小数在二进制里不一定能被有限表示。
- `0.1` 和 `0.2` 转成二进制后都是无限循环小数，只能近似存储。
- 近似值参与运算后，再转换回十进制就可能出现精度误差。
- 实际开发处理方式：
  - 对结果做误差容忍判断，如 `Number.EPSILON`
  - 金额运算尽量转整数处理（分、厘）
  - 使用专门的高精度计算库
- 面试里可补充：
  - 这不是 JavaScript 独有问题，而是很多语言的浮点数通病
  - 不建议直接用 `===` 比较浮点运算结果

# 追问
- 为什么有些小数可以准确表示，有些不行？
- `toFixed` 能彻底解决精度问题吗？
- `Number.EPSILON` 适合怎么用？
- 金额计算为什么通常建议转成整数？

# 参考答案
`0.1 + 0.2 !== 0.3` 的根本原因，不是 JavaScript 算错了，而是浮点数存储本身有精度限制。

JavaScript 的 `Number` 类型底层使用的是 IEEE 754 双精度浮点数格式。这个格式用二进制来表示数字，而像 `0.1`、`0.2` 这样的十进制小数，在二进制里通常不能被有限位精确表示，会变成无限循环小数。

因为存储空间有限，JavaScript 只能把它们截断成一个最接近的近似值。两个近似值再参与运算，结果自然也可能是一个近似值，所以最后就出现了类似 `0.30000000000000004` 这样的结果。

这不是 JavaScript 特有的问题，很多使用浮点数的语言都会遇到。

实际开发里，一般不会直接用 `===` 比较浮点计算结果，而是判断它们的差值是否足够小，比如：

```js
Math.abs(0.1 + 0.2 - 0.3) < Number.EPSILON
```

如果是金额这类对精度要求高的场景，更常见的做法是先把单位放大成整数，比如元转成分，再进行加减乘除，最后再换算回去。更复杂的场景也可以使用专门的高精度计算库。

所以面试里可以总结成一句话：`0.1 + 0.2 !== 0.3` 的本质是浮点数二进制表示精度有限，导致存储和计算都只能近似，处理时要用误差比较、整数化或高精度库来规避问题。

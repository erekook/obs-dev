# 题目
请你讲一下 JavaScript 里 `forEach` 和 `map` 的区别。它们看起来都在遍历数组，为什么不能随便混用？

# 回答要点
- 两者都会遍历数组，但设计目的不同。
- `forEach`：
  - 更适合做副作用操作
  - 没有有意义的返回值（返回 `undefined`）
  - 常用于打印、修改外部状态、触发请求等
- `map`：
  - 更适合基于原数组映射生成一个新数组
  - 返回值是新数组
  - 不改变原数组结构（除非回调里自己改）
- 为什么不能混用：
  - 用 `map` 只做副作用会让语义变差
  - 用 `forEach` 做数据转换会失去链式表达优势
- 面试里可补充：
  - 语义清晰比“都能跑”更重要
  - `map` 常用于 JSX 列表渲染、数据转换

# 追问
- 为什么说 `map` 更偏函数式思维？
- `forEach` 能被 `break` 中断吗？
- 如果要过滤再映射，为什么 `filter + map` 比 `forEach` 更清晰？
- `map` 回调里不 return 会发生什么？

# 参考答案
`forEach` 和 `map` 都能遍历数组，但它们的设计目的不一样，所以不能因为“都能循环”就随便替换。

`forEach` 更像是在说：“把数组里的每一项都拿出来执行一遍某个操作。”它通常适合做副作用，比如打印日志、累加外部变量、发请求、修改某个外部状态。它本身不会返回一个有意义的新结果，返回值通常是 `undefined`。

`map` 则更像是在说：“把原数组的每一项映射成一个新值，最终得到一个新数组。”所以它的重点不是遍历过程本身，而是“生成新数组”。这也是为什么它经常用在数据转换、列表渲染、结构重组这类场景里。

两者最大的区别，可以简单记成一句话：`forEach` 偏执行动作，`map` 偏生成结果。

也正因为这样，语义上不能随便混用。如果你只是想遍历数组顺手打印一下日志，却用了 `map`，那虽然代码能跑，但你创建了一个根本不用的新数组，语义也不清楚。反过来，如果你明明是想把数组转换成另一种结构，却用 `forEach` 手动 `push` 到新数组里，也会让代码显得更啰嗦，失去 `map` 本来清晰的表达力。

比如 JSX 列表渲染里，`map` 就特别自然，因为它正好是在“把数据项映射成一组新的节点”。

面试里如果总结，可以说：`forEach` 适合执行副作用，`map` 适合生成新数组；两者都能遍历，但一个强调过程，一个强调结果，所以不应该混用。

# 题目
为什么很多人说 Promise 比传统回调函数更好维护？它主要改善了哪些问题？

# 回答要点
- Promise 不是让异步变快，而是让异步流程更可组织、可组合。
- 相比回调，Promise 的优势：
  - 链式调用更清晰
  - 错误处理更集中
  - 更容易组合多个异步任务
  - 避免层层嵌套的回调地狱
- 传统回调的问题：
  - 嵌套深
  - 成功失败分支分散
  - 错误处理不统一
  - 流程可读性差
- 面试里可补充：
  - `async/await` 进一步在 Promise 基础上提升了可读性
  - Promise 也不是没有心智成本，但在工程实践里更统一

# 追问
- 什么叫回调地狱？
- Promise 的错误冒泡为什么更容易统一处理？
- Promise.all / race 这类能力为什么让异步组合更方便？
- Promise 和回调能否并存，实际项目怎么渐进迁移？

# 参考答案
很多人说 Promise 比传统回调更好维护，核心不是因为 Promise 更“高级”，而是它把异步流程从分散、嵌套、难追踪的写法，变成了更可组合、更线性的结构。

传统回调最大的问题是容易形成回调地狱。也就是一个异步结束后，在回调里再写下一个异步，下一个回调里再套下一层。层级一深，代码就会越来越难读，成功逻辑和失败逻辑也常常散落在不同地方。

Promise 的改进点首先是链式调用。你可以把多个异步步骤按顺序串起来，而不是一层层往里套，这样代码结构更平。

第二个很重要的改进是错误处理更集中。传统回调里，错误常常要在每一层手动判断和传递；而 Promise 链里，很多异常可以沿链路往后传，最后统一由 `.catch()` 处理，这让流程更清晰。

第三，Promise 更容易做异步组合。像多个请求并发等待、谁先完成用谁、无论成功失败都收集结果，这类能力用纯回调写会比较散，而 Promise 提供了更统一的抽象。

当然，Promise 并不是完全没有成本。它本身也有链式时序、错误传播、值穿透这些规则需要理解。但和传统多层回调相比，它在工程上的表达能力明显更强，也更适合作为团队协作里的统一模式。

再往前一步，`async/await` 其实也是建立在 Promise 之上的，它进一步把 Promise 链写成了更接近同步顺序代码的形式，所以现代前端异步代码维护体验才会更好。

面试里如果总结，可以说：Promise 相比回调更好维护，主要是因为它让异步流程更扁平、错误处理更集中、异步组合更统一，从而减少了回调地狱和流程分散带来的复杂度。

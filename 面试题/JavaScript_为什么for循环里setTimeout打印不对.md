# 题目
为什么在 JavaScript 里，`for` 循环配合 `setTimeout` 经常会出现“打印结果和预期不一样”的情况？

# 回答要点
- 这类问题常见根因：
  - `var` 的函数作用域
  - 异步回调执行时机晚于循环结束
- 典型现象：
  - `for (var i = 0; i < 3; i++) setTimeout(...)`
  - 最后打印多个相同值
- 原因解释：
  - 循环同步执行很快完成
  - `setTimeout` 回调稍后进入任务队列执行
  - 回调读取到的是同一个 `i`
  - 此时 `i` 已变成循环结束后的值
- 常见解决方案：
  - `let` 形成块级作用域
  - IIFE 立即执行函数保存当次值
  - 回调参数传值
- 面试里可补充：
  - 本质是作用域和异步时序共同作用，不只是 `setTimeout` 本身的问题

# 追问
- 为什么 `let` 在这里就能解决问题？
- IIFE 保存值的原理是什么？
- 如果不用 `setTimeout` 而是 Promise，会有何不同？
- 这类题本质是在考事件循环还是作用域？

# 参考答案
`for` 循环配合 `setTimeout` 打印结果不对，是一个非常经典的 JavaScript 面试题。它表面看像是异步问题，实际上本质上是“作用域 + 执行时机”一起造成的。

比如这段代码：

```js
for (var i = 0; i < 3; i++) {
  setTimeout(() => {
    console.log(i);
  }, 0);
}
```

很多人一开始以为会打印 `0 1 2`，但实际通常会打印 `3 3 3`。

原因在于：`for` 循环本身是同步执行的，它会非常快地跑完三次。`setTimeout` 只是把回调注册进去，真正执行要等当前同步代码全部结束之后。

而 `var` 是函数作用域，不会为每次循环单独创建一个新的 `i`。所以这三个回调里引用的其实都是同一个 `i`。等它们真正开始执行时，循环早就结束了，这个 `i` 的值已经变成了 3。

如果改成 `let`：

```js
for (let i = 0; i < 3; i++) {
  setTimeout(() => {
    console.log(i);
  }, 0);
}
```

就会打印 `0 1 2`。因为 `let` 是块级作用域，循环的每一次迭代都会有一个新的 `i` 绑定，所以每个回调拿到的都是当次循环对应的值。

如果不用 `let`，也可以用 IIFE 把当前值包起来，本质也是人为为每次循环创建一个独立作用域来保存值。

所以这道题的重点并不是“setTimeout 会不会延迟多久”，而是：同步循环先执行完，异步回调后执行；再加上 `var` 只共享一个变量，所以最后都读到了同一个结果。

面试里如果总结，可以说：这类问题的本质是 `var` 的作用域共享和异步回调延后执行共同导致的，`let`、IIFE 本质上都是在为每次循环保存独立的值。

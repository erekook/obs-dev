# 题目
为什么在 JavaScript 里，两个看起来内容一模一样的对象，直接用 `===` 比较通常也不相等？

# 回答要点
- 对象、数组、函数属于引用类型。
- `===` 比较引用类型时，比较的是：
  - 是否指向同一块内存 / 同一个引用地址
- 所以：
  - 内容一样但引用不同 → `false`
  - 同一个对象的不同变量引用 → `true`
- 与基本类型不同：
  - 基本类型比较通常看值本身
- 面试里可补充：
  - 这也是为什么状态管理里常强调“引用变化”
  - 深比较和浅比较是不同概念

# 追问
- 为什么框架优化里常用浅比较而不是深比较？
- 数组为什么也会有同样的问题？
- 如果想比较对象内容是否相同，常见做法有哪些？
- 为什么说“引用相等”和“值相等”要分开理解？

# 参考答案
在 JavaScript 里，两个对象内容看起来完全一样，但直接用 `===` 比较通常还是 `false`，这是因为对象属于引用类型。

所谓引用类型，意思不是变量里直接装着对象全部内容，而是变量里保存了一个“指向这个对象的引用”。当你用 `===` 比较两个对象时，JavaScript 比较的不是它们里面的字段是不是长得一样，而是它们是不是同一个对象引用。

比如：

```js
const a = { name: 'Tom' };
const b = { name: 'Tom' };
console.log(a === b); // false
```

这里 `a` 和 `b` 虽然内容一样，但它们是分别新建出来的两个对象，引用地址不同，所以结果是 `false`。

而如果是这样：

```js
const a = { name: 'Tom' };
const b = a;
console.log(a === b); // true
```

那就会返回 `true`，因为 `b` 只是引用了和 `a` 同一个对象。

这和基本类型不一样。像数字、字符串、布尔值这类基本类型，`===` 更多是在比“值本身”；而对象、数组、函数这种引用类型，比的是“是不是同一个引用”。

这也是为什么前端框架里经常会利用“引用是否变化”来判断状态是否更新，因为比较引用通常比做深层内容比较更快。

面试里如果总结，可以说：对象直接用 `===` 比较时，比的是引用身份，不是内容结构；所以内容一样但不是同一个对象，结果通常还是不相等。

# 题目
为什么有些 bug 在同步代码里看不出来，一到异步场景里就暴露了？前端常见的异步 bug 本质上都和什么有关？

# 回答要点
- 异步 bug 常见根因不是“语法不同”，而是时序和状态变复杂了。
- 常见问题来源：
  - 执行顺序和想象不一致
  - 多个异步结果返回先后不确定
  - 旧状态被晚到结果覆盖
  - 变量闭包引用了旧值
  - 组件销毁后异步结果仍回写
- 本质关键词：
  - 时序
  - 状态一致性
  - 生命周期
  - 竞态条件
- 面试里可补充：
  - 很多异步 bug 并不是“异步 API 有毒”，而是代码默认假设了顺序和状态不会变

# 追问
- 什么是竞态条件？
- 为什么搜索联想特别容易出现异步 bug？
- 组件卸载后还 setState 为什么是典型异步问题？
- 如何减少这类 bug？

# 参考答案
很多 bug 只有到了异步场景才暴露，不是因为异步代码本身更神秘，而是因为一旦进入异步，程序的执行顺序、状态变化和结果返回时机都不再像同步代码那样线性可控了。

同步代码里，你通常可以按“写在前面的先执行，写在后面的后执行”去理解逻辑。但异步里，发起请求的顺序和拿到结果的顺序，可能不是一回事。你以为 A 先发就会先回，实际上 B 可能更早返回，这就会导致状态被后来的旧结果覆盖，形成竞态问题。

另一个常见问题是状态在等待过程中变了。比如你发起请求时组件还在、条件还是旧值，等结果回来时页面可能已经切走、输入内容已经变了、筛选条件已经更新了。如果代码还按“发请求时的上下文”去写回结果，就很容易出 bug。

还有一些异步 bug 和闭包有关。比如回调函数里拿到的变量，其实是旧的一份引用或旧状态；当回调真正执行时，外部环境已经变了，于是逻辑和预期对不上。

所以很多异步 bug 的本质，可以概括成四个词：时序、状态一致性、生命周期、竞态条件。

解决这类问题，常见思路包括：给请求加取消或作废机制、在结果回写前确认当前状态是否仍有效、把旧请求结果丢弃、减少闭包里依赖的易变状态、在组件卸载时清理副作用等。

面试里如果总结，可以说：异步 bug 之所以常见，是因为异步把原本线性的执行模型变成了“时序不确定 + 状态可能变化”的系统，很多问题本质上都是竞态、旧状态回写和生命周期失配。

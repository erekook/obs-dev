# 题目
为什么 JavaScript 事件循环里，微任务（microtask）通常会比宏任务（macrotask）更早执行？这样设计的意义是什么？

# 回答要点
- 事件循环大致顺序：
  - 执行同步代码
  - 调用栈清空后，先清空微任务队列
  - 再执行下一个宏任务
- 微任务优先的意义：
  - 让本轮任务结束后产生的后续逻辑尽快完成
  - 保证 Promise 链等异步语义更连贯
  - 使状态更新和回调执行时机更可预测
- 常见微任务：
  - `Promise.then/catch/finally`
  - `queueMicrotask`
  - `MutationObserver`
- 常见宏任务：
  - `setTimeout`
  - `setInterval`
  - I/O
  - UI 事件回调
- 面试里可补充：
  - 微任务不是“绝对更重要”，而是调度优先级更高
  - 如果微任务不断追加，也可能导致宏任务和渲染被延后

# 追问
- 为什么大量微任务可能导致页面卡顿？
- 如果不先执行微任务，会对 Promise 链造成什么影响？
- `queueMicrotask` 和 `setTimeout(fn, 0)` 的行为差异是什么？
- 浏览器渲染通常插在事件循环的什么位置？

# 参考答案
JavaScript 事件循环里，微任务会在当前宏任务执行结束后、下一个宏任务开始前被优先清空。这个顺序不是偶然的，而是为了让“本轮执行过程中派生出来的后续逻辑”能尽快完成。

可以把宏任务理解成一轮比较完整的大任务，比如一次脚本执行、一次定时器回调、一次用户事件回调。而微任务更像这轮大任务结束前必须顺手处理完的一批“小尾巴”。

最典型的例子就是 Promise。一个 Promise 在当前任务里 resolve 之后，后续的 `then` 回调如果还要等到下一个宏任务再执行，整个异步链就会被拉得很散，时序也更难预测。把它放到微任务里，可以让 Promise 链在当前这轮逻辑结束后立刻衔接起来，看起来更连贯。

这种设计的好处是：本轮代码里产生的状态变化、后续回调、链式逻辑可以更快收敛，开发者也更容易推导执行顺序。所以我们才会看到同步代码执行完后，`Promise.then` 往往会先于 `setTimeout` 执行。

不过微任务优先也有代价。如果微任务里不断再塞新的微任务，事件循环会一直先处理它们，导致下一个宏任务迟迟得不到执行，页面渲染也可能被延后。这就是为什么大量滥用微任务可能造成卡顿。

面试里如果总结，可以说：微任务优先执行，是为了让当前任务派生出的后续逻辑在进入下一轮宏任务前尽快完成，从而保证 Promise 等异步机制的时序更连贯、更可预测。

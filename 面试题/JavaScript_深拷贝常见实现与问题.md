# 题目
请你讲一下 JavaScript 里的浅拷贝和深拷贝。深拷贝有哪些常见实现方式？各自有什么问题？

# 回答要点
- 浅拷贝只复制第一层，引用类型属性仍共享引用。
- 深拷贝会递归复制嵌套对象，目标对象与源对象相互独立。
- 常见浅拷贝方式：
  - `Object.assign`
  - 展开运算符
  - 数组的 `slice`、`concat`
- 常见深拷贝方式：
  - `JSON.parse(JSON.stringify(obj))`
  - 手写递归
  - `structuredClone`
  - 使用工具库如 `lodash.cloneDeep`
- 各方案问题：
  - JSON 方案无法处理 `undefined`、`function`、`Symbol`、循环引用、`Date`、`Map`、`Set` 等
  - 手写递归要考虑循环引用、原型链、特殊对象类型
  - `structuredClone` 支持更完整，但也不能克隆函数等
- 面试回答里要强调：
  - 不是所有场景都需要深拷贝
  - 深拷贝成本高，要结合数据规模和业务需求判断

# 追问
- 为什么 `JSON.parse(JSON.stringify(obj))` 不是通用深拷贝方案？
- `structuredClone` 相比手写递归的优势是什么？
- 如果对象存在循环引用，手写深拷贝怎么处理？
- React/Vue 项目里为什么有时不建议随意深拷贝大对象？

# 参考答案
浅拷贝和深拷贝的核心区别，在于是否会继续复制嵌套引用类型。

浅拷贝只会复制对象的第一层属性。如果属性值本身是基本类型，那么复制后相互独立；但如果属性值还是对象、数组这类引用类型，那么新旧对象内部其实仍然指向同一块内存。

比如 `Object.assign`、对象展开运算符、数组的 `slice` 和 `concat`，本质上都属于浅拷贝。

深拷贝则是把嵌套结构也递归复制一份，让新对象和原对象彻底分离。常见实现方式有几种。

第一种是 `JSON.parse(JSON.stringify(obj))`，优点是简单，面试里经常会被提到，但它的问题也很多，比如会丢失 `undefined`、函数、`Symbol`，不能处理循环引用，也不能正确处理 `Date`、`Map`、`Set` 等特殊对象。

第二种是手写递归深拷贝。它更灵活，可以按需处理不同类型，但实现复杂度高，通常要额外处理循环引用、数组、对象原型、特殊对象类型等问题。

第三种是现代环境里的 `structuredClone`。它是浏览器和部分运行时提供的原生深拷贝能力，支持的数据类型比 JSON 方案更完整，也能处理循环引用，通常更可靠。但它也不能克隆函数等不可结构化的数据。

实际开发里，还可能直接使用像 `lodash.cloneDeep` 这样的工具库。

面试时除了会背方案，更重要的是说明取舍。不是所有场景都需要深拷贝，因为深拷贝成本可能很高，尤其是大对象和深层嵌套数据结构。很多时候，只做必要的数据结构设计或局部复制，反而更合理。

# 题目
JavaScript 里做值比较时，`==`、`===` 和 `Object.is()` 有什么区别？实际开发里一般怎么选？

# 回答要点
- `==`：
  - 会做隐式类型转换后再比较
  - 规则复杂，容易产生意料之外结果
- `===`：
  - 严格相等，不做隐式类型转换
  - 开发中最常用
- `Object.is()`：
  - 语义上更像“SameValue”比较
  - 与 `===` 大部分情况一致
  - 但在 `NaN` 和 `+0/-0` 上行为不同
- 关键差异：
  - `NaN === NaN` 为 `false`，`Object.is(NaN, NaN)` 为 `true`
  - `+0 === -0` 为 `true`，`Object.is(+0, -0)` 为 `false`
- 实际建议：
  - 日常开发优先 `===`
  - 特殊场景需要精确区分 `NaN`、`+0/-0` 时可用 `Object.is`
  - 尽量少用 `==`
- 面试里可补充：
  - React 某些内部比较会用 `Object.is` 语义

# 追问
- 为什么 `==` 的比较规则容易出问题？
- `Object.is` 为什么要把 `NaN` 判成相等？
- `+0` 和 `-0` 在实际开发里有意义吗？
- 数组和对象做 `===` 比较时，比的到底是什么？

# 参考答案
JavaScript 里常见的相等比较有三种：`==`、`===` 和 `Object.is()`，它们最大的区别在于“比较时是否做类型转换，以及边界值怎么处理”。

`==` 是宽松相等，会在比较前做隐式类型转换。比如字符串和数字、布尔值和数字、`null` 和 `undefined` 等，都可能被转换后再比较。这导致它的规则很复杂，也很容易出现让人意外的结果，所以在实际开发里通常不推荐大量使用。

`===` 是严格相等，不会做隐式类型转换。类型不同，通常就直接返回 `false`。这让它的行为更直观，也是日常开发中最常用的比较方式。

`Object.is()` 在大多数普通场景下和 `===` 很像，但它对两个边界情况处理不同。

第一，`NaN === NaN` 是 `false`，但 `Object.is(NaN, NaN)` 是 `true`。第二，`+0 === -0` 是 `true`，但 `Object.is(+0, -0)` 是 `false`。

所以如果只是日常写业务代码，通常优先使用 `===` 就够了；如果你需要更精细地区分这些特殊情况，比如框架底层做状态变化判断时，就可能用到 `Object.is()`。

另外一个常见点是，对象和数组做 `===` 比较时，并不是比较内容，而是比较它们是否指向同一个引用地址。所以两个内容完全一样的对象，直接 `===` 比较通常也是 `false`。

面试里可以总结成一句话：`==` 会做类型转换，规则复杂；`===` 是日常首选；`Object.is()` 在大多数时候类似 `===`，但对 `NaN` 和 `+0/-0` 的处理更精确。

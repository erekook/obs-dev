# 题目
为什么闭包有时会导致内存占用增加？它到底“留住”了什么，实际开发里该怎么避免问题？

# 回答要点
- 闭包本身不是 bug，但会让外部作用域中的变量继续存活。
- 原因：
  - 内部函数仍然引用外部变量
  - 即使外部函数执行结束，这些变量也不能立刻释放
- 什么时候可能出问题：
  - 长期持有大量数据
  - 事件监听、定时器没有及时清理
  - 不必要地把大对象包进闭包
- 本质：
  - 不是“闭包一定泄漏”
  - 而是“被引用的数据无法回收”
- 避免思路：
  - 及时解绑事件和清除定时器
  - 不在闭包中保留不必要的大对象
  - 用完的引用及时置空或缩小作用域
- 面试里可补充：
  - 闭包和内存泄漏不是完全等价概念

# 追问
- 为什么说闭包是“变量活着”，而不是“函数套函数”本身？
- 定时器和事件监听为什么常和闭包问题一起出现？
- 如何判断一个闭包保留的数据是不是已经没有必要？
- 浏览器 DevTools 能怎么辅助排查这类问题？

# 参考答案
闭包有时会让内存占用增加，核心原因不是“闭包这个语法很危险”，而是闭包会让外部作用域中的某些变量继续被引用，从而不能被及时回收。

正常情况下，一个函数执行结束后，它内部创建的局部变量如果没有被别的地方继续引用，就可以被垃圾回收。而一旦这个函数内部返回了另一个函数，或者把内部函数交给了事件监听、定时器、异步回调，只要这个内部函数还在使用外部变量，那些变量就得继续留在内存里。

所以闭包真正“留住”的，不是函数壳子本身，而是被内部函数引用到的那部分作用域数据。

这本身并不一定有问题，很多场景闭包就是故意这么设计的，比如封装私有状态、防抖节流、函数工厂等。真正容易出问题的是：闭包长期保留了原本已经没必要存在的大对象、DOM 引用、列表数据或者上下文信息。

比如一个事件监听器绑定在页面上，但页面销毁后监听器没解绑，那么监听器回调形成的闭包可能还把旧页面的数据一起留着；或者一个定时器一直没清理，也会让相关作用域持续存活。

所以要避免问题，关键不是“别用闭包”，而是要管理好生命周期：用完及时解绑事件、清除定时器、不要在闭包里无意义保留大对象，必要时把引用缩小到最小范围。

面试里如果总结，可以说：闭包会增加内存占用，是因为它让被引用的外部变量继续存活；闭包不等于内存泄漏，但如果配合长期未清理的监听器、定时器或大对象引用，就可能演变成真正的问题。

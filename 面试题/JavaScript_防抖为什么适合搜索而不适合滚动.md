# 题目
为什么防抖（debounce）更适合搜索输入，而不太适合直接拿来处理滚动事件？

# 回答要点
- 防抖的特点：
  - 高频触发时不断重置计时器
  - 只有停止触发一段时间后才执行一次
- 搜索输入适合防抖：
  - 用户连续输入时，中间结果通常没有必要都请求
  - 更关心最终输入结果
  - 可以减少接口请求数量
- 滚动事件不适合纯防抖：
  - 滚动过程本身需要持续反馈
  - 如果只在停止后执行，用户滚动过程中页面可能没有及时响应
  - 会影响体验，比如吸顶、懒加载、进度同步不及时
- 滚动更常见方案：
  - 节流
  - `requestAnimationFrame`
  - `IntersectionObserver`
- 面试里可补充：
  - 不是说滚动场景永远不能用防抖，而是要看需求是否只关心“滚动结束后”的结果

# 追问
- 哪些滚动场景反而适合防抖？
- 为什么节流更适合持续反馈类交互？
- `requestAnimationFrame` 处理滚动回调有什么优势？
- 搜索联想除了防抖，还需要处理什么问题？

# 参考答案
防抖更适合搜索输入，原因在于它的行为模式和搜索联想这类需求非常匹配。

防抖的特点是：事件不断触发时，原本准备执行的函数会被一次次取消，只有等触发停止一小段时间之后，才真正执行最后一次。这种机制天然适合“我只关心最终结果”的场景。

搜索输入就是这样。用户连续敲字时，中间那些半成品关键字通常没有必要每次都发请求。比如用户想搜 “react”，如果每输入一个字符都立刻请求一次，就会产生很多无意义的接口调用。用防抖后，可以等用户停下来再请求，既减少压力，也更符合使用体验。

但滚动事件不一样。滚动是一种持续交互，很多时候页面需要在滚动过程中不断反馈，比如懒加载触发、吸顶效果、进度条同步、滚动位置计算等。如果用纯防抖，这些逻辑会一直等到用户停止滚动后才执行，过程中页面可能完全没有反应，体验会很差。

所以滚动场景更常见的思路是节流，或者用 `requestAnimationFrame`、`IntersectionObserver` 这类更适合高频连续场景的方案。因为它们能在持续滚动过程中，以可控频率不断给出反馈。

当然，也不是所有滚动场景都不能用防抖。如果你的需求本身就是“只在滚动结束后再做某件事”，比如滚动结束后保存位置、统计用户阅读段落，那防抖依然是合适的。

面试里如果总结，可以说：防抖适合只关心最终结果的交互，比如搜索输入；滚动更常需要持续反馈，所以通常更适合节流或其他更平滑的调度方式。

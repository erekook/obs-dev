# 题目
在 React 列表渲染中，为什么需要 `key`？如果 `key` 设置不合理，比如直接用数组下标，会带来什么问题？

# 回答要点
- `key` 是 React 用来识别同一层级节点身份的标识。
- React 在 Diff 过程中会根据 `key` 判断节点是复用、移动、删除还是新建。
- 合理的 `key` 能提高更新效率，并保证组件状态与正确数据项对应。
- 不推荐用数组下标作为 `key`，尤其是在列表会新增、删除、排序时。
- 下标作为 `key` 的问题：
  - 节点复用错误
  - 组件内部状态错位
  - 输入框内容、选中状态、动画表现异常
  - 增加不必要的重新渲染
- 最佳实践：
  - 使用稳定且唯一的业务 id
  - `key` 要在同级节点中唯一，不要求全局唯一
  - 避免每次渲染都生成随机 key

# 追问
- React 的 Diff 为什么不做完整树比较？
- `key` 改变后组件为什么会重新挂载？
- 哪些场景下用数组下标当 `key` 问题不大？
- Vue 中 `key` 的作用和 React 有什么异同？

# 参考答案
`key` 的本质作用，是帮助 React 在同一层级的多个子节点之间建立稳定身份。这样在下一次渲染时，React 就能知道哪个节点是“同一个东西”，从而决定是复用旧节点、移动位置、删除节点，还是创建新节点。

React 的 Diff 算法不是做完整的树编辑距离计算，因为那样成本太高。它基于一些假设来优化性能，其中一个关键点就是开发者通过 `key` 告诉它，哪些节点是稳定可追踪的。

如果 `key` 设置合理，比如使用后端返回的唯一 id，那么当列表顺序变化时，React 仍然知道每一项是谁，对应的组件状态也能跟着正确的数据项一起移动。

如果直接用数组下标作为 `key`，当列表发生插入、删除、排序时，就会出现问题。因为下标会变化，React 可能把原本属于 A 项的数据对应到 B 项组件上，造成组件复用错位。最常见的现象是：输入框内容错乱、勾选状态跑到别的项上、动画异常、局部组件状态不对。

比如一个待办列表，原来第 0 项是“学习”，第 1 项是“面试”。如果删除了第 0 项，那么原来第 1 项会变成新的第 0 项。此时如果 `key` 用的是下标，React 可能会错误复用之前第 0 个组件实例，导致内部状态和数据对不上。

所以最佳实践是：优先使用稳定、唯一、不会随着排序变化而变化的业务 id 作为 `key`。只有在列表完全静态、不会插入删除重排时，数组下标才勉强可以接受。

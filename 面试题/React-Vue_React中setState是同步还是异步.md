# 题目
React 里 `setState` / `setXXX` 到底是同步还是异步？为什么很多时候你立刻读取不到更新后的值？

# 回答要点
- 这个问题不能简单回答“同步”或“异步”，关键在于 React 的更新调度机制。
- 常见现象：
  - 调用状态更新函数后，当前执行上下文里立刻读取旧值
- 原因：
  - React 会把状态更新收集起来
  - 通过批量更新和调度，在合适时机统一处理
- 结果表现：
  - 在事件处理函数中，更新通常不会立刻反映到当前变量读取上
  - 下一次渲染时才拿到新值
- 正确写法：
  - 依赖旧值时用函数式更新
  - 更新后逻辑放在 `useEffect`、回调或下一轮渲染语义中处理
- 面试里可补充：
  - React 18 自动批处理让更多场景下更新合并
  - `setState` 更准确理解是“发起一次状态更新请求”

# 追问
- 为什么函数式更新能避免旧值问题？
- React 批量更新带来了什么好处？
- `useEffect` 为什么适合观察更新后的值？
- `flushSync` 是什么，为什么不应滥用？

# 参考答案
React 里的 `setState` 或 `useState` 的 `setXXX`，如果直接问“它是同步还是异步”，其实不够准确。更准确的说法是：它不是立刻修改你当前代码里拿到的那个状态变量，而是向 React 提交一个更新请求，由 React 在后续渲染流程中处理。

这就是为什么你经常会看到这样的现象：

```js
setCount(count + 1);
console.log(count);
```

这里打印出来的通常还是旧值。不是因为状态完全“异步得像 setTimeout”，而是因为当前这段代码运行时，React 还没有完成下一轮渲染，你拿到的仍然是本轮渲染里的旧状态快照。

React 之所以这样设计，是为了做批量更新和调度优化。比如在一次事件处理里连续调用多次状态更新，React 可以把它们合并，减少不必要的重复渲染，提高性能。

如果你的新状态依赖旧状态，正确写法通常是函数式更新：

```js
setCount(prev => prev + 1);
```

因为这样 React 会基于最新可用状态来计算，而不是依赖你当前闭包里可能已经过期的值。

如果你想在状态更新完成后执行某些逻辑，通常应该把逻辑放到 `useEffect` 里观察状态变化，而不是在调用更新函数后立刻依赖当前变量读取结果。

面试里如果总结，可以说：React 的状态更新不是立即改掉当前变量，而是进入 React 的调度和批处理流程，所以当前上下文里往往读到旧值；依赖旧值时要用函数式更新，依赖新值后的副作用通常放到后续渲染阶段处理。

# 题目
为什么 React 列表渲染里的 `key` 不能随便用随机数？这样做会带来什么问题？

# 回答要点
- `key` 的作用是帮助 React 识别同一层级节点的稳定身份。
- `key` 需要稳定且可预测，而随机数每次渲染都会变。
- 使用随机 key 的后果：
  - React 无法复用旧节点
  - 每次都当成全新节点处理
  - 组件会被反复卸载和重新挂载
  - 内部状态丢失
  - 性能变差
- 常见异常表现：
  - 输入框内容重置
  - 焦点丢失
  - 动画闪烁
  - 局部组件副作用重复触发
- 正确做法：
  - 使用稳定唯一的业务 id
  - 至少保证在同级中唯一且不随渲染变化
- 面试里可补充：
  - 随机 key 的问题比“用 index”更严重，因为它几乎必然导致完全不可复用

# 追问
- 为什么数组下标作为 key 在某些场景还能勉强接受，但随机数不行？
- `key` 改变后 React 为什么会重新挂载组件？
- 如果列表没有 id，该如何设计更稳定的 key？
- `key` 只需要在全局唯一，还是局部唯一？

# 参考答案
React 里的 `key` 不是随便写给框架看的一个标记，它的核心作用是告诉 React：“这个节点前后两次渲染是不是同一个东西。”

所以 `key` 最重要的要求不是花哨，而是稳定。如果你用了随机数，比如 `Math.random()`，那每次渲染得到的 key 都会变。对 React 来说，这就意味着“旧节点全没了，新节点全来了”。

一旦发生这种情况，React 就没法复用之前的组件实例和 DOM 节点，只能把旧节点全部卸载，再把新节点重新挂载。这样会直接带来两个问题。

第一个是性能差。因为原本可以复用的节点全都不能复用，每次更新都变成大规模重建。

第二个是状态丢失。比如列表里有输入框、展开收起状态、播放状态、焦点状态，这些都依赖组件实例被正确复用。如果 key 每次都变，组件就会被反复销毁重建，用户会看到输入框内容被重置、焦点丢失、动画闪烁、某些副作用重复触发等问题。

这也是为什么随机数 key 通常比数组下标还更糟。因为数组下标至少在列表不变时可能保持稳定，而随机数几乎每次渲染都会变，完全失去了“身份标识”的意义。

正确做法通常是使用业务数据中的唯一 id。只要它在同一层级里唯一，并且在多次渲染之间保持稳定，就可以作为合适的 key。

面试里可以总结成一句话：`key` 的价值在于稳定标识节点身份，而随机数最大的问题恰恰是“不稳定”，所以它会让 React 无法复用节点，既影响性能，也破坏组件状态。

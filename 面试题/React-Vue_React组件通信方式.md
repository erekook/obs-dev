# 题目
请你讲一下 React 组件之间常见的通信方式。父子组件、兄弟组件、跨层级组件一般分别怎么通信？

# 回答要点
- React 数据流默认是单向的，核心是 props 向下传递。
- 父子通信：
  - 父传子用 props
  - 子传父通常通过父组件传入回调函数
- 兄弟组件通信：
  - 通过共同父组件提升状态（lift state up）
- 跨层级通信：
  - `Context`
  - 状态管理库（Redux、Zustand、MobX 等）
- 特殊场景：
  - `ref` / `forwardRef` / `useImperativeHandle` 暴露实例能力
  - 自定义事件总线不常作为 React 主流推荐方案，但某些场景可用
- 选择原则：
  - 就近通信优先简单方案
  - 全局共享状态再考虑状态管理
  - 不要一上来就把所有状态放全局

# 追问
- 为什么说 Context 不完全等于状态管理？
- 什么情况下状态提升会让代码变得难维护？
- `useImperativeHandle` 适合什么场景？
- Redux 和 Zustand 在组件通信层面解决的问题有什么区别？

# 参考答案
React 的组件通信，本质上是围绕“单向数据流”展开的。也就是说，数据通常从父组件通过 props 往下传，组件之间不会像双向绑定那样自动互相同步。

最基础的父子通信很好理解。父组件给子组件传数据，用 props 就可以；如果子组件想把信息传回父组件，一般是父组件先传一个回调函数给子组件，子组件在合适的时候调用这个回调，把数据带回去。

兄弟组件之间不能直接通信，通常的做法是把它们共同需要的数据提升到最近的公共父组件里。父组件统一管理状态，再分别通过 props 传给两个兄弟组件，这就是常说的“状态提升”。

如果组件层级很深，一级一级传 props 很麻烦，这时候可以用 Context。Context 适合做跨层级共享，比如主题、语言、用户信息这类相对稳定的全局上下文。

当状态更复杂，比如多个页面、多个模块都要共享，而且有较多更新逻辑时，通常会使用 Redux、Zustand、MobX 这类状态管理方案。它们本质上不是为了替代 props，而是为了更好地管理共享状态。

另外还有一些特殊通信手段，比如父组件通过 `ref` 获取子组件暴露的方法，这在表单组件、弹窗组件、滚动容器等场景里比较常见。函数组件里通常会配合 `forwardRef` 和 `useImperativeHandle` 使用。

如果总结一下，React 组件通信的常见思路是：父子之间用 props 和回调，兄弟之间靠状态提升，跨层级用 Context，更复杂的全局共享用状态管理库。面试时如果再补一句“通信方式要按作用范围和复杂度选择”，会比较完整。

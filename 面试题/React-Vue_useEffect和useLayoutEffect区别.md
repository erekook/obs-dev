# 题目
请你讲一下 React 中 `useEffect` 和 `useLayoutEffect` 的区别。它们分别适合什么场景？

# 回答要点
- 两者都是处理副作用的 Hook，但执行时机不同。
- `useEffect`：
  - 在浏览器完成绘制后异步执行
  - 不会阻塞页面绘制
  - 更适合大多数副作用
- `useLayoutEffect`：
  - 在 DOM 更新后、浏览器绘制前同步执行
  - 可能阻塞绘制
  - 适合读取布局、同步修改样式等场景
- 常见场景：
  - `useEffect`：请求数据、订阅事件、定时器、日志上报
  - `useLayoutEffect`：测量 DOM、避免闪烁、同步滚动位置调整
- 注意点：
  - `useLayoutEffect` 用多了可能影响性能
  - 服务端渲染环境下要注意告警问题
- 面试里可补充：
  - 二者都会在依赖变化后执行，也都支持清理函数
  - 选择关键在于“副作用是否依赖布局结果、是否需要阻止闪屏”

# 追问
- 为什么读取 DOM 尺寸更适合 `useLayoutEffect`？
- 在 SSR 场景里使用 `useLayoutEffect` 有什么问题？
- `useEffect` 为什么通常更推荐作为默认选择？
- React 严格模式下副作用执行两次该怎么理解？

# 参考答案
`useEffect` 和 `useLayoutEffect` 都是 React 里处理副作用的 Hook，它们的主要区别在于执行时机。

`useEffect` 会在组件渲染完成并且浏览器把页面绘制出来之后再执行，所以它不会阻塞页面显示。也正因为这样，它适合大多数日常副作用，比如发请求、注册事件监听、开启定时器、做日志上报等。

`useLayoutEffect` 则更早。它会在 React 把 DOM 更新到页面之后、浏览器真正绘制之前同步执行。这意味着你可以在用户看到页面之前，先读取布局信息或者立即做一些同步修正，比如计算元素尺寸、调整滚动位置、修正某些样式，避免页面先闪一下再变正确。

也正因为 `useLayoutEffect` 会阻塞浏览器绘制，所以它的成本更高。如果只是普通副作用，没有依赖 DOM 布局结果，一般更推荐使用 `useEffect`。

举个例子，如果你只是组件挂载后请求接口数据，那用 `useEffect` 就够了；但如果你需要在渲染后立刻拿到某个元素的宽高，并根据宽高决定布局，或者你要在页面显示前把滚动条滚到指定位置，那么 `useLayoutEffect` 会更合适。

两者都支持依赖数组和清理函数，都会在依赖变化时重新执行并清理上一次副作用，所以使用方式很像。区别核心不是 API 写法，而是时机。

面试时可以把结论总结成一句话：默认优先用 `useEffect`，只有当副作用必须在浏览器绘制前同步完成，比如测量布局、同步修正界面，才考虑 `useLayoutEffect`。

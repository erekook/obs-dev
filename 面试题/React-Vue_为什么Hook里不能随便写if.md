# 题目
为什么 React Hook 不能随便写在 `if`、循环或者普通函数里？这条规则背后的原因是什么？

# 回答要点
- Hook 依赖调用顺序来对应内部状态槽位。
- React 期望：
  - 每次组件渲染时 Hook 调用顺序一致
- 如果把 Hook 写进 `if` / 循环：
  - 某次渲染可能执行，某次不执行
  - Hook 顺序发生变化
  - React 无法正确把状态对应回原来的 Hook
- 正确规则：
  - 只在 React 函数组件顶层调用
  - 只在自定义 Hook 顶层调用
- 面试里可补充：
  - 这不是语法限制，而是 React 内部状态管理机制决定的
  - “条件逻辑”应放在 Hook 内部，而不是决定是否调用 Hook 本身

# 追问
- 为什么说 Hook 是靠“位置”而不是“名字”区分的？
- 自定义 Hook 为什么可以调用其他 Hook？
- 如果确实有条件逻辑，该怎么写才符合规则？
- 这个规则和类组件时代的状态管理有什么本质差异？

# 参考答案
React Hook 不能随便写在 `if`、循环或者普通函数里，核心原因不是“官方不让你这么写”，而是 React 内部管理 Hook 状态的方式决定了它必须依赖稳定的调用顺序。

你可以把 Hook 理解成：React 在组件每次渲染时，会按照你代码里的调用顺序，一个一个把状态和副作用槽位对上。比如第一个 `useState` 对应第一格状态，第二个 `useEffect` 对应第二格，以此类推。

只要每次渲染时 Hook 的调用顺序完全一致，React 就能知道“这次的第一个 `useState` 还是上次那个第一个 `useState`”。

问题出在，如果你把 Hook 放进 `if` 里，那么某次渲染条件成立会执行，另一次渲染条件不成立就不执行。这样后面的 Hook 顺序就整体错位了，React 就不知道该把哪一份状态对应给哪一个 Hook，最终会导致状态混乱。

循环里也类似，因为执行次数可能变化；普通函数里则不是 React 受控的组件渲染流程，React也没法按它的规则去追踪这些调用。

所以官方规则才会要求：Hook 只能在 React 函数组件顶层，或者自定义 Hook 顶层调用。这样才能保证每次渲染的调用顺序稳定。

如果你确实有条件逻辑，正确思路通常不是“条件决定要不要调用 Hook”，而是“先正常调用 Hook，再在 Hook 内部或者返回值使用阶段做条件判断”。

面试里如果总结，可以说：Hook 规则的本质是为了保证每次渲染时 Hook 调用顺序稳定，因为 React 是靠调用顺序而不是变量名来关联内部状态的；一旦顺序变化，状态映射就会错乱。

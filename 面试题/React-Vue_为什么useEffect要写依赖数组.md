# 题目
为什么 React 的 `useEffect` 需要写依赖数组？如果依赖写错了，会出现什么问题？

# 回答要点
- 依赖数组的作用：
  - 告诉 React，这个副作用依赖哪些值
  - 只有这些值变化时才重新执行 effect
- 不同写法含义：
  - 不写依赖数组：每次渲染后都执行
  - 空数组：通常只在挂载后执行一次
  - 写具体依赖：依赖变化时重新执行
- 为什么需要它：
  - 控制副作用执行时机
  - 避免不必要的重复执行
  - 保证 effect 使用的是期望的最新值
- 写错依赖的后果：
  - 漏依赖：闭包拿到旧值、逻辑过期
  - 多依赖：effect 频繁重跑，性能差或死循环
  - 依赖不稳定对象/函数：导致反复执行
- 面试里可补充：
  - React Hook lint 规则是帮助发现依赖问题
  - 依赖数组本质上和闭包快照有关

# 追问
- 为什么漏依赖会出现“拿到旧值”的问题？
- 为什么把函数写进依赖数组可能导致 effect 一直执行？
- `useCallback`、`useMemo` 在这个问题上起什么作用？
- 有没有场景故意不把某些值写进依赖数组？风险是什么？

# 参考答案
`useEffect` 需要依赖数组，本质上是为了让 React 知道：这个副作用依赖哪些外部值，什么时候应该重新执行。

React 组件每次渲染，函数体都会重新执行一遍。effect 里的代码其实会“捕获”当前这次渲染里的变量值。如果没有一个明确机制告诉 React 什么时候重新跑 effect，就很容易出现副作用执行时机失控，或者拿到旧的变量值。

依赖数组就是这个机制。

如果你不写依赖数组，effect 会在每次渲染后都执行。
如果你传空数组，通常表示这个 effect 不依赖会变化的外部值，因此只在挂载后执行一次。
如果你写了具体依赖，React 就会在这些依赖变化时重新执行 effect。

依赖写错最常见的两个问题，一个是“漏依赖”，一个是“依赖不稳定”。

漏依赖时，effect 可能一直使用旧渲染周期里捕获的值，出现所谓“闭包拿到旧值”的问题。比如定时器、事件监听、异步请求回调里，经常会出现这种 bug。

依赖不稳定时，比如你把每次渲染都会新建的对象或函数直接放进去，effect 就可能每次都重新执行，甚至触发死循环。

所以 React 官方才会推荐配合 Hook lint 规则来检查依赖数组，因为它不是“可有可无的优化项”，而是 effect 正确性的一个核心部分。

面试里如果总结，可以说：依赖数组的作用是精确声明副作用依赖哪些值，从而控制 effect 的执行时机；依赖写错会导致旧值问题、重复执行甚至死循环，本质上和 React 渲染闭包机制有关。

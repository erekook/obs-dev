# 题目
为什么在 React 组件的 render 过程中不应该做副作用操作？如果做了，可能会有什么问题？

# 回答要点
- render 的职责是根据状态描述 UI，应该尽量保持纯函数特性。
- 副作用包括：
  - 发请求
  - 操作 DOM
  - 修改外部变量
  - 订阅事件
  - 定时器
- 为什么不应放在 render：
  - render 可能执行多次
  - React 可能为了调度、中断、重试而重复调用
  - 会导致副作用重复触发
  - 影响渲染可预测性
- 可能后果：
  - 重复请求
  - 重复订阅
  - DOM 状态错乱
  - 性能问题
  - 难以排查的 bug
- 正确做法：
  - 把副作用放到 `useEffect` / `useLayoutEffect`
  - 让 render 只负责计算 UI 结果
- 面试里可补充：
  - 这体现了 React “render 纯、effect 脏”的基本思想

# 追问
- 为什么 React 严格模式下这个问题更容易暴露？
- `useEffect` 和 render 在职责上怎么区分？
- render 里创建对象和函数算副作用吗？
- 为什么说“同样输入应得到同样 UI 输出”对 render 很重要？

# 参考答案
在 React 里，render 的核心职责是：根据当前的 props 和 state，计算出界面应该长什么样。也就是说，它本质上更像一个“描述 UI 的计算过程”，而不是执行外部动作的地方。

之所以不应该在 render 里做副作用，是因为 render 不保证只执行一次。React 在调度更新时，可能会为了性能优化、中断恢复、严格模式检查等原因，多次执行 render。如果你把发请求、操作 DOM、注册事件、开启定时器这类副作用放进去，就会随着 render 被重复执行，产生很多问题。

最常见的问题就是重复请求、重复订阅、重复定时器，以及外部状态被多次改写。这样不仅会浪费性能，还很容易出现难排查的 bug。比如一次界面刷新，结果接口被打了两遍，或者事件监听器被重复绑定多次。

React 之所以强调把副作用放到 `useEffect` 或 `useLayoutEffect`，本质上就是为了把“计算 UI”和“执行外部动作”这两件事分开。render 负责纯计算，effect 负责和外部世界打交道。

这样设计的好处是渲染流程更可预测。只要相同的输入进入 render，就应该得到相同的 UI 输出。副作用被隔离出去之后，React 才更容易做调度、优化和调试。

面试里如果总结，可以说：render 应该尽量保持纯函数特性，因为它可能被重复执行；副作用如果放进去，会造成重复请求、状态错乱和不可预测行为，所以应放到 effect 阶段处理。

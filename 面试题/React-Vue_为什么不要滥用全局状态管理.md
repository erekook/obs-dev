# 题目
为什么前端项目里不建议一上来就把所有状态都放进全局状态管理？滥用全局状态会有什么问题？

# 回答要点
- 全局状态管理有价值，但并不是所有状态都值得全局化。
- 更适合全局的状态：
  - 用户信息
  - 权限信息
  - 主题
  - 多页面共享配置
- 不适合全局的状态：
  - 组件局部 UI 状态
  - 临时输入值
  - 只在单页局部使用的数据
- 滥用全局状态的问题：
  - 状态边界不清晰
  - 依赖关系复杂
  - 调试困难
  - 容易引发无关组件更新
  - 心智负担变重
- 更合理的原则：
  - 能局部就局部
  - 需要跨层共享再考虑提升
  - 多页面/多模块共享再考虑全局
- 面试里可补充：
  - 状态管理设计的核心是作用域和生命周期匹配

# 追问
- 什么样的状态适合放在组件内部？
- 为什么“提升状态”比“直接全局化”更稳妥？
- 全局状态如何导致无关组件重渲染？
- React Query 这类数据获取状态管理和传统全局状态库有什么分工差异？

# 参考答案
前端项目里，全局状态管理当然有价值，但问题在于：它解决的是“跨模块共享和统一管理”的问题，不是“所有状态都该放进去”的理由。

如果一上来就把所有状态都放进全局 store，短期看似统一，长期往往会让状态边界越来越模糊。比如一个弹窗开关、一个输入框临时值、一个局部折叠状态，这些本来只属于某个组件内部的状态，如果也全都进全局，就会让整个项目的状态来源变得又多又散。

滥用全局状态最直接的问题是可维护性下降。因为很多局部逻辑被抽到全局之后，组件本身看起来“很干净”，但真实依赖却藏在 store 里，阅读和调试时反而更费劲。谁在改这个状态、哪些组件依赖它、生命周期应该持续多久，都变得不那么直观。

另一个问题是性能和耦合。全局状态一旦设计得过大，某些状态变化可能会影响一批并不真正关心它的组件更新，或者让模块之间产生本来不该有的隐性耦合。

更合理的思路通常是按作用域分层。只在当前组件内部使用的状态，就放组件内部；需要父子、兄弟或局部树共享的状态，可以先提升到更合适的上层；只有当一个状态真的需要跨页面、跨模块共享时，再考虑放进全局状态管理。

所以状态管理设计的关键，不是“有没有用 Redux/Zustand/Pinia”，而是状态的作用范围和生命周期是否匹配。

面试里如果总结，可以说：全局状态适合少量真正跨模块共享的数据，滥用全局状态会让边界模糊、依赖复杂、维护成本上升；状态应该按作用域从局部到全局逐级选择，而不是一把梭全塞进 store。

# 题目
为什么在 React / Vue 这类框架里，通常不建议直接修改 state 或响应式数据对象本身，而更强调创建新值或通过框架提供的方式更新？

# 回答要点
- 直接修改原对象可能带来状态变化不可追踪的问题。
- 框架更新依赖通常基于：
  - 状态变更通知
  - 引用变化检测
  - 响应式追踪机制
- 为什么不建议直接改：
  - 可能不会触发更新
  - 影响调试和状态回溯
  - 容易产生共享引用副作用
  - 让数据流不清晰
- 创建新值/规范更新的好处：
  - 变化更可预测
  - 更利于 Diff 和优化
  - 更方便做时间旅行调试、撤销恢复
- 面试里可补充：
  - React 更强调不可变更新
  - Vue 虽有响应式系统，也仍然强调按框架推荐方式维护状态

# 追问
- React 为什么特别强调不可变数据？
- “引用变化”对性能优化和 memo 有什么帮助？
- Vue3 用 Proxy 后是不是就可以随便改了？
- 直接修改嵌套对象时最容易踩什么坑？

# 参考答案
在 React、Vue 这类框架里，不建议直接修改状态对象本身，核心原因不是“语法不允许”，而是这样做会让状态变化变得不透明、不稳定，也更难被框架正确感知。

先看 React。React 很多更新判断、性能优化、memo 比较，本质上都依赖“引用有没有变化”。如果你直接改原对象内容，但对象引用没变，某些组件可能根本意识不到数据发生了变化，或者优化逻辑会被破坏。所以 React 非常强调不可变更新，也就是不要在原对象上改，而是创建一个新的对象或数组。

再看 Vue。Vue 的响应式系统确实比 React 更擅长追踪对象属性变化，尤其是 Vue3 用 Proxy 之后能力更强。但这并不意味着“随便改就永远没问题”。如果状态在多个地方共享引用，或者更新方式不统一，仍然容易造成副作用、调试困难和维护混乱。

从工程角度看，创建新值或者通过框架推荐方式更新状态，有几个明显好处。第一，状态变化路径更清晰，方便调试。第二，更容易做撤销恢复、时间旅行调试、状态快照对比。第三，也更有利于框架和开发者利用浅比较做性能优化。

直接修改原对象最大的风险，往往不是“当下就报错”，而是它会把问题埋在后面。比如一个对象被多个组件共享引用，你改了其中一个地方，其他地方也悄悄变了，最后很难定位是谁改的。

所以面试里可以总结成一句话：不直接改 state，不只是为了“触发更新”，更是为了让状态变化可追踪、可预测、可优化。React 尤其强调不可变更新，Vue 虽然响应式更强，但规范更新方式依然很重要。

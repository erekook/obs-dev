# 题目
为什么有时候你明明改了对象里的某个属性，页面却没有按预期更新？这背后通常和什么机制有关？

# 回答要点
- 页面不更新通常不是“改值失败”，而是“变更没有被正确感知”。
- 常见原因：
  - 直接修改对象，引用没变
  - 框架的更新依赖引用变化或响应式追踪
  - 修改方式绕过了框架的响应式机制
- React 常见理解：
  - 状态更新通常依赖新引用
  - 直接改原对象再 set 回去，可能无法触发预期更新
- Vue 常见理解：
  - 更新依赖响应式系统追踪
  - 某些写法可能让依赖收集或触发不符合预期
- 正确思路：
  - 用不可变更新方式
  - 创建新对象 / 新数组
  - 按框架推荐的状态更新方式来改
- 面试里可补充：
  - 本质常常不是“值变没变”，而是“框架怎么看见这个变化”

# 追问
- 为什么“引用变了”对 React 很重要？
- 深层对象更新时为什么更容易踩坑？
- Vue 的响应式追踪和 React 的不可变更新思路有什么差异？
- 如果数据更新了但视图没变，排查顺序一般怎么走？

# 参考答案
有时候对象属性明明改了，页面却没更新，最常见的根因不是“赋值没成功”，而是这个变化没有被框架用它预期的方式感知到。

你可以把问题理解成两层：
1. JavaScript 里的数据确实变了；
2. 但框架未必知道“该重新更新视图了”。

在 React 里，这类问题非常常见，因为 React 状态更新通常强调不可变数据思路。也就是说，React 更习惯通过“新引用”来判断数据是否发生变化。如果你直接改了原对象里的某个属性，再把同一个对象引用传回去，很多时候就会导致更新判断不符合预期。

比如：

```js
state.user.name = 'Tom';
setUser(state.user);
```

虽然 `name` 变了，但 `state.user` 这个对象引用还是原来的，组件更新和依赖比较就可能出问题。更推荐的是创建新对象：

```js
setUser({ ...user, name: 'Tom' });
```

在 Vue 里，问题的角度略有不同。Vue 依赖响应式系统去追踪读取和修改，如果某些修改方式绕过了响应式追踪，或者依赖没有按预期收集，也可能出现“数据改了但页面没反应”的现象。

所以这类问题的共同本质其实是：不是只看数据有没有变化，而是看这个变化有没有被框架正确追踪到，并触发视图更新链路。

面试里如果总结，可以说：对象属性改了但页面没更新，常见原因是直接修改原对象导致变更没有被框架按预期感知；React 更强调新引用，Vue 更依赖响应式追踪，解决思路通常都是按框架推荐方式去做状态更新。

# 题目
为什么一个列表在排序或插入删除之后，里面的输入框内容有时会“跑到别的项上去”？这背后通常是什么问题？

# 回答要点
- 常见根因：列表 `key` 不稳定，尤其是用数组下标。
- 为什么会乱：
  - 框架在更新列表时会尝试复用已有节点/组件实例
  - 如果 key 不能正确标识数据项身份，复用就会错位
- 结果表现：
  - 输入框内容跑到其他项
  - 勾选状态错乱
  - 焦点丢失或跳动
  - 局部组件状态和数据不匹配
- 解决思路：
  - 使用稳定唯一的业务 id 作为 key
  - 避免用 index 作为会变化列表的 key
- 面试里可补充：
  - 这本质上不是“输入框的问题”，而是“节点身份识别”的问题
  - key 决定了框架如何在更新中复用组件实例

# 追问
- 为什么静态列表用 index 可能问题不大，但动态列表不行？
- 为什么随机数 key 反而更糟？
- 排序时框架到底是怎么复用节点的？
- 这个问题在 React 和 Vue 里有没有共通点？

# 参考答案
列表排序后输入框内容“跑到别的项上去”，最常见的根因其实不是输入框本身，而是列表项的 `key` 不稳定，导致框架在更新时错误复用了组件实例。

无论是 React 还是 Vue，列表更新时都会尽量复用已有节点和组件实例，以减少不必要的重建。但框架要想正确复用，前提是它得知道“这一项前后是不是同一个东西”。这个身份标识，通常就是 `key`。

如果你用了数组下标做 key，那么一旦列表插入、删除或排序，下标就会变化。对框架来说，原来第 0 项和现在第 0 项好像是“同一个 key”，于是它会把旧节点复用到新的数据项上。这样就会出现输入框内容、勾选状态、焦点等局部状态跟着节点走，而不是跟着真实数据项走。

所以你看到的“内容跑到别的项上”，本质上是“组件实例复用错位”。

最好的解决办法通常是：使用稳定且唯一的业务 id 作为 key。这样无论怎么排序、插入或删除，框架都能正确知道每一项到底是谁，从而把状态和数据保持一致。

这也是为什么静态列表偶尔用 index 看起来没出事，但动态列表里就很容易踩坑。因为静态列表顺序不变，index 恰好还算稳定；一旦顺序变化，它就不再能表示真实身份。

面试里如果总结，可以说：列表操作后输入框内容错位，根本原因通常是 key 不稳定，导致框架在 Diff 过程中把组件实例复用错了；解决关键是用稳定唯一的 key 标识数据项身份。

# 题目
为什么 React / Vue 列表渲染都强调要有稳定的 `key`？它到底帮助框架做了什么？

# 回答要点
- `key` 的本质：同层节点的稳定身份标识。
- 它的作用：
  - 帮助框架在更新时识别哪些节点可复用
  - 判断节点是新增、删除、移动还是更新
  - 保持组件实例与数据项正确对应
- 稳定 key 带来的好处：
  - 减少不必要重建
  - 保留正确状态
  - 提升更新效率
- key 不稳定的后果：
  - 输入框状态错位
  - 焦点丢失
  - 动画异常
  - 组件被错误复用或重复挂载
- 面试里可补充：
  - key 只需要在同级唯一
  - 稳定性往往比“看起来唯一”更重要

# 追问
- 为什么数组下标在增删排序场景里会出问题？
- 为什么随机数 key 比 index 更糟？
- key 改变时组件为什么会重新挂载？
- Vue 和 React 在 key 的使用原则上有哪些共通点？

# 参考答案
React 和 Vue 在列表渲染里都强调稳定的 `key`，核心原因是：框架在前后两次渲染结果之间做比较时，需要知道“这个节点前后是不是同一个东西”。而 `key` 就是这个身份标识。

如果没有稳定的 `key`，框架只能按照位置去猜测节点对应关系。这样一旦列表发生插入、删除、排序，就很容易把原本属于 A 项的组件实例，错误地复用到 B 项上，导致状态错位。

有了稳定的 `key` 之后，框架就能更明确地知道：哪些节点只是内容变了，哪些节点是新加的，哪些节点被删掉了，哪些节点只是位置移动了。这样更新过程就更准确，也更高效。

它的价值不仅是性能，更重要的是“状态正确性”。比如列表里有输入框、复选框、展开收起状态、动画状态，如果 `key` 不稳定，这些状态很容易跑到别的项上，用户会感觉页面行为很奇怪。

这也是为什么 `key` 不只是“写一个不重复值”就行，还要尽量稳定。业务里的唯一 id 通常是最理想的 key，因为它在多次渲染之间不会变。相反，数组下标在列表会增删排序时不稳定，而随机数更糟，因为它几乎每次渲染都变，等于告诉框架“这些节点全是新的”。

所以面试里可以总结成一句话：稳定的 `key` 让框架能正确识别节点身份，从而既提升更新效率，也保证组件状态不会错位。它解决的核心不是“让语法通过”，而是“让 Diff 和复用变得可靠”。

# 题目
为什么在 React / Vue 这类框架里，通常强调“父传子用 props”，而不是让子组件自己到处去拿需要的数据？

# 回答要点
- `props` 的核心价值：
  - 明确数据来源
  - 保持单向数据流
  - 降低组件耦合
- 父传子的好处：
  - 组件职责清晰
  - 依赖显式可见
  - 更容易测试和复用
  - 更方便追踪状态变化来源
- 子组件“自己去拿”的问题：
  - 隐式依赖变多
  - 与外部环境耦合增强
  - 复用性下降
  - 调试和维护成本上升
- 面试里可补充：
  - 不是说组件永远不能读上下文或全局状态
  - 而是默认优先局部显式传递，作用域扩大后再考虑 Context/Store

# 追问
- 为什么说显式依赖比隐式依赖更好维护？
- props drilling 到很深时怎么办？
- Context 和 props 的边界通常怎么判断？
- 组件复用时，为什么 props 方式更友好？

# 参考答案
在 React、Vue 这类框架里，强调父传子用 props，本质上是为了让数据流保持清晰、可追踪、可维护。

当父组件通过 props 把数据传给子组件时，子组件依赖什么是一眼能看出来的。你读组件代码时，很容易知道：它需要哪些输入、这些输入从哪里来、改动会影响谁。这种显式依赖对维护非常重要。

如果让子组件自己到处去拿数据，比如直接耦合某个全局 store、随处读取上下文、或者依赖外部单例，那组件表面上看起来 props 很少，但真实依赖反而被藏起来了。这样组件一旦脱离当前环境，就很难复用，也更难测试。

props 还有一个重要价值是保持单向数据流。也就是数据由上往下传，组件之间的关系更稳定，状态变化来源也更容易追踪。你知道某个值变了，通常就去找父级或上游数据源，而不是在一堆隐式依赖里猜到底是谁改了它。

这并不是说组件永远不能直接读上下文或全局状态。像主题、用户信息、跨很多层共享的配置，这些当然可以用 Context 或状态管理工具。但在默认情况下，如果只是父子局部关系，props 往往是更清晰、更稳妥的选择。

所以面试里如果总结，可以说：父传子用 props 的核心不是“语法规定”，而是为了保持单向数据流和显式依赖，让组件更容易理解、复用、测试和维护；只有在共享范围明显扩大时，才更适合用 Context 或全局状态。

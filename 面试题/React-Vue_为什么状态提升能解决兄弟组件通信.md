# 题目
为什么“状态提升”（lifting state up）可以解决兄弟组件之间的通信问题？它的核心思想是什么？

# 回答要点
- 兄弟组件之间通常没有直接共享状态通道。
- 状态提升的做法：
  - 把共享状态放到它们最近的公共父组件
  - 父组件把数据通过 props 下发
  - 子组件通过回调把变更意图传回父组件
- 核心思想：
  - 单一数据源
  - 数据向下流，事件向上传
- 为什么有效：
  - 避免兄弟组件各自维护一份不一致状态
  - 父组件统一协调更新
- 面试里可补充：
  - 状态提升适合局部共享
  - 如果跨层级太深或共享范围太大，再考虑 Context / 全局状态管理

# 追问
- 为什么兄弟组件不建议互相直接改对方状态？
- 状态提升什么时候会开始变得难维护？
- Context 和状态提升的边界怎么判断？
- 单一数据源为什么能减少 bug？

# 参考答案
状态提升能解决兄弟组件通信，核心原因是：兄弟组件之间本来没有天然的数据共享关系，但它们通常都有同一个父组件，而这个父组件正好可以成为它们共享状态的“统一来源”。

如果两个兄弟组件都各自维护一份看起来相同的数据，就很容易出现状态不一致的问题。比如一个组件改了值，另一个组件却还停留在旧状态。状态提升的思路，就是把这份共享数据挪到它们最近的公共父组件里统一管理。

这样一来，父组件就成了单一数据源。父组件通过 props 把当前状态下发给两个兄弟组件；如果某个子组件想修改状态，它不能直接去改另一个兄弟组件，而是通过回调把“我想改成这样”的意图告诉父组件，再由父组件统一更新状态并重新分发。

所以这个模式本质上是：数据向下流，事件向上传。父组件像一个协调者，负责保存共享状态并广播给子组件。

它之所以有效，是因为所有依赖这份状态的组件都只认同一个来源，而不是每个人维护自己的副本。这样不但逻辑更清晰，也更容易排查问题。

当然，状态提升也不是万能的。如果共享范围越来越大、层级越来越深，props 一层层传下去会变得繁琐，这时候才适合进一步考虑 Context 或全局状态管理。

面试里如果总结，可以说：状态提升之所以能解决兄弟组件通信，是因为它把共享数据集中到最近公共父组件中，建立单一数据源，再通过“props 下发 + 回调上抛”完成协同更新。

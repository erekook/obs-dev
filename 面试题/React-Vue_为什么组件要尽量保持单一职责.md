# 题目
为什么前端组件设计里常说要让组件保持“单一职责”？如果一个组件职责过多，会带来什么问题？

# 回答要点
- 单一职责的含义：
  - 一个组件尽量只处理一类相对聚焦的功能
  - 展示、状态、业务逻辑边界清晰
- 好处：
  - 更容易复用
  - 更容易测试
  - 更容易维护和阅读
  - 更方便定位问题
- 职责过多的问题：
  - 组件臃肿
  - props 过多
  - 状态耦合严重
  - 修改一个需求容易影响其他功能
  - 渲染和逻辑优化变困难
- 实践思路：
  - 拆分容器组件与展示组件
  - 抽离公共 Hook / 逻辑层
  - 把大组件按功能块拆分
- 面试里可补充：
  - 单一职责不等于越碎越好
  - 组件拆分要考虑边界清晰和协作成本

# 追问
- 什么情况下组件拆得太碎反而不好？
- props 太多通常说明了什么设计问题？
- Hook 抽离和组件拆分分别适合解决什么问题？
- 单一职责为什么有助于性能优化？

# 参考答案
前端组件设计里强调“单一职责”，核心是为了让组件边界清晰。也就是说，一个组件最好只解决一类相对聚焦的问题，而不是把展示、状态、请求、校验、弹窗控制、列表逻辑全都糅在一起。

这样做最大的好处是可维护性。组件职责越单一，你看代码时越容易知道它到底负责什么；改动时也更容易判断影响范围，不会出现“改了一个小需求，却把别的功能顺手搞坏了”的情况。

第二个好处是复用性。一个职责清晰的组件更容易在不同页面、不同业务里被复用，而一个和具体页面逻辑深度耦合的大组件，往往只能在原地继续膨胀。

第三个好处是测试和调试更容易。因为组件功能聚焦，测试边界也会更清楚，出问题时你更容易快速定位到底是展示层、状态逻辑，还是外部依赖出了问题。

如果一个组件职责过多，常见表现就是：props 越来越多、状态越来越杂、内部 if/else 越来越多，读起来像一整页业务脚本。这样的组件不仅难维护，也更难做性能优化，因为任意一块逻辑变化都可能带动整棵组件树一起更新。

当然，单一职责也不等于无脑拆得越碎越好。拆分的关键不是“文件数量多”，而是边界是否自然。一个合理的做法通常是：把展示和业务逻辑分层，把复用逻辑抽成 Hook，把过大的组件按明确功能块拆开。

面试里如果总结，可以说：组件保持单一职责，本质上是为了让边界清晰、改动可控、复用容易；职责越混杂，耦合和维护成本就越高。

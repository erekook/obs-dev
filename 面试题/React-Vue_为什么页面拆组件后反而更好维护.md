# 题目
为什么把页面拆成多个小组件，通常会比把逻辑都堆在一个大组件里更好维护？这样做真正带来的好处是什么？

# 回答要点
- 拆组件的核心不是“文件变多”，而是职责边界更清晰。
- 好处包括：
  - 单个组件关注点更单一
  - 更容易理解和修改
  - 更方便复用
  - 更容易测试
  - 更利于局部优化
- 大组件常见问题：
  - 状态、UI、请求、事件全混在一起
  - 修改一点容易牵一大片
  - 重渲染范围和影响面更大
- 面试里可补充：
  - 拆组件不是越碎越好
  - 合理拆分的关键是按职责、数据边界和复用价值来切

# 追问
- 什么样的组件拆分算“过度拆分”？
- 按 UI 拆和按业务职责拆有什么区别？
- 为什么组件边界清晰后更利于性能优化？
- 拆组件后 props 太多怎么办？

# 参考答案
把页面拆成多个小组件，更好维护的核心原因不是“代码被分散了”，而是复杂度被切开了。

一个大组件里如果同时塞满了页面结构、表单状态、接口请求、弹窗逻辑、列表渲染、权限判断和事件处理，任何一个需求改动都可能牵动整片代码。你读起来累，改起来也容易误伤别的逻辑。

而拆成多个职责明确的小组件后，每个组件只关心自己那一小块事情。这样带来的第一个好处就是理解成本下降。你看一个组件时，不需要同时把整页所有状态都装进脑子里。

第二个好处是修改更安全。某个区域要改样式、改交互、改数据展示，通常只需要定位到那个组件，而不必在一个超大的文件里担心连带影响。

第三个好处是复用。如果一个筛选栏、表格头、弹窗表单、空状态区写得足够独立，后面别的页面也可能直接复用，而不是每次重新拷贝一份逻辑。

第四个好处是更利于测试和优化。组件边界清晰后，你更容易单独测试某一块输入输出，也更容易针对某一块做 memo、懒加载或局部渲染优化。

当然，拆组件也不是越碎越好。如果为了拆而拆，把一个很简单的 UI 硬拆成很多层，反而会让 props 层层传递、文件来回跳转，维护体验也会变差。所以关键不是“拆得多”，而是“按职责和边界拆得合理”。

面试里如果总结，可以说：拆组件的本质是把复杂度按职责切分，让每个部分更单一、可理解、可复用、可测试；它不是机械拆文件，而是通过清晰边界降低后续维护成本。

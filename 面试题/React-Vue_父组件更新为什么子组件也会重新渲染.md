# 题目
为什么在 React / Vue 里，父组件一更新，子组件常常也会跟着重新渲染？什么时候这种重新渲染是正常的，什么时候值得优化？

# 回答要点
- 父组件更新后，框架通常会重新执行父组件的渲染逻辑。
- 子组件为什么也会跟着走：
  - 子组件是父组件渲染结果的一部分
  - 框架需要重新判断子节点输出是否变化
- 这不等于：
  - 一定发生了大量真实 DOM 更新
  - 一定是性能问题
- 正常情况：
  - 数据依赖确实变化
  - 页面结构需要重新计算
- 值得优化的情况：
  - 子组件 props 没变却频繁重复渲染
  - 子树很大、渲染很重
  - 高交互场景导致无意义更新
- 常见优化手段：
  - React：`memo`、`useMemo`、`useCallback`
  - Vue：合理拆分组件、稳定 props、减少不必要响应式依赖
- 面试里可补充：
  - “重新渲染” 和 “重新挂载” 不一样
  - 优化前应先确认是否真有性能瓶颈

# 追问
- 为什么重新 render 不一定等于真实 DOM 重新创建？
- React.memo 为什么有时不生效？
- Vue 的响应式依赖是如何影响子组件更新范围的？
- 组件拆分为什么常能改善无意义更新？

# 参考答案
在 React 和 Vue 里，父组件一更新，子组件常常也会跟着重新渲染，这本质上是框架更新机制的正常表现。

原因很简单：子组件本来就是父组件渲染结果的一部分。父组件状态或 props 变化后，框架会重新执行父组件的渲染逻辑，而父组件在重新生成 UI 结构时，也需要重新判断它下面的子节点该长什么样。

但这里有一个很容易混淆的点：子组件“重新渲染”不等于真实 DOM 一定被全部重建。很多时候只是框架重新跑了一遍组件函数或模板计算，然后通过 Diff 发现真实 DOM 没什么变化，所以最终更新成本可能并不大。

所以，父更新子也跟着 render，本身并不一定是问题。如果子组件依赖的数据确实变了，这种更新完全正常。

真正值得优化的是另一类情况：比如子组件 props 没有变化，但因为父组件频繁更新，导致一棵很重的子树反复执行渲染逻辑；或者列表很多、交互很频繁，重复 render 已经开始影响性能。

这时候才需要优化。在 React 里常见手段是 `memo`、`useMemo`、`useCallback`；在 Vue 里则更强调合理拆分组件、控制响应式依赖范围，以及避免把无关状态耦合到一起。

所以面试里最好不要把“父组件一更新子组件也更新”直接说成 bug。更成熟的说法是：这是框架正常的更新传播机制，是否要优化，要看这些重新渲染有没有造成实际性能负担。

面试里如果总结，可以说：父组件更新时子组件跟着重新渲染是因为子节点属于父渲染结果的一部分，但这不一定意味着真实 DOM 大量更新；只有当无意义渲染带来明显性能成本时，才值得做针对性优化。

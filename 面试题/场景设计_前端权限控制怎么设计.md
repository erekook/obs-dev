# 题目
如果让你设计一个前端权限控制方案，你会怎么做？请从路由权限、按钮权限和安全边界几个角度来回答。

# 回答要点
- 前端权限控制的目标：提升体验、减少无权限操作入口，但不能替代后端鉴权。
- 常见权限粒度：
  - 路由级
  - 页面模块级
  - 按钮/操作级
- 路由权限：
  - 登录校验
  - 根据角色或权限码动态生成可访问路由
  - 无权限时跳转 403 或首页
- 按钮权限：
  - 根据权限码控制显示/禁用
  - 可封装指令、Hook、高阶组件或公共组件
- 权限数据来源：
  - 登录后由后端返回角色、菜单、权限码
  - 前端缓存并在刷新后恢复
- 安全边界：
  - 前端权限只做展示层控制
  - 真正的接口访问权限必须由后端校验
- 面试里可补充：
  - 菜单权限、路由权限、操作权限往往是一套权限模型的不同表现

# 追问
- 为什么说前端权限控制不能保证安全？
- 动态路由和静态路由权限控制各有什么优缺点？
- 如果权限在运行时发生变化，前端如何更新页面？
- 按钮是“隐藏”还是“禁用”更合适，为什么？

# 参考答案
设计前端权限控制时，我会先强调一个前提：前端权限控制的主要价值是优化用户体验和界面可见性，不能把它当成真正的安全防线。真正的安全校验必须由后端完成。

前端权限通常可以分成三层。第一层是路由权限，也就是用户能不能进入某个页面；第二层是页面模块权限，比如某个区块是否可见；第三层是按钮或操作权限，比如能不能新增、编辑、删除。

路由权限一般会在用户登录后，根据后端返回的角色、菜单或权限码，动态生成可访问路由。用户访问一个没有权限的页面时，可以跳到 403 页面或首页。这样用户从入口上就不会看到不该访问的页面。

按钮权限则更细粒度。比如同样在一个列表页里，有的用户能看但不能删，有的用户能新增和导出。这类权限一般会根据权限码控制按钮是否显示，或者是否禁用。工程上可以封装成指令、Hook、权限组件等统一使用，避免到处手写判断。

权限数据的来源通常是登录成功后由后端返回，比如角色列表、菜单树、功能码等。前端会把这些信息保存在状态管理里，并在刷新页面后恢复。

不过一定要强调，前端即使把按钮隐藏了，用户还是可能手动构造请求。所以后端接口必须再次校验用户是否真的有权限执行这个操作。前端权限控制只是“不给看、不好点”，后端鉴权才是“真的不让做”。

如果总结一下，一个完整的前端权限方案通常包括：登录态校验、路由动态控制、按钮级权限封装、权限数据统一管理，以及后端接口的最终鉴权兜底。面试里把“体验控制”和“安全边界”这两个层面区分清楚，会很加分。

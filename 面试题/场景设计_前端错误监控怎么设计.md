# 题目
如果让你设计一个前端错误监控方案，你会重点监控哪些错误？整体思路怎么搭？

# 回答要点
- 前端错误监控目标：
  - 及时发现线上问题
  - 辅助定位问题根因
  - 评估影响范围和优先级
- 常见监控对象：
  - JS 运行时错误
  - 资源加载错误
  - Promise 未捕获错误
  - 接口异常
  - 页面性能指标
  - 白屏/卡顿
- 采集手段：
  - `window.onerror`
  - `unhandledrejection`
  - 资源 error 事件
  - 请求拦截器
  - Performance API
- 上报内容：
  - 错误信息
  - 堆栈
  - 页面 URL
  - 用户环境
  - 用户操作上下文 / breadcrumb
  - 版本号
- 工程设计：
  - 去重与采样
  - 限流
  - source map 还原
  - 保护隐私，避免上报敏感信息
- 面试里可补充：
  - 监控不仅是“抓错误”，还要能告警、聚合、回溯和看趋势

# 追问
- 为什么需要 breadcrumb？
- source map 在线上错误定位里起什么作用？
- 如果错误量很大，怎么做采样和聚合？
- 白屏问题怎么监控，难点在哪里？

# 参考答案
设计前端错误监控时，我会先明确目标不是单纯“把错误打出来”，而是要让团队能发现问题、定位问题、衡量影响，并推动修复。

首先，监控范围不能只盯着 JS 报错。线上问题常见的还包括资源加载失败、接口异常、Promise 未捕获错误、页面卡顿、白屏、性能退化等。所以一个完整方案通常既要监控“显式报错”，也要监控“体验异常”。

采集层面，JavaScript 运行时错误可以通过 `window.onerror`，未捕获的 Promise 错误可以通过 `unhandledrejection`，资源加载错误可以监听对应的 error 事件，请求异常则可以在 fetch/axios 拦截器层统一采集。性能相关可以借助 Performance API、Web Vitals 等能力。

上报时不能只传一条错误 message。真正有价值的信息通常包括：错误堆栈、页面地址、用户浏览器和系统信息、当前版本号、接口信息，以及用户在出错前做了哪些关键操作，也就是 breadcrumb。这些上下文对排查问题非常重要。

工程上还要考虑几个现实问题。第一，线上错误可能很多，必须做去重、采样和限流，否则日志会被刷爆。第二，生产环境代码经过压缩混淆，必须配合 source map 才能把堆栈还原成可读源码位置。第三，采集时要避免把敏感信息，比如密码、token、隐私数据直接上报出去。

再进一步，一个成熟的错误监控系统通常不只是采集，还要有聚合、告警、趋势分析和版本对比能力。比如某个错误在新版本上线后突然激增，这类信息比单条日志更有价值。

面试里如果总结，可以说：前端错误监控要覆盖错误采集、上下文上报、源码还原、聚合告警和隐私控制，目标是把“线上问题”变成可观测、可定位、可追踪的工程问题。

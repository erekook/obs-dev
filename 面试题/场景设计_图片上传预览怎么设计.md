# 题目
如果让你设计一个“图片上传并即时预览”的功能，你会重点考虑哪些前端问题？

# 回答要点
- 功能不只是“选文件然后展示一下”，还涉及校验、性能和状态管理。
- 常见关注点：
  - 文件类型和大小校验
  - 本地预览方式
  - 上传进度展示
  - 失败重试与删除
  - 多图顺序管理
- 预览实现：
  - `FileReader`
  - `URL.createObjectURL`
- 体验细节：
  - 选中后立即看到缩略图
  - 上传中状态
  - 上传失败提示
  - 支持移除和重新选择
- 性能与安全：
  - 大图压缩/裁剪（必要时）
  - 及时 `revokeObjectURL`
  - 不只看文件后缀，要结合 MIME 类型校验
- 面试里可补充：
  - 前端预览和真正上传是两条相关但独立的流程

# 追问
- `FileReader` 和 `createObjectURL` 有什么区别？
- 为什么要及时 `revokeObjectURL`？
- 如果用户上传超大图片，前端能做哪些优化？
- 多图上传时如何处理部分成功、部分失败？

# 参考答案
设计图片上传并即时预览，前端最重要的是把“本地预览”和“真正上传”这两件事分开看。因为用户选中文件后，页面可以先立刻做本地预览，不需要等文件真的传到服务器。

第一步通常是文件校验。比如限制只允许图片类型、限制大小、限制数量。有些项目还会检查图片尺寸或长宽比。这里只看文件后缀不够稳，通常还要结合 MIME 类型一起判断。

第二步是本地预览。常见有两种方式：`FileReader` 和 `URL.createObjectURL`。前者会把文件读成 Data URL，后者则直接生成一个临时本地 URL。实际开发里，如果只是做图片预览，`createObjectURL` 往往更直接高效，但用完后要记得释放。

第三步是上传状态管理。预览出现后，用户还需要知道当前文件是“等待上传”“上传中”“上传成功”还是“上传失败”。如果支持多图上传，还要考虑顺序维护、单张删除、失败重试、部分成功部分失败等情况。

第四步是性能问题。如果用户上传的是超大图片，前端可以在合适场景下做压缩、裁剪或限制分辨率，避免直接把原图上传和展示导致卡顿或带宽浪费。

从交互上看，一个成熟的上传预览组件，通常不只是展示图，还要让用户能取消、替换、重新上传、查看进度，并且在失败时给清晰反馈。

面试里如果总结，可以说：图片上传预览的关键是把“本地预览、文件校验、上传状态、错误处理和性能优化”拆开设计，既保证即时反馈，也保证后续上传流程可控。

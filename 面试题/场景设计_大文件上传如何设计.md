# 题目
如果让你设计一个“大文件上传”方案，你会怎么做？请从前端、服务端配合、断点续传、秒传、失败重试和用户体验几个角度来回答。

# 回答要点
- 大文件上传的核心目标：稳定、可恢复、可观测、对用户友好。
- 前端常见方案：
  - 文件切片（chunk）
  - 为每个切片编号
  - 记录文件 hash 或唯一标识
  - 并发上传切片，但控制并发数
- 服务端配合：
  - 接收分片
  - 记录已上传分片
  - 全部分片完成后合并文件
  - 校验文件完整性
- 断点续传：
  - 重新上传前先查询服务端已有分片
  - 前端只补传缺失部分
- 秒传：
  - 前端计算文件 hash
  - 服务端判断该文件是否已存在
  - 若存在则直接返回成功或建立引用关系
- 失败重试：
  - 单个切片失败可重试
  - 避免整文件重新上传
  - 控制重试次数并给出错误提示
- 用户体验：
  - 展示整体进度和单片状态
  - 支持暂停/继续
  - 页面刷新后可恢复上传状态更好
- 安全与稳定性：
  - 校验类型、大小、权限
  - 避免前端仅靠后缀判断
  - 限制并发、防止服务端压力过大

# 追问
- 文件 hash 很大时计算很慢，前端如何优化？
- 为什么不能无限提高并发数来加快上传？
- 如果切片上传成功但合并失败，应该如何处理？
- 直传 OSS / S3 和先传业务服务器各有什么优缺点？

# 参考答案
设计大文件上传时，我会优先考虑两个问题：一是网络不稳定时能不能恢复，二是上传过程对用户是否友好。因为大文件最怕的是上传到一半失败后从头再来。

前端通常会先把文件切成多个分片，每个分片带上文件唯一标识、分片序号和总分片数，然后按一定并发数上传。这里不建议一次把所有切片都并发出去，而是要控制并发，比如 3 到 6 个，避免浏览器、网络和服务端压力过大。

为了支持断点续传，前端在上传前可以先向服务端查询当前文件已经成功上传了哪些分片，之后只上传缺失的部分。这样即使网络中断、页面关闭后重新进入，也不需要从头开始。

如果要支持秒传，一般会在前端计算文件 hash，上传前先把 hash 发给服务端。服务端检查这个文件是否已经存在，如果存在，就可以直接返回上传成功，或者只建立用户和文件之间的引用关系，而不必再次传输完整文件。

服务端需要负责保存分片、记录上传状态，并在所有分片都完成后进行合并。合并后最好再做一次完整性校验，比如校验 hash，确保文件没有损坏。

失败重试也很关键。应该以“切片”为单位重试，而不是整个文件重传。这样成本更低，用户体验也更好。如果某些切片多次重试仍失败，需要给用户明确提示。

从用户体验上，最好展示整体进度、上传速度、剩余时间，并支持暂停和继续。更进一步的话，可以把上传状态持久化到本地，这样刷新页面后还能恢复。

如果总结一下，一个完整的大文件上传方案通常包括：切片上传、服务端分片记录、断点续传、秒传判断、失败重试、最终合并和校验。这几个点答全，面试官一般会觉得你的设计比较成熟。

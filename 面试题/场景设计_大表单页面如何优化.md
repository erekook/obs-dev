# 题目
如果一个页面上有很多表单项，输入会卡、校验也复杂，你会怎么优化这个大表单页面？

# 回答要点
- 大表单问题通常来自：
  - 状态过多
  - 每次输入引发整页重渲染
  - 联动逻辑复杂
  - 校验时机不合理
- 优化方向：
  - 拆分表单模块/子组件
  - 减少无关区域重渲染
  - 局部状态管理
  - 延迟或分时校验
  - 合理使用受控/非受控方案
- 渲染优化：
  - 组件 memo
  - 精准订阅表单状态
  - 避免父组件持有全部频繁变化状态
- 校验优化：
  - `blur` 时校验而不是每次输入都重校验
  - 异步校验做防抖
  - 区分实时校验和提交校验
- 用户体验：
  - 分步表单
  - 自动保存草稿
  - 错误定位和滚动到首个错误项
- 面试里可补充：
  - 表单库选型也很关键，比如 React Hook Form、Formik、Antd Form 思路不同

# 追问
- 为什么大表单全量受控时容易卡顿？
- React Hook Form 为什么在大表单场景里常被认为更轻？
- 异步校验用户名可用性时如何避免频繁请求？
- 分步表单和单页大表单各有什么优缺点？

# 参考答案
大表单页面卡顿，通常不是单个输入框的问题，而是“状态管理、渲染范围、校验策略、交互设计”几个问题叠在一起。

首先是渲染层面。很多大表单一输入就卡，是因为所有字段都放在一个大组件里，任意一个字段变化都会导致整页重渲染。所以第一步通常是拆分组件，把表单按模块拆开，尽量让每个输入只影响自己相关的区域。

第二是状态管理。不是所有字段都一定要做强受控。如果是非常大的表单，全部用 React state 严格受控，输入时会带来很高的更新成本。某些场景下可以结合更轻量的表单库，或者局部使用非受控思路，减少频繁同步状态带来的性能压力。

第三是校验策略。很多项目会在用户每敲一个字符时做复杂校验甚至发异步请求，这很容易卡。更合理的做法是把校验分层，比如格式类校验可以轻量实时做，但复杂校验放到 `blur` 或提交时再做；异步校验则应加防抖，并处理好竞态问题。

第四是用户体验。一个字段非常多的单页表单，用户本身就容易疲劳。可以考虑按步骤拆分，或者分区折叠显示。提交失败时也要能自动定位到第一个错误项，避免用户自己满屏找问题。

如果表单很长，自动保存草稿也是一个不错的优化，它不一定提升性能，但能显著提升体验。

面试里如果总结，可以说：大表单优化要从“减少无关重渲染、降低状态同步成本、优化校验时机、改进表单交互结构”四个方向一起做，而不是只盯着某一个输入框性能。

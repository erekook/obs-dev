# 题目
如果接口返回很慢，前端页面应该怎么设计，才能既不“假装没问题”，又不让用户一直干等？

# 回答要点
- 慢接口不仅是技术问题，也是体验设计问题。
- 常见设计手段：
  - 明确加载状态
  - 骨架屏 / 占位
  - 分段渲染，先展示可先到的内容
  - 超时提示或重试入口
  - 缓存旧数据并标记刷新中
- 核心原则：
  - 让用户知道系统在工作
  - 尽量减少“整页卡死”感
  - 给用户下一步选择
- 场景区分：
  - 关键数据必须等
  - 非关键数据可延后
  - 实时性要求不高时可先展示旧数据
- 面试里可补充：
  - 不要把整个页面完全绑死在一个慢接口上
  - 体验优化和技术优化应一起考虑

# 追问
- 为什么骨架屏通常比纯 loading 更好？
- 什么情况下可以先展示旧数据？
- 为什么“整页白屏等接口”体验很差？
- 如果接口超时，前端除了报错还能做什么？

# 参考答案
接口返回很慢时，前端不应该只是单纯地“放一个 loading 一直转”，因为这会让用户不知道系统到底还在处理，还是已经卡住了。更好的思路是，把“等待过程”设计成一个有反馈、有层次、有退路的体验。

第一层是明确反馈。页面至少要告诉用户：请求已经发出，现在正在加载什么内容。如果完全没有状态提示，用户往往会误以为页面坏了。

第二层是减少“整页不可用”的感觉。很多时候并不是所有内容都必须等那个慢接口返回后才能显示。能先展示的结构、静态信息、缓存内容、局部模块，都可以先出来，让用户感觉页面是在逐步完成，而不是整页卡死。

第三层是给用户选择。如果等待时间比较长，应该考虑提供超时提示、重试按钮、稍后再试、或者先展示旧数据并标记“正在刷新”。这样用户不会被动困在一个无限 loading 里。

第四层是按业务重要性分层。如果某块数据不是首屏关键路径，就不应该拖住整个页面。真正必须阻塞的，通常只应该是少数关键数据；非关键内容可以懒加载、延迟展示或者异步补齐。

如果页面本身适合缓存策略，比如列表页、详情页、配置页等，还可以先展示上一次成功数据，再后台刷新。这种体验往往比从零开始空白等待更平滑。

所以面试里如果总结，可以说：慢接口场景下，前端重点不是“掩盖慢”，而是通过加载反馈、分段展示、重试机制和旧数据兜底，把用户等待从“无感知的卡住”变成“可理解、可接受、可操作”的过程。

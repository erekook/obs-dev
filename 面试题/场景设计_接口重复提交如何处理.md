# 题目
如果一个表单或按钮触发的接口容易被用户重复提交，你会怎么设计防重方案？请从前端、后端和用户体验几个角度来回答。

# 回答要点
- 重复提交常见原因：
  - 用户快速连点
  - 网络慢导致误以为没点上
  - 页面重试或请求重放
- 前端方案：
  - 按钮点击后立即禁用
  - 增加 loading 状态
  - 请求未完成前拦截重复触发
  - 防抖 / 节流（按场景使用）
- 后端方案：
  - 幂等设计
  - 唯一请求标识（requestId / token）
  - 业务唯一约束
- 用户体验：
  - 清晰的提交中反馈
  - 成功失败提示
  - 网络慢时避免用户误操作
- 面试里可补充：
  - 仅靠前端不够安全
  - 真正可靠的防重必须后端兜底

# 追问
- 什么是幂等？哪些接口天然适合设计成幂等？
- 防抖和按钮禁用在提交场景里分别解决什么问题？
- 为什么只做前端防重不够？
- 如果用户刷新页面导致请求重发，如何处理？

# 参考答案
接口重复提交是一个很典型的业务问题，真正成熟的方案一定是“前端减少重复触发，后端保证最终幂等”。

从前端看，最直接的方式是用户点击提交按钮后，立刻把按钮置为 loading 或禁用状态，在请求返回之前不允许再次点击。这能挡住最常见的“快速连点”问题。

如果是某些高频触发操作，还可以结合防抖或节流。但要注意，防抖和节流更多是控制事件触发频率，不等于完整的防重提交方案。对于真正的表单提交，按钮禁用和请求中锁通常更直接。

不过，只靠前端是不够的。因为用户可能开多个标签页、重复发请求、刷新页面重试，甚至恶意构造请求，所以后端必须做幂等控制。常见做法有两种：一种是给每次提交生成唯一请求标识，请求到后端时校验这个标识是否已经处理过；另一种是通过业务唯一约束来防止重复数据写入，比如同一用户、同一订单、同一时间窗内只允许生成一条记录。

从用户体验角度，还要给明确反馈。比如按钮进入“提交中”状态，成功后提示已提交，失败后允许重试。如果网络很慢，但页面没有任何反馈，用户就很容易多次点击。

所以如果面试里回答，我会强调三层：第一层前端通过 loading、禁用按钮、请求锁减少重复触发；第二层后端通过幂等和唯一约束兜底；第三层通过明确反馈降低用户误操作。

一句话总结就是：防重提交不能只靠前端拦截，真正可靠的是前端控频 + 后端幂等 + 明确交互反馈。

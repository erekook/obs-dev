# 题目
如果一个前端接口请求失败了，什么时候应该自动重试？重试机制应该怎么设计才更合理？

# 回答要点
- 不是所有失败都应该重试，要先区分失败类型。
- 适合自动重试的情况：
  - 网络抖动
  - 短暂超时
  - 临时性服务不可用（如 502、503、504）
- 不适合盲目重试的情况：
  - 业务参数错误
  - 权限错误
  - 资源不存在
  - 明确的客户端错误（如多数 4xx）
- 设计要点：
  - 限制最大重试次数
  - 间隔递增（指数退避）
  - 可加随机抖动避免雪崩
  - 区分幂等请求和非幂等请求
- 用户体验：
  - 重试中要有状态提示
  - 最终失败要可感知并允许手动重试
- 面试里可补充：
  - 自动重试不是“越积极越好”，重点是控制风险和避免放大问题

# 追问
- 为什么 POST 类请求重试要更谨慎？
- 什么是指数退避，为什么要加抖动？
- 如果接口已经超时，但服务端其实成功处理了，会有什么风险？
- 前端自动重试和后端幂等设计应该如何配合？

# 参考答案
前端接口失败后，是否自动重试，关键不在于“请求失败了没”，而在于“这个失败是不是临时性的、重试有没有意义、重试会不会带来副作用”。

像网络抖动、弱网短暂超时、服务端临时不可用（比如 502、503、504）这类问题，通常是适合自动重试的。因为这些错误往往不是业务本身有问题，而是链路暂时不稳定，稍后重试成功的概率比较高。

但如果是参数错误、权限不足、资源不存在、表单校验失败这类明确的客户端或业务错误，盲目重试通常没有意义，只会增加无效请求。

设计重试机制时，第一要控制次数，不能无限重试。第二要控制节奏，通常不会立刻连续重试，而是用递增间隔，比如指数退避。这样可以避免系统在异常时被前端请求雪上加霜地压垮。更进一步，还可以加一点随机抖动，让大量客户端不要在同一时间点一起重试。

另一个很重要的点是区分请求类型。像 GET 这种读请求通常更适合自动重试；而像创建订单、支付、提交表单这类可能产生副作用的请求，重试必须非常谨慎，否则可能造成重复提交。这里往往还需要后端提供幂等支持。

从用户体验看，自动重试最好不要完全黑箱。用户至少应该知道当前在重试中，如果最终还是失败，也要给清晰提示，并允许手动再试。

面试里如果总结，可以说：合理的重试机制是“只对临时性错误重试、限制次数、使用退避策略、谨慎处理非幂等请求”，重点不是多试几次，而是既提高成功率，又不放大风险。

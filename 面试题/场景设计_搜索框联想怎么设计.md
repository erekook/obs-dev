# 题目
如果让你设计一个“搜索框联想”功能，你会重点考虑哪些问题？请从请求控制、用户体验和性能优化几个方面来回答。

# 回答要点
- 搜索联想目标：响应快、结果准、请求稳、交互自然。
- 请求控制：
  - 输入防抖
  - 长度阈值控制
  - 空字符串不请求或返回默认推荐
  - 取消过期请求 / 忽略旧响应
- 用户体验：
  - loading 状态
  - 无结果提示
  - 键盘上下选择
  - 回车搜索
  - 高亮命中关键词
  - 点击外部收起
- 性能优化：
  - 请求缓存
  - 高频关键词复用
  - 结果列表虚拟化（结果很多时）
  - 控制渲染频率
- 正确性：
  - 处理响应乱序
  - 避免旧关键字结果覆盖新关键字结果
- 面试里可补充：
  - 前端只是体验层，召回和排序逻辑核心仍在后端搜索服务

# 追问
- 为什么搜索联想通常更适合防抖而不是节流？
- 如何解决“请求后发先至”导致结果错乱的问题？
- 本地缓存联想结果时要注意什么？
- 如果要支持拼音、模糊匹配，这部分更适合前端还是后端做？

# 参考答案
设计搜索框联想时，我会先关注三个点：请求不要乱发、结果不要乱序、用户操作要顺手。

请求控制上，最常见的是做防抖。因为用户连续输入时，如果每敲一个字都立刻请求，既浪费资源，也容易让结果不停跳动。通常会等用户停顿一小段时间再发请求，比如 200 到 300ms。还可以加最小长度限制，比如输入少于 2 个字符时不请求，或者返回默认推荐词。

另一个关键问题是响应乱序。比如用户先输入 “re”，又很快输入 “react”，如果 “re” 的请求后返回，就可能把“react”的结果覆盖掉。所以要么取消旧请求，要么给每次请求打标记，只处理最后一次有效响应。

用户体验方面，除了展示联想结果，还要处理很多细节。比如输入时显示 loading，没有结果时给明确提示，支持键盘上下选择和回车确认，匹配词高亮，点击输入框外部收起面板，这些都会直接影响是否“像一个成熟产品”。

性能上，可以对相同关键词做短期缓存，减少重复请求；如果联想结果很多，也要控制渲染开销。大多数搜索联想不会多到必须上虚拟列表，但如果结果条目较多，也可以考虑。

如果后端能力更强，真正的模糊匹配、拼音搜索、纠错、排序等能力通常还是应该放在后端搜索服务里实现，前端主要负责输入控制、请求管理和结果展示。

面试里可以总结成一句话：搜索联想不是“输入就请求”这么简单，关键在于防抖、乱序处理、交互细节和结果展示体验。

# 题目
如果做一个搜索结果页，分页和无限滚动应该怎么选？前端设计上要考虑哪些取舍？

# 回答要点
- 两者都能分批展示数据，但适用场景不同。
- 分页更适合：
  - 结果有明确页码语义
  - 用户需要定位、跳转、回到某一页
  - 适合后台管理、文档检索等
- 无限滚动更适合：
  - 信息流式浏览
  - 用户主要是“往下逛”而不是精确定位
  - 适合内容社区、电商推荐流等
- 设计取舍：
  - 用户控制感 vs 连续浏览体验
  - SEO / 可分享性
  - 滚动位置恢复
  - 请求加载策略
- 前端关注点：
  - 加载状态
  - 去重和竞态处理
  - 回退后恢复现场
  - 长列表性能优化
- 面试里可补充：
  - 有时“分页 + 加载更多”是折中方案

# 追问
- 为什么搜索页常比信息流更适合分页？
- 无限滚动在返回上一个页面时会有哪些体验问题？
- 为什么分页页码更利于 SEO 和分享？
- 什么时候“加载更多”比纯无限滚动更合理？

# 参考答案
搜索结果页选分页还是无限滚动，关键不在于“哪种更现代”，而在于用户在这个页面上的目标到底是什么。

如果用户的核心目标是查找、定位、比较，并且希望能明确知道“第几页、第几条结果”，分页通常更合适。比如文档检索、后台数据查询、招聘职位搜索这类场景，用户更在意可定位性、可回溯性和结果边界。

而如果用户主要是在浏览内容、不断往下看，比如推荐流、社区内容流、电商逛商品，无限滚动会更自然，因为它减少了翻页动作，让浏览更连续。

从前端设计角度看，分页的优势是状态更稳定。页码明确，URL 也更容易表达当前结果位置，刷新、分享、回退、SEO 都更友好。无限滚动虽然浏览顺滑，但会带来滚动位置恢复、历史记录管理、长列表性能、重复请求和“用户到底看到了哪里”这些额外问题。

还有一个现实取舍是用户控制感。分页给用户很强的“边界感”和“我现在在哪”的感受；无限滚动则更强调沉浸式连续浏览，但在找回特定位置时通常更麻烦。

有些产品会用折中方案，比如“分页 + 每页较大数量”，或者“第一页先分页，内部再支持加载更多”，又或者直接提供一个“加载更多”按钮，让用户既保留控制感，又避免页码跳转太生硬。

所以面试里如果回答，我会强调：搜索结果页如果偏“查找和定位”，优先考虑分页；如果偏“浏览和消费”，无限滚动更自然。最终选择要结合用户目标、SEO、回退恢复和长列表性能一起权衡。

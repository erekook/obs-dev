# 题目
如果让你设计一个“无限滚动列表”页面，你会重点考虑哪些问题？请从数据加载、性能优化、用户体验和异常处理几个方面来回答。

# 回答要点
- 核心目标：滚动体验流畅、加载稳定、数据一致、可恢复。
- 数据加载：
  - 分页或 cursor 分页
  - 触底加载或接近底部预加载
  - 避免重复请求、并发请求和乱序覆盖
- 性能优化：
  - 列表虚拟化
  - 图片懒加载
  - 节流滚动事件或使用 `IntersectionObserver`
  - 避免一次渲染过多 DOM
- 用户体验：
  - loading 状态
  - “没有更多了” 提示
  - 保持滚动位置
  - 返回列表时恢复现场
  - 首屏骨架屏/占位
- 异常处理：
  - 请求失败重试
  - 去重
  - 接口慢或超时提示
  - 空状态处理
- 数据一致性：
  - 防止翻页过程中数据插入导致重复或漏项
  - cursor 分页通常比 page 分页更稳
- 可访问性与交互补充：
  - 是否需要回到顶部
  - 是否提供分页备用入口
  - 对 SEO 页面要谨慎使用纯无限滚动

# 追问
- 为什么 cursor 分页在信息流场景里通常更合适？
- 列表虚拟化会带来哪些副作用？
- 如果滚动触发加载过于频繁，怎么优化？
- 无限滚动和传统分页各有什么优缺点？

# 参考答案
设计无限滚动列表时，我会先考虑四件事：怎么稳定加载数据、怎么保证滚动性能、怎么让用户体验清晰、以及接口异常时怎么兜底。

数据加载方面，最基础的是分页策略。简单场景可以用 page + pageSize，但如果是内容会动态插入的信息流，cursor 分页通常更稳，因为它能减少翻页过程中重复和漏项问题。前端触发加载时，也要避免重复请求和并发请求，比如在请求未结束时加锁，防止用户快速滚动导致连续触发。

性能上，无限滚动最大的风险是 DOM 越积越多，所以列表虚拟化很重要，只渲染可视区域附近的内容。再配合图片懒加载、滚动事件节流，或者直接用 `IntersectionObserver` 监听底部哨兵元素，通常会更省性能。

用户体验方面，不能只是“滚到了底部就悄悄请求”。应该有明确的加载中状态、加载失败提示、没有更多数据的提示。如果用户从详情页返回列表，最好还能恢复之前的滚动位置和已加载内容，否则体验会很差。

异常处理也很关键。比如请求失败时要能重试；接口超时时要提示；如果后端数据在翻页期间发生变化，要考虑前端去重，避免一条内容出现两次。

如果是更复杂的业务，还要考虑 SEO 和可访问性。很多内容站不会只依赖无限滚动，而是会保留分页入口或“加载更多”按钮，兼顾爬虫和用户控制感。

总结来说，一个成熟的无限滚动方案，不只是“滚动到底加载下一页”，而是要同时兼顾分页策略、渲染性能、状态反馈、异常恢复和数据一致性。

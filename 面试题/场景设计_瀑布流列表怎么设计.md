# 题目
如果让你设计一个瀑布流列表页面，你会重点考虑哪些前端问题？

# 回答要点
- 瀑布流不只是排版问题，还涉及数据加载、性能和滚动体验。
- 核心点：
  - 多列布局算法
  - 图片加载后高度变化处理
  - 滚动加载策略
  - 性能优化
- 布局实现思路：
  - CSS 多列 / Grid（简单场景）
  - JS 计算最短列分配（更可控）
- 常见难点：
  - 图片未加载前高度未知
  - 布局抖动
  - 重排成本高
  - 长列表下 DOM 过多
- 优化方向：
  - 图片预留比例或占位骨架
  - 懒加载图片
  - 无限滚动配合请求节流
  - 长列表场景可考虑虚拟化或分段渲染
- 面试里可补充：
  - 瀑布流体验关键是“滚动连续性”和“布局稳定性”

# 追问
- 为什么图片高度未知会让瀑布流更难做？
- CSS 多列和 JS 算法布局各有什么优缺点？
- 瀑布流和普通网格列表在性能优化上有哪些不同？
- 如果用户快速滚动到底部，如何避免重复加载请求？

# 参考答案
设计瀑布流列表时，前端真正要解决的，不只是“把卡片摆成几列”，而是让它在图片高度不一致、数据不断追加的情况下，依然保持布局稳定和滚动流畅。

首先是布局策略。最简单的做法可以用 CSS 多列或某些 Grid 思路实现，但如果你希望更精确地控制每个卡片落在哪一列，常见做法还是用 JavaScript 维护每一列当前高度，把新卡片分配到“当前最短的那一列”。这样整体视觉会更均衡。

第二个关键问题是图片高度。很多瀑布流卡片在图片加载前并不知道真实高度，如果没有预留尺寸，图片一加载出来就会把后面的布局顶来顶去，产生明显抖动。所以通常会结合图片宽高比信息，提前占位，或者用骨架屏保持布局稳定。

第三是数据加载。瀑布流通常会配合无限滚动，所以需要在滚动到底部附近时加载下一页，同时做好请求节流、防重复请求和“没有更多数据”提示。

第四是性能。因为瀑布流列表往往项目很多、图片很多，所以要特别注意图片懒加载、滚动监听优化，以及长列表带来的 DOM 数量问题。数据量非常大时，还要考虑是否需要分段渲染或某种程度的虚拟化。

从体验角度看，瀑布流最重要的不是列数，而是用户滚动过程中不能觉得页面在乱跳、卡顿或突然大片空白。所以布局稳定性和滚动连续性，比“算法优雅”更关键。

面试里如果总结，可以说：瀑布流的核心在于多列分配、图片高度处理、滚动加载和性能控制，重点不是把元素摆开，而是让异步图片和持续追加数据的场景下，页面依然稳定流畅。

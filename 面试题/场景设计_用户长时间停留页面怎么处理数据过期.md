# 题目
如果用户在一个页面停留很久，数据可能已经过期了，前端通常应该怎么处理这种情况？

# 回答要点
- 长时间停留会带来：
  - 数据过期
  - 权限或登录态变化
  - 列表内容与真实状态不一致
- 常见处理方式：
  - 页面回到前台时刷新关键数据
  - 定时轮询或后台刷新
  - 手动刷新提示
  - 数据失效标记与重新拉取
- 场景区分：
  - 实时性要求高的页面适合主动刷新
  - 普通信息页可采用轻提示或按需刷新
- 关键点：
  - 不能盲目高频请求
  - 要平衡实时性、成本和用户打断感
- 面试里可补充：
  - 可结合 `visibilitychange`、聚焦事件、缓存失效时间（stale time）等机制

# 追问
- 为什么页面切回前台是一个很常见的刷新时机？
- 轮询和用户手动刷新各有什么优缺点？
- 什么场景更适合 stale-while-revalidate 思路？
- 如果数据刷新后和用户当前操作冲突，该怎么处理？

# 参考答案
用户长时间停留在页面上，前端一个很现实的问题就是：页面里展示的数据可能早就不是最新的了。这个问题在列表页、详情页、审批页、实时监控页都很常见。

处理思路首先要看页面的实时性要求。如果是监控面板、订单状态、客服会话这类对实时性要求高的页面，前端通常会做主动刷新，比如定时轮询、WebSocket、或者页面重新回到前台时立即重新拉取关键数据。

如果是普通信息页，没必要一直高频请求。这种场景更常见的做法是：当用户切回页面、重新聚焦窗口、重新触发某个操作时，再检查数据是否已经 stale，如果过期就自动刷新，或者提示用户“数据可能已更新，点击刷新”。

这里一个很常用的时机就是 `visibilitychange`。因为用户从别的标签页或应用切回来时，很可能已经过了一段时间，这时候刷新关键数据通常比较合理，也更符合用户直觉。

另外，前端还可以结合数据缓存策略。比如设置一个 stale time，在一定时间内认为数据可直接使用，超过时间后就标记为过期，下一次合适时机再后台刷新。这种方式通常比无脑轮询更平衡。

不过也要注意用户体验。如果用户正在编辑表单、勾选列表、处理审批流程，这时后台突然把整页数据强刷掉，反而会打断操作。所以刷新策略还要考虑当前交互上下文，必要时只更新局部，或者先提示用户有新数据可刷新。

面试里如果总结，可以说：页面长时间停留后的数据过期问题，关键是按场景选择“主动刷新、切回前台刷新、轻提示刷新”哪种策略，并平衡实时性、请求成本和用户当前操作体验。

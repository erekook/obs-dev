# 题目
如果前端项目里用户登录态过期了，你会怎么设计处理方案？请从接口拦截、用户体验和安全性几个角度来回答。

# 回答要点
- 登录态过期是常见场景，要兼顾体验和安全。
- 识别方式：
  - 接口返回 401/403
  - 业务状态码标识 token 失效
- 前端常见处理：
  - 统一请求拦截
  - 清理本地登录态
  - 跳转登录页或弹登录失效提示
- 刷新 token 方案：
  - access token 短期有效
  - refresh token 用于换新 access token
  - 刷新成功后重放原请求
- 并发请求问题：
  - 多个请求同时过期时避免重复刷新
  - 可用队列或锁统一等待刷新结果
- 用户体验：
  - 尽量无感刷新
  - 刷新失败后再提示重新登录
  - 保留当前页面路径，登录后可回跳
- 安全性：
  - token 过期时间合理设置
  - refresh token 更严格保护
  - 退出登录时彻底清理状态
- 面试里可补充：
  - 前端只是协作方，真正的过期校验在服务端

# 追问
- 为什么不建议把 access token 设置得特别长？
- 多个请求同时触发刷新时，如何避免发多个 refresh 请求？
- refresh token 也过期了怎么办？
- 如果接口请求正在上传文件，token 过期后怎么处理更合理？

# 参考答案
登录态过期处理，核心是三个目标：统一、平滑、安全。

首先，前端一般不会自己判断 token 是否绝对有效，而是以服务端返回结果为准。常见情况是接口返回 401，或者返回某个业务状态码表示登录已失效。这时候最好在请求层统一处理，而不是每个页面分别写一套逻辑。

常见做法是使用请求拦截器和响应拦截器。当发现 access token 过期时，如果系统有 refresh token 机制，前端会先拿 refresh token 去换一个新的 access token。刷新成功后，把之前失败的请求重新发送，这样用户很多时候是无感知的。

这里有个常见难点是并发请求。假设页面同时发了 5 个请求，它们都返回 token 过期，如果每个请求都各自去刷新一次，就会造成重复刷新甚至状态混乱。所以更合理的方案是做一个“刷新锁”或者请求队列，保证同一时刻只有一个刷新请求在执行，其他失败请求先挂起，等刷新成功后再统一重试。

如果 refresh token 也失效了，或者刷新失败，那就应该清理本地登录态，跳转到登录页，并提示用户重新登录。为了体验更好，通常还会保留当前页面地址，登录成功后再跳回原页面。

安全上，access token 一般不应该有效期太长，因为一旦泄露风险会持续很久；refresh token 的保护要更严格，退出登录时也要确保彻底清理。

如果总结一下，一个成熟的登录态过期方案通常包括：统一拦截识别、无感刷新、失败重试、并发控制、失败后强制重新登录，以及登录后回跳恢复。面试里把这些点讲全，就比较完整。

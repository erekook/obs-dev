# 题目
如果让你设计一个购物车页面，商品数量变化时总价、优惠、选中状态都要实时联动，你会怎么设计前端数据结构和更新逻辑？

# 回答要点
- 这是典型的状态建模和派生数据问题。
- 核心思路：
  - 原始状态尽量最小化
  - 总价、已选数量、优惠金额等优先做派生计算
- 常见原始状态：
  - 商品列表
  - 每项数量
  - 每项选中状态
  - 优惠规则/活动信息
- 派生状态：
  - 已选商品总价
  - 已选商品数量
  - 满减结果
  - 最终应付金额
  - 全选状态
- 更新逻辑建议：
  - 单一数据源
  - 通过 reducer / store / computed 统一派生
  - 避免把总价直接当作可独立修改状态保存
- 性能与体验：
  - 局部更新
  - 防止不必要重渲染
  - 异步接口失败时状态回滚
- 面试里可补充：
  - 金额计算注意精度问题
  - 大型购物车场景可结合虚拟列表和分组状态管理

# 追问
- 为什么总价更适合做派生值而不是单独维护？
- 如果商品勾选和数量更新很频繁，如何减少重渲染？
- 满减和优惠券叠加规则复杂时，前端如何保持逻辑可维护？
- 如果更新数量时接口失败，前端状态怎么处理更合理？

# 参考答案
购物车这种场景，本质上考的是“如何设计状态，避免联动逻辑失控”。

我会先把原始状态控制在最小集合，只存真正需要被用户直接修改的数据。比如商品列表、每个商品的数量、选中状态、以及后端返回的价格和优惠规则。这些是原始输入。

像总价、选中商品数、优惠金额、应付金额、全选状态，这些都更适合做派生数据，而不是单独当成可写状态维护。因为如果你把它们也作为独立状态保存，就很容易在多个地方更新时出现不一致。

比如用户改了一个商品数量，如果你既要改商品数据，又要手动改总价、已选数量、优惠金额，就会让逻辑非常脆弱。更稳的方式是：商品数据一变，其他值统一通过计算得出。

在实现上，可以用 reducer、状态管理库，或者在 Vue 里用 computed、在 React 里用 memo/selectors 这类方式，统一从原始状态推导出这些结果。

如果页面比较大，还要注意渲染性能。比如单个商品数量变化时，不应该导致整个购物车所有项都重渲染；金额计算要注意浮点精度问题，最好统一转成最小货币单位再算。

另外，购物车通常还会有和后端同步的问题。比如用户点加号后，前端先乐观更新数量，但如果接口失败，需要能提示错误并回滚到之前状态。

面试里如果总结，可以说：购物车联动的关键不是“怎么把总价算出来”，而是“把原始状态和派生状态分清楚”，用单一数据源驱动所有联动结果，这样逻辑才稳定、可维护。

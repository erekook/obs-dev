# 题目
如果一个页面里可能同时出现多种弹窗、抽屉、确认框，前端通常该怎么管理，才能避免状态越来越乱？

# 回答要点
- 多弹窗问题本质上是“状态管理和层级管理”问题。
- 常见混乱来源：
  - 每个弹窗一个零散布尔值
  - 打开关闭逻辑分散
  - 弹窗之间互斥关系不清楚
  - 传参和回调层层嵌套
- 常见优化思路：
  - 统一管理弹窗状态
  - 用“当前弹窗类型 + 参数”代替大量布尔值
  - 明确互斥规则和优先级
  - 把通用弹窗能力抽象成统一入口
- 面试里可补充：
  - 简单页面不一定要上复杂方案
  - 关键是状态模型要和业务复杂度匹配

# 追问
- 为什么多个布尔值很容易失控？
- 什么情况下适合做全局弹窗管理？
- 弹窗参数和回调怎么设计更清晰？
- 弹窗很多时，如何避免“打开一个关不掉另一个”？

# 参考答案
一个页面里弹窗、抽屉、确认框一多，代码很容易失控。根本原因通常不是“弹窗组件难写”，而是状态模型没设计好。

最常见的问题是：每个弹窗都配一个 `visible` 布尔值，页面里堆着 `isEditOpen`、`isDeleteOpen`、`isPreviewOpen`、`isExportOpen`…… 一开始还行，后面一多，谁和谁能同时开、谁打开前要先关谁、每个弹窗需要什么参数，就会越来越乱。

更稳的做法通常是把它当成一个“当前激活的是哪种弹窗”的状态来管理。也就是用“弹窗类型 + 参数”代替大量分散布尔值。比如当前是编辑弹窗，带上某条记录 id；或者当前是删除确认框，带上待删除对象信息。这样状态模型会更集中，也更容易推理。

如果页面复杂，甚至多个地方都可能触发弹窗，还可以进一步抽成统一入口，比如一个弹窗管理器，让打开、关闭、传参、确认回调的方式都统一。这样比在页面各处散落 `setState` 更容易维护。

另外，多弹窗场景一定要明确规则：哪些可以同时存在，哪些必须互斥，优先级怎么定。很多 bug 其实不是组件本身有问题，而是业务层没定义清楚层级关系。

当然，也不是所有页面都需要做成全局弹窗系统。简单页面用局部状态就够，过度抽象反而会增加理解成本。关键是状态设计要跟复杂度匹配。

面试里如果总结，可以说：多弹窗管理的重点不是“怎么多写几个 visible”，而是建立清晰的状态模型；通常用统一的弹窗类型、参数和互斥规则来管理，会比大量分散布尔值更稳、更好维护。

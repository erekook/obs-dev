# 题目
请你讲一下 Babel、SWC、esbuild 分别是什么。它们主要区别在哪里，实际项目里怎么选？

# 回答要点
- 三者都和前端代码编译/转换有关，但定位和侧重点不同。
- Babel：
  - 生态成熟
  - 插件体系丰富
  - 转换能力强
  - 速度相对慢一些
- SWC：
  - Rust 实现
  - 速度快
  - 可替代部分 Babel / TypeScript 转译工作
  - 插件生态相对没 Babel 成熟
- esbuild：
  - Go 实现
  - 极快
  - 擅长构建、打包和快速转译
  - 功能侧重速度与工程效率
- 常见选择思路：
  - 需要复杂语法插件、成熟生态时倾向 Babel
  - 追求更快构建和主流能力时可考虑 SWC / esbuild
  - 很多工具链会混合使用，不一定非选其一
- 面试里可补充：
  - TypeScript 转译不等于类型检查
  - 速度和生态通常是主要权衡点

# 追问
- 为什么 Babel 生态更成熟？
- esbuild 为什么快？
- SWC 能完全替代 Babel 吗？
- 为什么很多项目仍然保留 Babel，即使已经用了更快的构建工具？

# 参考答案
Babel、SWC、esbuild 都可以处理前端代码的转换工作，但它们的优势和定位不太一样。

Babel 是最经典、历史最久的 JavaScript 转译工具。它最大的优势是生态成熟，插件和 preset 非常丰富，很多语法转换、框架支持、定制需求都能找到成熟方案。所以如果项目依赖比较复杂的语法插件，或者对兼容性和生态完整度要求高，Babel 仍然很常见。

SWC 是用 Rust 实现的编译工具，目标之一就是在保证主流能力的前提下，比 Babel 更快。它可以承担很多 Babel 和 TypeScript 转译层面的工作，所以在现代工程化里越来越常被拿来替代部分 Babel 职责。

esbuild 则是用 Go 实现的，最大的标签就是“非常快”。它不仅能做代码转译，还很擅长构建和打包，因此在 Vite 这类现代工具链里经常扮演非常重要的角色，比如依赖预构建、开发期快速转换等。

三者最核心的区别，通常可以从两个维度看。第一是生态成熟度，Babel 最强；第二是速度，SWC 和 esbuild 通常明显更快。实际项目里往往不是绝对二选一，而是看需求组合使用。

比如开发阶段可能借助 esbuild 提升速度，生产构建或复杂语法转换仍然保留 Babel；或者直接用 SWC 替代大部分 Babel 转译工作。很多时候选择不是“谁最强”，而是“谁最适合当前项目的速度需求和插件依赖”。

还有一个面试里容易混淆的点是：这些工具能做 TypeScript 语法转译，不代表它们一定做完整的类型检查。类型检查通常还是要交给 TypeScript 编译器或独立流程。

如果总结一句话，可以说：Babel 强在生态和定制，SWC 和 esbuild 强在速度；项目怎么选，核心看是否更重视编译性能还是插件兼容能力。

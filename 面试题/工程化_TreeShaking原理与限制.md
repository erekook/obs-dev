# 题目
请你讲一下 Tree Shaking 是什么。它为什么能减少打包体积？又有哪些前提和限制？

# 回答要点
- Tree Shaking 是打包阶段移除未使用代码的优化手段。
- 它依赖静态分析，所以和模块系统有关。
- 常见前提：
  - 使用 ES Module
  - 构建工具支持静态分析
  - 代码尽量保持无副作用可判断
- 为什么能减少体积：
  - 未被引用的导出不进入最终产物
  - 死代码可在压缩阶段进一步移除
- 常见限制：
  - CommonJS 静态分析能力弱
  - 模块有副作用时不容易安全删除
  - 动态引用、聚合导出、整包导入可能影响效果
- 常见关键词：
  - sideEffects
  - dead code elimination
  - `usedExports`
- 面试里可补充：
  - Tree Shaking 不是“万能瘦身”
  - 要结合编码方式和依赖库设计

# 追问
- 为什么 ESM 更利于 Tree Shaking，而 CommonJS 不行或效果较差？
- `sideEffects: false` 有什么作用？乱配会出什么问题？
- 为什么 `import _ from 'lodash'` 往往不如按需引入效果好？
- Tree Shaking 和压缩工具删除死代码是什么关系？

# 参考答案
Tree Shaking 可以理解为“摇掉没用的代码”。它是现代构建工具在打包阶段做的一种优化，目标是把项目里实际没有用到的导出内容尽量从最终产物里移除，从而减少包体积。

它之所以能做到这一点，关键在于静态分析。也就是说，构建工具需要在不真正执行代码的前提下，就能分析出模块之间的依赖关系，以及哪些导出被使用了、哪些没有被使用。

这也是为什么 ES Module 更适合 Tree Shaking。因为 `import` 和 `export` 的结构是静态的，构建工具在编译阶段就能分析依赖；而 CommonJS 的 `require` 是运行时执行的，依赖关系更动态，静态分析难度更大，所以效果通常较差。

Tree Shaking 之所以能减小体积，是因为那些没有被引用的函数、变量、组件等导出内容，可以不进入最终打包结果，或者在后续压缩阶段被进一步删除。

不过它并不是万能的。比如模块存在副作用，构建工具就不能轻易删掉整段导入代码。这里的副作用可能是修改全局变量、注册事件、引入样式文件等。再比如，如果使用动态导入方式不明确、整包导入工具库、或者模块写法不利于静态分析，Tree Shaking 效果都会打折扣。

很多项目里还会配置 `sideEffects` 字段，告诉构建工具哪些文件可以安全地认为没有副作用。这个配置如果写得正确，可以提升 Tree Shaking 效果；但如果乱写，可能把本来必须执行的代码错误删除，导致运行问题。

所以面试里可以总结成一句话：Tree Shaking 的本质是基于静态分析删除未使用代码，它依赖 ESM 和可判断的无副作用代码结构，能优化包体积，但效果受模块规范、导入方式和代码副作用影响很大。

# 题目
请你讲一下 Webpack 和 Vite 的主要区别。为什么 Vite 在开发环境通常会更快？它在生产环境又是如何完成打包的？

# 回答要点
- 两者都是前端工程化工具，但设计思路不同。
- Webpack 的核心是基于依赖图进行整体打包，开发时也常常需要先构建整个项目。
- Vite 开发环境基于原生 ES Module，按需加载模块，不需要一上来完整打包。
- Vite 启动快、热更新快的原因：
  - 依赖预构建
  - 源码按需编译
  - 浏览器直接请求模块
- Webpack 热更新在大型项目中可能更重，因为要维护更复杂的打包过程。
- Vite 生产环境并不是“不打包”，而是通常使用 Rollup 做生产构建。
- 兼容性方面：
  - Webpack 生态历史更久，插件体系更成熟
  - Vite 对现代前端栈更友好，但某些老项目迁移成本更高
- 选择要结合项目规模、历史包袱、生态适配情况来判断。

# 追问
- 什么是依赖预构建，Vite 为什么要用 esbuild 做这一步？
- 为什么说原生 ESM 改变了开发服务器的工作方式？
- Vite 和 Webpack 的插件体系有什么差别？
- 老旧浏览器兼容在 Vite 里通常怎么处理？

# 参考答案
Webpack 和 Vite 都是前端工程化工具，但它们的开发体验差异很大，核心原因在于底层机制不同。

Webpack 的传统思路是先分析整个项目依赖关系，构建一张完整的依赖图，再把模块经过 loader 和 plugin 处理后打包输出。所以在开发环境下，哪怕只是启动本地服务，也往往需要先做一轮相对完整的构建。项目越大，启动和热更新的成本越明显。

Vite 的思路不一样。它在开发环境下利用浏览器原生支持 ES Module 的能力，不先把整个项目打成一个大包，而是让浏览器按需请求模块。开发服务器只在模块被请求时再进行转换和返回，因此启动速度通常非常快。

Vite 还有一个关键优化是“依赖预构建”。像 `react`、`vue`、`lodash` 这类第三方依赖通常不会频繁变化，Vite 会先用 esbuild 把这些依赖预处理成更适合浏览器直接加载的形式，这样后续开发时加载会更快。

热更新方面，Vite 通常只需要处理变更的那个模块以及与之相关的边界，而不是重新走一遍重量级打包流程，所以在大型项目里会显得更轻、更快。

不过 Vite 并不是完全不打包。它只是在开发阶段尽量避免整体打包；到了生产环境，通常还是使用 Rollup 来做最终构建、代码分割、压缩和输出。

如果总结成一句话：Webpack 更像“先整体打好再运行”，Vite 更像“开发时按需提供模块，生产时再正式打包”。所以 Vite 在现代项目开发体验上通常更快，但 Webpack 在历史生态和复杂定制方面依然很强。

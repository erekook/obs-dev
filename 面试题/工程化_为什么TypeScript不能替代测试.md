# 题目
为什么很多团队既用 TypeScript，又依然要写测试？TypeScript 不是已经能帮我们发现很多问题了吗？

# 回答要点
- TypeScript 和测试解决的问题不同。
- TypeScript 主要擅长：
  - 静态类型检查
  - 提前发现参数、返回值、属性访问等类型问题
- 测试主要擅长：
  - 验证运行时真实行为
  - 校验业务逻辑、边界条件、交互流程
- TypeScript 不能覆盖的问题：
  - 运行时数据是否符合预期
  - 异步流程是否正确
  - UI 交互是否符合要求
  - 第三方接口返回异常时的处理
- 面试里可补充：
  - 类型安全不等于业务正确
  - TS 和测试是互补关系，不是替代关系

# 追问
- 为什么“类型对了”不代表“功能对了”？
- API 返回数据即使有类型定义，为什么运行时仍可能出错？
- 哪些逻辑更适合依赖 TS，哪些更值得写测试？
- Zod/运行时校验库在这类问题里起什么作用？

# 参考答案
很多团队用了 TypeScript 之后，仍然会继续写测试，核心原因是：TypeScript 和测试解决的是两类不同的问题。

TypeScript 最擅长的是静态检查。它能在你写代码或编译时，帮你发现参数类型不匹配、属性访问错误、返回值类型不一致、空值风险等问题。这对提升代码可靠性非常有帮助。

但 TypeScript 的能力主要停留在“代码结构层面”。它能告诉你“这个值按声明应该是个字符串”，却不能保证运行时从接口里拿到的东西真的就是你以为的那个字符串。它也不能替你验证一整段业务逻辑在真实执行时是否符合预期。

比如一个下单流程、一个价格计算函数、一个复杂表单校验逻辑，即使类型全部正确，也仍然可能业务规则写错、边界处理漏掉、异步流程顺序不对。测试的价值就在于验证这些“运行起来到底对不对”的问题。

所以可以简单理解成：TypeScript 更像是在代码还没运行前帮你做语法和结构层面的守门；测试则是在逻辑真正执行时，帮你确认行为是否符合预期。

这也是为什么“类型安全”不等于“业务正确”。一个函数参数类型完全没问题，不代表它处理优惠券、库存、用户权限这些业务规则时就一定正确。

所以面试里如果总结，可以说：TypeScript 负责提升静态阶段的正确性，测试负责验证运行时和业务层的正确性；前者不能替代后者，它们是互补关系，不是二选一。

# 题目
为什么前端项目也需要做单元测试？它主要帮团队解决什么问题？

# 回答要点
- 前端单元测试的核心价值：
  - 保障核心逻辑正确性
  - 降低重构风险
  - 提升协作和回归效率
- 适合测试的内容：
  - 工具函数
  - 数据转换逻辑
  - 状态处理逻辑
  - 组件中的关键交互行为
- 带来的收益：
  - 发现边界问题更早
  - 改代码时更有信心
  - 减少手工回归成本
  - 文档化代码行为
- 面试里可补充：
  - 单元测试不是替代 E2E，而是和集成测试、端到端测试分工不同
  - 不是所有前端代码都值得写单元测试
- 常见误区：
  - 为了覆盖率而测试
  - 过度测试实现细节而不是行为

# 追问
- 什么样的前端代码更值得优先补单元测试？
- 为什么说测试实现细节会让测试脆弱？
- 单元测试、集成测试、E2E 测试怎么分工？
- 如果组件 heavily 依赖 UI 库，测试重点应该放哪？

# 参考答案
很多人一开始觉得前端主要是“页面代码”，靠手点一遍就行，但项目一复杂，前端里其实也有大量值得被验证的逻辑，比如数据处理、状态更新、权限判断、表单校验、边界分支、组件交互等。

单元测试最核心的价值，是让团队在改动代码时更有把握。比如一个价格计算函数、一个复杂表单校验逻辑、一个 reducer、一个权限工具，如果没有测试，每次改动都只能靠人工回归；而有了单元测试，很多回归可以在本地和 CI 里自动完成。

它带来的另一个好处是降低重构风险。很多时候代码不是不敢改，而是不敢碰。因为一碰可能牵一发而动全身。测试相当于给关键逻辑加了一层安全网，让你知道这次改动有没有把原有行为破坏掉。

另外，好的测试其实也像文档。通过测试用例，后来接手代码的人可以更快理解“这个函数在边界情况下应该怎么表现”“这个组件交互规则到底是什么”。

不过前端测试也不是越多越好。不是每一行样式、每个纯展示性组件都必须写单元测试。更值得优先测试的，通常是核心业务逻辑、边界条件多、容易回归出错、重用率高的部分。

面试里还可以补充一点：单元测试不是万能，也不能替代集成测试和 E2E。它更适合验证局部逻辑是否正确，而不是完整模拟整条用户流程。

面试里如果总结，可以说：前端做单元测试，主要是为了给关键逻辑建立可自动验证的安全网，减少回归成本、降低重构风险，并提升团队协作效率。

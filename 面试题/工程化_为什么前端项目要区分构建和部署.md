# 题目
为什么前端工程里经常会把“构建（build）”和“部署（deploy）”分开说？它们不是一回事吗？

# 回答要点
- 构建和部署是相邻但不同的两个阶段。
- 构建：
  - 把源码处理成可上线产物
  - 包括打包、转译、压缩、代码分割、生成静态资源
- 部署：
  - 把构建产物发布到可访问环境
  - 包括上传到服务器/CDN/静态托管、配置域名、缓存策略等
- 为什么要区分：
  - 出问题时更容易定位
  - 构建产物可复用
  - 发布流程更标准化
  - 便于 CI/CD 编排
- 面试里可补充：
  - “代码能 build 成功”不等于“线上一定可访问且正确运行”

# 追问
- 为什么 build 成功后页面仍可能打不开？
- 构建产物复用在多环境发布里有什么价值？
- CI/CD 里为什么常把 build 和 deploy 拆成不同 job？
- 前端部署除了上传文件，还常涉及哪些配置？

# 参考答案
前端里的构建（build）和部署（deploy）虽然通常前后连着发生，但它们解决的不是同一个问题。

构建，关注的是“怎么把源码变成可上线的产物”。比如把 TypeScript 转成浏览器能运行的 JavaScript，把多个模块打包拆包，压缩代码，抽离 CSS，生成带 hash 的静态资源文件。这一步的结果通常是一份 `dist` 或 `build` 目录。

部署，关注的是“怎么把这份产物真正放到用户能访问的环境里”。比如把文件上传到服务器、Nginx、对象存储、CDN、静态托管平台，配置访问路径、缓存策略、回源规则、域名和 HTTPS 等。

所以 build 成功，只能说明源码处理成功了，不代表用户已经能正常访问。反过来，deploy 阶段出问题，也不一定是代码本身错了，可能是资源路径、服务器配置、缓存、CDN、路由回退规则等问题。

把两者分开说，有几个好处。首先是定位问题更清晰：到底是构建产物生成错了，还是发布环境配置错了。其次是构建产物可以复用，比如同一份 build 结果可以分别部署到测试、预发布和生产环境。再者，在 CI/CD 里把 build 和 deploy 拆成不同阶段，也更方便做缓存、审批、灰度和回滚。

面试里如果总结，可以说：构建解决的是“把源码变成可发布产物”，部署解决的是“把产物变成用户可访问服务”；两者相邻但职责不同，区分它们有助于工程化流程的稳定和问题定位。

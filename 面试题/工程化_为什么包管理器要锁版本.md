# 题目
为什么前端项目里包管理器要有 lock 文件（如 package-lock.json、pnpm-lock.yaml、yarn.lock）？它主要解决了什么问题？

# 回答要点
- lock 文件的核心作用：
  - 锁定依赖树的具体版本
  - 保证不同环境安装结果一致
- 如果没有 lock 文件：
  - 同样的 `package.json` 在不同时间安装，可能拿到不同依赖版本
  - 容易出现“我这能跑、你那不行”
- 带来的收益：
  - 团队协作一致性
  - CI / 生产构建稳定性
  - 更容易回溯问题来源
- 面试里可补充：
  - 锁的不只是直接依赖，还包括传递依赖
  - 这对大型前端项目尤其重要

# 追问
- 为什么只看 `package.json` 还不够？
- 什么是直接依赖和传递依赖？
- lock 文件冲突为什么常见，怎么处理更稳？
- 库项目和应用项目对 lock 文件的态度会有什么差异？

# 参考答案
包管理器的 lock 文件，最核心的意义就是：把依赖安装结果固定下来，保证不同机器、不同时间、不同环境装出来的是同一棵依赖树。

只看 `package.json` 往往是不够的，因为 `package.json` 里写的很多版本并不是绝对精确的，有时会带范围符号，比如 `^`、`~`。这意味着同一个项目，今天安装和下周安装，拿到的某个依赖小版本可能就不一样。

更麻烦的是，这种变化不只发生在你手动写进去的直接依赖上，还可能发生在这些依赖再依赖的“传递依赖”上。结果就是：代码明明没改，团队里有人重新安装后就突然出问题，CI 也可能和本地结果不一致。

lock 文件就是为了解决这个问题。它会把当时解析出来的完整依赖树，包括直接依赖和传递依赖的具体版本，都记录下来。以后别人再安装时，包管理器就尽量按这个锁定结果还原，而不是重新自由解析一遍。

这对前端项目非常重要，因为前端工程依赖非常多，链路也很深。没有 lock 文件，项目稳定性会明显下降；有了 lock 文件，构建结果更可预期，排查问题时也更容易判断是不是依赖版本变化导致的。

所以面试里如果总结，可以说：lock 文件的作用不是“多一个配置文件”，而是把依赖安装从“按范围解析”变成“按确定结果复现”，从而保证团队协作和构建环境的一致性。

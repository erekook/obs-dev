# 题目
为什么前端项目里，开发环境和生产环境有时候表现会不一样？常见原因有哪些，遇到这种问题你会怎么排查？

# 回答要点
- 开发环境和生产环境不是完全同一套执行条件。
- 常见差异来源：
  - 构建方式不同
  - 压缩和 Tree Shaking
  - 环境变量不同
  - mock / 代理配置不同
  - React/Vue 开发模式附加检查不同
  - CDN / 缓存 / 资源路径差异
- 可能表现：
  - 本地正常，线上报错
  - 开发模式没问题，生产打包后出问题
  - 接口地址、权限、静态资源加载异常
- 排查思路：
  - 先确认环境变量和构建配置
  - 本地跑 production build 复现
  - 看网络请求、路径、Source Map、运行时报错
  - 对比 dev/prod 依赖和行为差异
- 面试里可补充：
  - 真正稳定的项目，会尽量缩小 dev/prod 差距
  - 但差距永远不可能完全为零

# 追问
- 为什么本地开发 server 正常，打包后静态资源路径可能错？
- React 严格模式为什么会让开发环境和生产环境行为不同？
- Tree Shaking 和压缩会暴露哪些潜在 bug？
- 为什么 CI / 预发布环境对前端项目很重要？

# 参考答案
前端项目里开发环境和生产环境表现不一样，其实很常见，因为它们本来就不是完全相同的运行条件。

开发环境更强调调试体验，通常有热更新、未压缩代码、更友好的报错提示、代理转发、本地 mock 等能力；生产环境则更强调性能和真实部署形态，会经过压缩、代码分割、资源路径处理、缓存策略、CDN 分发等一整套流程。

所以差异可能出在很多地方。比如环境变量值不同，导致接口地址不一样；开发环境有本地代理，但线上没有；资源在开发环境是相对路径可用，打包后挂到 CDN 或子路径下就出错；或者开发环境里某些问题没暴露，生产压缩后才因为时序、依赖或副作用问题炸出来。

框架本身也会放大这种差异。比如 React 开发模式会有额外检查，严格模式下某些逻辑可能执行更多次，这会让你在开发和生产看到不一样的行为。

排查这类问题时，我通常不会只盯着“线上报错那一刻”，而是先问：开发和生产到底有哪些条件不同。最有效的一步往往是先在本地跑一次生产构建并用生产模式预览，看能不能复现。如果本地 production 也能复现，问题通常就在构建或运行模式差异上；如果不能复现，就要继续看环境变量、部署路径、CDN、缓存、接口权限这些线上条件。

所以从工程角度说，CI、预发布环境、生产构建验证都很重要，因为它们就是为了尽量提早暴露这种“开发没事，线上才炸”的问题。

面试里如果总结，可以说：dev 和 prod 表现不同，本质上是因为它们的构建方式、运行条件和外部环境不同；排查重点是先缩小差异范围，再逐层对比构建、配置、资源路径和运行时行为。

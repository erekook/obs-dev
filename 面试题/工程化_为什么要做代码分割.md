# 题目
为什么前端项目要做代码分割（Code Splitting）？它主要解决了什么问题？

# 回答要点
- 代码分割的核心目标：减少首屏必须加载的代码量。
- 不做代码分割的问题：
  - 首包过大
  - 下载慢
  - 解析执行慢
  - 首屏渲染被拖慢
- 代码分割常见方式：
  - 路由级分割
  - 组件级按需加载
  - 动态 `import()`
  - vendor 拆分
- 带来的好处：
  - 提升首屏加载速度
  - 让非当前页面代码延后加载
  - 提高缓存复用率
- 面试里可补充：
  - 代码分割不是越碎越好
  - 切得太碎会增加请求和管理成本
  - 要结合加载路径和使用频率设计

# 追问
- 路由级分割为什么是最常见的方案？
- 代码分割和 Tree Shaking 的区别是什么？
- 为什么拆包太碎反而可能变差？
- 公共依赖包单独拆出来有什么利弊？

# 参考答案
前端项目做代码分割，最核心的目的就是：不要让用户在进入页面的第一时间，就被迫下载整个应用的所有代码。

如果项目越来越大，但所有页面和组件的代码都打进同一个大包里，那么即使用户只是打开首页，也得先下载和解析“设置页、图表页、编辑器页、后台页”这些根本暂时用不到的代码。这样首包会越来越大，首屏加载速度也会越来越慢。

代码分割就是把这些代码按使用场景拆开，让浏览器只在真正需要的时候再加载对应部分。最常见的方式是路由级分割，比如用户访问首页时只加载首页相关代码，等切到详情页时再加载详情页代码。

除了路由级分割，也可以做组件级按需加载，比如某个复杂弹窗、富文本编辑器、图表组件只有特定操作时才展示，那就没必要一开始就把它们打进首屏包里。

代码分割带来的收益不只是“下载更快”。因为首包变小之后，浏览器解析和执行 JavaScript 的压力也会下降，这对首屏渲染时间同样有帮助。另外，拆出来的公共包如果稳定，还能更好地利用缓存复用。

不过代码分割也不是越碎越好。如果拆得太细，可能会产生过多的小请求，增加调度和管理复杂度，反而不划算。所以一个成熟的方案，通常会根据页面访问路径、资源体积和复用情况来设计，而不是机械地“能拆就拆”。

面试里如果总结，可以说：代码分割的本质是把“用户暂时不需要的代码”延后加载，从而减小首包、提升首屏性能，但拆分粒度要结合实际场景权衡。

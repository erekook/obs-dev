# 题目
为什么很多团队会做统一组件库？它对前端开发效率和项目质量有什么帮助？

# 回答要点
- 统一组件库的核心价值：
  - 复用
  - 一致性
  - 降低重复开发成本
- 带来的好处：
  - UI 风格统一
  - 交互行为统一
  - 减少重复造轮子
  - 提升多人协作效率
  - 降低维护成本
- 工程层收益：
  - 统一 API 设计
  - 统一主题和样式规范
  - 更方便做测试、文档和版本管理
- 质量层收益：
  - 常见组件问题可集中修复
  - 可沉淀无障碍、性能、边界处理经验
- 面试里可补充：
  - 组件库不是单纯一堆 UI 组件，还包括设计规范、开发约定和发布机制
  - 不应为了“有组件库”而做组件库

# 追问
- 什么样的团队或项目适合做组件库？
- 组件库和业务组件有什么区别？
- 为什么组件库要特别重视文档和示例？
- 如果组件库抽象过度，会有什么问题？

# 参考答案
很多团队做统一组件库，最核心的原因其实很务实：同样的按钮、表单、弹窗、表格，如果每个项目、每个页面都各写一遍，不仅浪费时间，还很容易越写越乱。

统一组件库的第一层价值是复用。常见组件抽出来以后，后续项目可以直接使用，不用反复从零开始。第二层价值是一致性。因为所有项目都基于同一套基础组件，页面风格、交互方式、表单行为、错误提示、主题能力就更容易保持统一。

这不仅提升开发效率，也提升产品体验。用户在不同页面、不同系统之间切换时，会感受到一致的交互逻辑，而不是每个地方都像不同团队拼起来的。

从工程角度看，组件库还能把很多经验沉淀下来。比如某个输入框组件处理过焦点问题、边界校验、无障碍支持、性能优化，这些能力以后所有项目都能复用，而不是每个业务自己踩一遍坑。

另外，组件库通常不仅仅是一堆 UI 代码，它往往还包含设计规范、组件 API 约定、文档站点、示例、版本管理和发布机制。真正成熟的组件库，本质上是在统一团队的界面语言和开发协作方式。

当然，组件库也不是越大越好。如果抽象过度，把本来很业务化的东西硬塞进公共层，反而会让组件难用、难维护。所以做组件库的前提通常是：团队里真的有大量重复场景，且这些场景足够稳定、可抽象。

面试里如果总结，可以说：统一组件库的价值在于复用、一致性和经验沉淀，它既提升开发效率，也提升项目质量，但前提是抽象边界要合理。

# 题目
为什么前端项目也需要预发布环境（staging / pre-release）？它和开发环境、生产环境分别解决什么问题？

# 回答要点
- 预发布环境的核心价值：
  - 在接近生产的条件下做上线前验证
- 三种环境大致分工：
  - 开发环境：便于调试、开发效率优先
  - 预发布环境：接近真实部署条件，验证集成结果
  - 生产环境：真实用户使用环境
- 预发布环境能发现的问题：
  - 构建产物问题
  - 环境变量问题
  - 资源路径/CDN/代理差异
  - 接口联调和权限问题
  - 真实部署后的兼容性问题
- 面试里可补充：
  - 很多“本地正常、线上报错”的问题，本质上是在缺少中间验证层
  - 预发布是降低发布风险的重要缓冲区

# 追问
- 为什么本地开发环境不能完全替代预发布环境？
- 预发布环境和生产环境为什么要尽量接近？
- 如果团队没有预发布环境，常见风险有哪些？
- CI/CD 在预发布流程中通常扮演什么角色？

# 参考答案
前端项目需要预发布环境，最核心的原因是：开发环境和生产环境差异太大，只靠本地调试很难保证上线一定没问题。

开发环境的重点是效率。它通常有热更新、代理转发、未压缩代码、mock 能力、开发模式检查等特性，这些都很方便开发，但它并不是最终用户真正访问的那种运行状态。

而生产环境则是真实部署条件：资源可能挂在 CDN、路径前缀可能不同、环境变量不同、接口权限更严格、缓存策略更复杂、构建代码经过压缩和切分。很多问题只有在这种接近真实部署的条件下才会暴露。

预发布环境的作用，就是在正式上线前，提供一个“尽量接近生产，但还不会影响真实用户”的验证场所。它是开发环境和生产环境之间的一层缓冲。

在这个环境里，团队可以提前发现很多典型问题：比如资源路径错了、某个环境变量漏配、某个接口在正式权限体系下访问失败、构建后代码才触发运行时错误、CDN 或缓存行为和本地完全不同等等。

如果没有预发布环境，很多问题就只能靠“直接上线试试”，那风险会非常高。特别是多人协作、大项目、复杂部署链路的场景，预发布环境几乎是降低发布事故概率的关键环节。

从流程上看，CI/CD 往往会先把代码部署到预发布环境，跑一轮构建验证、测试和人工验收，确认没有问题后再推进到生产环境。

面试里如果总结，可以说：预发布环境的价值在于用接近生产的真实条件，提前暴露本地看不出来的问题，它是开发调试和正式上线之间的重要风险缓冲层。

# 题目
为什么前端项目也需要在 CI 里做代码检查、测试和构建验证？仅靠本地开发自觉不够吗？

# 回答要点
- CI 的核心价值：
  - 把质量检查自动化、标准化
  - 避免“在我机器上没问题”
- 前端 CI 常做的事：
  - Lint
  - 单元测试
  - 类型检查
  - 构建验证
  - 有时还会跑 E2E / 视觉回归
- 为什么本地自觉不够：
  - 环境不一致
  - 人会遗漏步骤
  - 团队标准难统一
  - PR 质量无法稳定保证
- 带来的收益：
  - 更早发现问题
  - 阻止有风险代码进入主分支
  - 提升协作效率和发布稳定性
- 面试里可补充：
  - CI 不是为了“折磨开发”，而是把人工 review 不擅长的重复检查自动化

# 追问
- 为什么“能本地跑起来”不代表“可以安全合并”？
- 构建验证为什么重要，即使代码 lint 和测试都过了？
- CI 太慢会带来什么问题，怎么优化？
- 前端项目里哪些检查最值得优先接入 CI？

# 参考答案
前端项目需要 CI，本质上不是因为前端特殊，而是因为团队协作下，任何依赖“大家自己记得检查”的流程，最终都会不稳定。

本地开发当然重要，但只靠开发者自觉去跑 lint、测试、类型检查、构建验证，问题在于：不同人会漏步骤，不同机器环境也可能不一致。有的人只跑了页面，有的人没做完整构建，有的人 Node 版本不一样，这些都会让“本地没问题”变得不可靠。

CI 的价值就在于，它提供了一个统一、可重复、自动执行的质量关口。只要代码进入 PR 或准备合并，CI 就会在统一环境里跑一遍关键检查，比如 lint、单元测试、类型检查和构建。这样问题能更早暴露，而不是等合并后才炸在主分支或线上。

对前端来说，构建验证尤其重要。因为有些问题页面上不一定立刻看出来，但一到正式构建阶段才会报错，比如动态导入路径问题、类型声明冲突、环境变量缺失、某些包在生产构建时出问题等。

CI 还能提升协作效率。很多机械性检查，本来如果全靠代码 review 去发现，不仅费人，还不稳定。把这部分交给自动化，review 就可以更专注在业务逻辑、架构设计和边界处理上。

当然，CI 也不能设计得太重，否则大家会被等待时间拖慢。所以一个成熟团队通常会分层做：PR 阶段跑关键检查，重型验证按需触发或并行化优化。

面试里如果总结，可以说：前端 CI 的价值在于把质量检查从“依赖个人习惯”升级为“统一自动门禁”，从而提升合并质量、协作效率和发布稳定性。

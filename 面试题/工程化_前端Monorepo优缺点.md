# 题目
请你讲一下前端项目使用 Monorepo 的优缺点。什么场景下它会比较合适？

# 回答要点
- Monorepo 指多个相关项目或包放在同一个代码仓库中管理。
- 常见适用场景：
  - 多个前端应用共享组件库
  - 设计系统 + 业务项目并行维护
  - 多包协同开发
- 优点：
  - 依赖和版本管理更统一
  - 跨包联调方便
  - 公共代码复用更直接
  - CI/CD 和代码规范更容易统一
  - 原子提交更方便
- 缺点：
  - 仓库体积可能变大
  - 构建和安装复杂度上升
  - CI 设计更复杂
  - 权限边界和团队协作治理要求更高
- 常见工具：
  - pnpm workspace
  - Turborepo
  - Nx
  - Lerna（了解）
- 面试里可补充：
  - Monorepo 不是目标本身，关键看项目耦合度和协作成本

# 参考答案
Monorepo 的意思是把多个相关项目、应用或者包，放在同一个仓库里统一管理。前端里比较典型的场景是：一个组件库、一个管理后台、一个移动端 H5、一个工具包，它们彼此有关联，而且会共享一些基础代码和工程配置。

它的最大优点是统一。比如依赖版本可以统一管理，代码规范、构建配置、测试规范也更容易统一。一个公共组件改动后，业务项目可以立刻联调，不需要先发包再安装，大大降低协作成本。

另外，Monorepo 很适合做原子提交。比如你同时改了组件库和使用这个组件库的业务项目，在一个提交里就能完成，代码关系也更清晰。

不过它也不是没有代价。随着仓库越来越大，安装、构建、测试、CI 都会更复杂。如果没有好的工具链支持，很容易变成“所有人改一点东西，全仓库都跟着慢”。所以 Monorepo 一般会搭配 pnpm workspace、Turborepo、Nx 这类工具，用来做依赖管理、任务缓存和增量构建。

从团队治理角度看，Monorepo 对协作规范要求更高。因为大家在同一个仓库里工作，如果目录边界、权限边界、提交流程设计不好，也可能带来混乱。

所以它适不适合，关键不是“Monorepo 听起来高级”，而是看这些项目之间到底是不是强相关、是否真的需要频繁共享和联调。如果项目彼此几乎独立、发布节奏完全不同，用 Monorepo 反而可能增加管理成本。

面试里可以总结成一句话：Monorepo 适合强关联、多包协作、共享基础设施的场景，优点是统一和高效联动，缺点是仓库治理和工程复杂度会明显上升。

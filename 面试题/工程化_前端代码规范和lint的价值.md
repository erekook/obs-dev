# 题目
请你讲一下前端项目里为什么要做代码规范和 Lint 校验。ESLint、Prettier 分别解决什么问题？

# 回答要点
- 代码规范的核心价值：
  - 提高可读性
  - 统一团队风格
  - 提前发现潜在问题
  - 降低协作和维护成本
- ESLint：
  - 主要做代码质量和部分风格规则校验
  - 能发现潜在 bug、反模式、不规范写法
  - 可结合框架和 TypeScript 规则扩展
- Prettier：
  - 主要做代码格式化
  - 统一缩进、分号、引号、换行等风格
  - 目标是减少格式争议
- 两者关系：
  - ESLint 更偏“代码对不对、是否合理”
  - Prettier 更偏“代码长什么样”
- 常见落地方式：
  - 编辑器保存自动修复
  - Git hook 提交前校验
  - CI 阶段阻断不合规代码
- 面试里可补充：
  - 代码规范不是越严越好，要平衡效率与约束

# 追问
- 为什么有了 Prettier 还需要 ESLint？
- ESLint 能完全替代 TypeScript 的类型检查吗？
- 为什么很多团队会把 Lint 接入 Git hook 和 CI？
- 规则过多、过严会带来什么问题？

# 参考答案
前端项目做代码规范和 Lint 校验，本质上是为了让代码更容易协作、更容易维护，也更早发现问题。

如果一个团队里每个人写代码的风格、命名习惯、结构习惯都不一样，项目规模一大，阅读和维护成本会明显上升。代码规范能解决这类统一性问题，而 Lint 工具则能把这些规范自动化执行，而不是靠人工 review 一条条挑。

ESLint 的重点是代码质量和规则检查。它不仅能检查一些风格问题，更重要的是能发现潜在 bug 和不合理写法，比如变量未定义、依赖遗漏、危险用法、某些反模式等。所以 ESLint 更像一个“代码质量守门员”。

Prettier 的重点则是格式化。它负责统一代码长什么样，比如缩进、引号、分号、换行、对象和数组排版等。它的价值是把格式问题自动处理掉，避免团队把时间浪费在“到底该不该加分号”这类低价值争论上。

所以两者并不是冲突关系，而是分工不同。可以简单理解为：ESLint 关注“代码是否合理”，Prettier 关注“代码是否整齐”。

在实际项目里，常见做法是开发时在编辑器里自动格式化，提交前通过 Git hook 跑一次 Lint，CI 阶段再统一校验，确保不合规代码不会进主分支。

不过规范也不是越多越好。如果规则过严、过碎，会让开发者疲于应付工具，反而降低效率。所以比较成熟的做法是：把真正影响质量和协作的问题纳入规则，把纯个人偏好的争议尽量交给自动格式化工具统一处理。

面试里如果总结一句话，可以说：代码规范解决团队协作一致性，ESLint 保证代码质量，Prettier 保证代码格式，它们一起让项目更稳定、更可维护。

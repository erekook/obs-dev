# 题目
前端项目里的环境变量一般怎么管理？开发、测试、生产环境的配置要注意什么问题？

# 回答要点
- 环境变量的作用：
  - 区分不同运行环境配置
  - 避免把接口地址、开关项硬编码进业务代码
- 常见配置内容：
  - API 基础地址
  - 埋点开关
  - 功能开关
  - 构建标识
- 常见管理方式：
  - `.env` 系列文件
  - 构建工具注入（Vite、Webpack、Next 等）
  - CI/CD 注入环境变量
- 不同环境注意点：
  - 开发环境便于本地调试
  - 测试环境便于联调和灰度验证
  - 生产环境配置要稳定、可追踪
- 安全点：
  - 前端环境变量最终会进入构建产物，不能把真正的密钥放进去
  - 区分“可公开配置”和“服务端私密配置”
- 面试里可补充：
  - 环境变量命名要统一
  - 配置变更要可审计、可回滚

# 追问
- 为什么前端项目里不能把 API Key 直接放环境变量里当秘密？
- `.env.development`、`.env.production` 一般怎么生效？
- CI/CD 动态注入环境变量相比写死文件有什么优势？
- 功能开关和环境变量是一回事吗？

# 参考答案
前端项目里管理环境变量，核心是为了把“会随着环境变化的配置”从业务代码里抽出来。比如开发环境接口地址、测试环境埋点开关、生产环境资源前缀等，都不应该直接硬编码在代码里。

最常见的方式是使用 `.env` 系列文件，比如 `.env.development`、`.env.test`、`.env.production`，再由构建工具在启动或打包时把这些变量注入到前端代码里。像 Vite、Webpack、Next.js 都有各自约定的变量注入方式。

开发环境通常更强调调试方便，比如连本地 mock 或测试接口；测试环境更强调和后端联调、灰度验证；生产环境则更关注稳定性和可追踪性，配置不能随意变来变去。

不过要特别注意一个常见误区，就是“前端环境变量不是秘密”。因为前端变量最终会被打进浏览器可下载的代码里，只要用户能拿到前端资源，就有机会看到这些值。所以像真正的 API Secret、数据库密码、第三方服务私钥，绝对不能放在前端环境变量里。

前端环境变量更适合存“可以公开但需要按环境切换”的配置，比如接口基础地址、构建版本号、是否开启某个实验功能等。真正敏感的信息，应该放在服务端环境中，由后端保管。

在工程实践里，除了本地 `.env` 文件，也常会通过 CI/CD 在构建或部署阶段动态注入配置。这样做的好处是更灵活，也更适合统一管理和审计。

面试里如果总结一句话，可以说：前端环境变量本质上是构建时注入的公开配置，不是安全存储；管理重点是按环境区分、统一命名、避免硬编码，并明确和服务端私密配置的边界。

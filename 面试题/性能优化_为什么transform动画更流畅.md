# 题目
为什么前端里通常说用 `transform` 和 `opacity` 做动画会更流畅？它和直接改 `top/left`、`width/height` 有什么本质区别？

# 回答要点
- 动画是否流畅，核心看是否频繁触发布局和重绘。
- `transform`、`opacity` 的优势：
  - 通常不会触发布局（reflow）
  - 更容易交给合成层处理
  - 浏览器优化空间更大
- `top/left`、`width/height` 的问题：
  - 可能引发布局计算
  - 进一步触发重绘
  - 影响范围可能扩散到周围元素
- 渲染链路角度：
  - 布局（Layout）
  - 绘制（Paint）
  - 合成（Composite）
  - `transform/opacity` 更接近只影响合成阶段
- 面试里可补充：
  - 并不是所有 `transform` 动画都一定“零成本”
  - 过多图层也可能带来内存和管理开销

# 追问
- 为什么改 `top/left` 往往会触发重排？
- 什么是合成层？
- `will-change` 为什么不能乱用？
- 如果元素很多，全部做 transform 动画就一定更好吗？

# 参考答案
前端里常说 `transform` 和 `opacity` 做动画更流畅，核心原因在于它们通常不会触发布局重算，浏览器更容易把这类变化放到渲染流程的后半段处理。

浏览器渲染页面大致会经历样式计算、布局、绘制、合成几个阶段。如果你修改 `width`、`height`、`top`、`left` 这类会影响元素几何位置和尺寸的属性，浏览器往往要重新计算布局，甚至影响到周围元素，接着还要重绘，成本比较高。

而 `transform` 更像是在元素已经排好版之后，对它做视觉层面的平移、缩放、旋转。很多情况下，这类变化不需要重新计算布局，也不必让周围元素重新排布。`opacity` 也是类似，只改变透明度，不改变盒模型和布局关系。

这就使得浏览器更容易把 `transform` 和 `opacity` 的动画放到合成阶段处理。合成阶段通常比重新布局、重绘轻量得多，所以动画更容易保持流畅。

不过也不能把它理解成“用了 transform 就一定完美”。如果页面里同时有大量元素都做动画，或者强行创建太多图层，也会增加内存和合成管理开销。所以它只是“更适合做高频动画”，不是完全没有成本。

在实际开发里，如果你做位移、缩放、淡入淡出动画，通常优先考虑 `transform` 和 `opacity`；如果你直接改 `top/left` 或尺寸属性，就更容易触发重排和重绘，导致卡顿。

面试里可以总结成一句话：`transform` 和 `opacity` 更流畅，不是因为它们神奇，而是因为它们通常避开了昂贵的布局和重绘阶段，更接近浏览器渲染链路里的合成阶段。

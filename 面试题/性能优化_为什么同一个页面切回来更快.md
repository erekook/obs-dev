# 题目
为什么用户从一个页面跳到另一个页面，再返回原页面时，常常会感觉比第一次打开更快？前端层面通常有哪些原因？

# 回答要点
- 返回更快通常和“已有状态被复用”有关。
- 常见原因：
  - 浏览器缓存命中
  - 页面没有被完全销毁
  - 路由状态和组件状态仍保留
  - 数据缓存（如 React Query / SWR / 本地 store）
  - DNS / 连接 / 资源预热已完成
- 单页应用常见情况：
  - 只是路由切换，不一定整页重建
  - 某些组件和数据仍然在内存中
- 浏览器级机制：
  - Back-Forward Cache（bfcache）
  - 内存缓存
- 面试里可补充：
  - “更快”不一定是重新请求本身更快，也可能是根本没重新走完整流程

# 追问
- 什么是 bfcache？它和普通 HTTP 缓存有什么区别？
- 单页应用为什么更容易出现“返回秒开”的体验？
- 哪些行为会破坏页面返回时的快速恢复？
- 如果想提升返回体验，前端通常可以怎么设计缓存策略？

# 参考答案
用户返回一个页面时感觉更快，最常见的原因不是“这个页面突然性能变好了”，而是第一次打开时已经把很多准备工作做完了，第二次回来可以直接复用。

前端层面，第一类原因是资源和数据缓存。静态资源可能已经在浏览器内存或磁盘缓存里，接口数据也可能还保存在应用状态管理或数据请求缓存里，比如 React Query、SWR、本地 store 等。所以返回时，不一定需要重新从头加载。

第二类原因是页面本身没有被完全销毁。尤其在单页应用里，很多所谓“页面切换”本质上只是路由切换，某些组件、状态、滚动位置甚至整个页面树都可能还留在内存里。这样用户返回时，恢复成本就很低。

第三类是浏览器本身的能力。比如浏览器的 Back-Forward Cache（bfcache）可以直接把整个页面快照保留下来，用户点击返回时，页面就像“被冻结后又恢复”一样，速度会非常快。这和普通的 HTTP 缓存还不完全一样，因为它可能连页面状态、滚动位置都一起保留。

另外，网络链路层面也可能有贡献。比如 DNS 缓存、TCP/TLS 连接复用、资源预热等，都让第二次访问少走了一部分前置流程。

所以“切回来更快”本质上往往是：资源复用 + 状态保留 + 浏览器恢复机制共同作用的结果，而不是每次都重新完整地请求、解析、渲染一遍。

面试里如果总结，可以说：页面返回更快，通常是因为浏览器和前端应用复用了已有资源、数据和状态，甚至直接恢复了页面快照，因此很多加载链路被跳过了。

# 题目
为什么前端页面要做懒加载（Lazy Load）？它到底优化了什么，又有哪些常见副作用？

# 回答要点
- 懒加载核心思想：
  - 当前不需要的资源先不加载
  - 等真正需要时再加载
- 优化点：
  - 减少首屏资源数量
  - 缩短首屏加载时间
  - 降低带宽消耗
  - 减少浏览器初始解析和执行压力
- 常见懒加载对象：
  - 图片
  - 路由页面
  - 大型组件
  - 第三方库
- 常见实现方式：
  - `loading="lazy"`
  - `IntersectionObserver`
  - 动态 `import()`
- 常见副作用：
  - 首次进入该区域时会有等待
  - 如果时机控制不好，可能出现白块/闪烁
  - 过度懒加载可能影响体验或 SEO
- 面试里可补充：
  - 懒加载不是越多越好
  - 首屏关键资源不应被错误地延后加载

# 追问
- 为什么首屏资源不适合盲目懒加载？
- 图片懒加载和路由懒加载的本质相同吗？
- 懒加载常和预加载怎么搭配？
- 如果出现滚到可视区才开始请求导致白屏，怎么优化？

# 参考答案
前端做懒加载，核心目的就是：把“暂时用不到的资源”延后，优先保证当前最重要的内容先出来。

如果一个页面里有很多图片、模块、路由页面或者第三方库，而这些内容用户在进入页面时根本还没看到，那么一开始就把它们全都下载、解析、执行，其实是在浪费首屏时间和带宽。

懒加载优化的重点主要有几个方面。第一，它能减少首屏必须加载的资源数量，让页面更快显示出第一批有效内容。第二，它能降低初始网络请求和浏览器解析执行压力，尤其在资源多、组件重的页面里很有价值。第三，它还能节省用户带宽，特别是图片很多或模块很多的场景。

图片懒加载是最常见的例子。首屏之外的图片先不加载，等它快进入视口时再请求。路由懒加载也是类似思路：用户没进入某个页面前，不必先把那个页面对应的 JS 都打进当前首包里。

不过懒加载也不是完全没有代价。最常见的问题是：真正需要显示时，它才开始加载，所以第一次进入那个区域时可能会有短暂等待。如果处理不好，就会出现白块、闪烁、占位不稳等体验问题。

所以懒加载不是越多越好。首屏关键资源，比如主内容、LCP 图片、主业务逻辑脚本，通常不应该被错误地延后加载。更合理的做法是：非关键资源懒加载，关键资源优先加载，必要时再配合预加载或占位策略。

面试里如果总结，可以说：懒加载的本质是把非关键资源延后，从而优化首屏速度和资源利用率；但如果用得不当，也可能把等待从“页面初始”挪到“用户滚动或点击之后”，所以关键在于分清资源优先级。

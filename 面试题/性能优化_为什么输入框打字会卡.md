# 题目
为什么一个输入框在打字时会卡顿？如果你要排查前端输入延迟，会优先看哪些问题？

# 回答要点
- 输入卡顿通常是主线程在每次输入时做了太多事。
- 常见原因：
  - 每次输入都触发整页重渲染
  - 输入回调里有重计算
  - 实时校验过重
  - 频繁请求且没有防抖
  - 大量 DOM 更新或布局计算
- 在框架中的常见场景：
  - 受控组件状态更新范围过大
  - 父组件更新带动大子树重复渲染
- 排查方向：
  - 看输入事件回调是否过重
  - 看组件是否无意义重复渲染
  - 看是否有同步校验/格式化逻辑
  - 看是否每次都触发接口请求
  - 用 Performance 面板看长任务
- 优化思路：
  - 拆分组件
  - 减少更新范围
  - 防抖异步请求
  - 延后非关键逻辑
  - 必要时使用非受控或局部状态
- 面试里可补充：
  - 输入卡顿本质是交互型性能问题，往往和“每次键入都引发全链路工作”有关

# 参考答案
输入框打字会卡，最常见的原因不是输入框本身有问题，而是每敲一个字符时，前端在背后做了太多同步工作。

输入是高频交互，一次打字会立刻触发事件、状态更新、可能的组件渲染、校验、请求甚至布局计算。如果这条链路里任何一环太重，用户就会直接感觉到输入延迟。

在 React/Vue 这类框架里，一个典型问题是：输入框放在一个很大的组件里，每次输入都会触发整棵大组件树重新渲染。这样即使输入框本身很简单，用户每敲一个字，也得等一大堆无关 UI 跟着重算。

另一个常见问题是输入回调本身太重，比如每次输入都做复杂校验、格式化、过滤大列表、更新很多 DOM，或者直接发请求却没有防抖。

排查这类问题时，我通常会先看输入事件回调里到底做了什么，再看组件更新范围是不是过大，接着用浏览器 Performance 面板观察有没有长任务、重排重绘热点。如果是框架项目，也会配合看组件 render 次数是否异常。

优化上，思路通常是“把每次输入必须做的事降到最少”。比如：把大组件拆开，让输入只影响局部；异步请求加防抖；复杂校验改到 `blur` 或提交时；非关键逻辑延后；必要时减少受控范围。

面试里如果总结，可以说：输入框卡顿的本质，是每次键入都牵动了过多同步计算、渲染或请求；优化重点是缩小输入触发的工作范围，让主线程把时间优先留给用户交互本身。

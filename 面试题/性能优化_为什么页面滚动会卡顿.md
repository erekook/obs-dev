# 题目
页面滚动为什么会卡顿？如果你要排查一个滚动不流畅的问题，会从哪些方向下手？

# 回答要点
- 滚动卡顿通常和主线程负担过重有关。
- 常见原因：
  - 滚动事件回调太频繁
  - 回调里做了重计算
  - 频繁读写布局属性导致强制同步布局
  - 列表 DOM 太多
  - 图片/动画过重
- 排查方向：
  - 看 Performance 面板是否有长任务
  - 查滚动监听是否节流
  - 看是否频繁触发重排/重绘
  - 检查列表渲染规模
  - 检查动画是否使用了不合适的属性
- 常见优化手段：
  - 节流或用 `requestAnimationFrame`
  - 使用 `IntersectionObserver`
  - 虚拟列表
  - 减少不必要 DOM 和样式计算
  - 尽量用 `transform/opacity` 做动画
- 面试里可补充：
  - 滚动性能问题往往不是单点，而是“事件 + 布局 + 渲染”叠加

# 追问
- 为什么滚动事件特别容易暴露性能问题？
- 什么叫强制同步布局？
- 为什么 `IntersectionObserver` 常常比手写滚动监听更稳？
- 列表很长时，什么时候该上虚拟列表？

# 参考答案
页面滚动会卡顿，最本质的原因通常是：浏览器想要保持平滑滚动和连续绘制，但主线程在滚动过程中被其他任务占住了。

滚动本身是高频行为，用户一滑动就会持续触发一系列更新。如果你在滚动回调里做了大量计算、频繁操作 DOM、读取布局信息再立刻改样式，就很容易让主线程忙不过来，结果就是掉帧、卡顿、滚动发粘。

常见问题之一是滚动监听本身太重。比如每次 scroll 都跑复杂逻辑、请求接口、遍历大量节点，这会直接把高频事件变成性能负担。

另一个常见问题是布局抖动。比如一边读 `offsetHeight`、`scrollTop` 这类布局属性，一边马上修改样式，浏览器可能被迫频繁同步布局，性能开销会非常高。

如果页面里本身 DOM 数量很多，比如很长的列表、很多图片、复杂卡片结构，那么滚动时浏览器还要处理更多布局和绘制任务，这也会让问题变得更明显。

排查时，我通常会先打开 Performance 面板看有没有长任务、重排、重绘热点，再看滚动回调是否过于频繁、是否缺少节流，接着检查是不是长列表没有做虚拟化，或者动画使用了容易触发布局的属性。

优化上，常见手段包括：给 scroll 回调做节流或改用 `requestAnimationFrame` 调度；能不用 scroll 监听的场景尽量用 `IntersectionObserver`；长列表做虚拟列表；动画尽量用 `transform/opacity`；避免滚动过程中做不必要的同步计算。

面试里如果总结，可以说：滚动卡顿通常是主线程在高频滚动过程中被重计算、重排重绘或海量 DOM 拖慢，排查要从事件监听、布局计算和渲染开销三个层面一起看。

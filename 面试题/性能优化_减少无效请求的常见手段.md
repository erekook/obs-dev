# 题目
前端项目里，怎么减少无效请求和重复请求？你会从哪些层面做优化？

# 回答要点
- 无效请求常见来源：
  - 重复点击
  - 组件重复挂载触发
  - 条件不足时仍发请求
  - 过期请求结果无意义
  - 参数相同却重复拉取
- 常见优化手段：
  - 请求防抖/节流
  - 请求去重
  - 缓存相同请求结果
  - 取消过期请求
  - 条件判断后再发请求
  - 组件卸载时清理副作用
- 体验层：
  - 按钮 loading/禁用
  - 避免用户误触发
- 数据层：
  - 查询缓存（如 React Query/SWR）
  - stale-while-revalidate 思路
- 正确性：
  - 防止旧请求覆盖新请求
  - 处理并发和竞态问题
- 面试里可补充：
  - 前端减少无效请求是优化体验和资源，后端仍需做限流和幂等兜底

# 追问
- 请求去重和缓存有什么区别？
- 为什么组件卸载后还要处理未完成请求？
- 如何避免“后发请求先完成”导致页面数据错乱？
- React Query / SWR 这类库在这个问题上提供了哪些能力？

# 参考答案
减少无效请求和重复请求，核心是三件事：不要乱发、不要重复发、发错了要能及时作废。

前端里很多无效请求其实不是接口本身的问题，而是触发时机没控制好。比如用户连续点击按钮、输入框每敲一个字都直接请求、组件反复挂载、页面条件还没准备好就先发请求，这些都会造成资源浪费和体验抖动。

第一层优化通常是触发层控制。像搜索联想这类场景可以做防抖，滚动加载可以做节流；提交按钮在请求进行中要禁用或显示 loading，避免用户重复点击。

第二层是请求层优化。对于参数完全相同、时间非常接近的请求，可以做请求去重；对于已经没有意义的旧请求，可以取消掉，或者至少在响应回来时忽略它，防止旧结果覆盖新数据。

第三层是数据层优化。对于一些短时间内会重复访问的数据，可以做缓存，比如使用 React Query、SWR 这类库，它们会自动做缓存、复用请求、后台刷新等处理。这样用户来回切页面时，不需要每次都重新拉一遍同样的数据。

另外，组件卸载时也要注意。如果请求还没结束，但页面已经切走了，后面结果回来再去更新状态，就可能导致警告或错误逻辑。所以要么取消请求，要么做状态保护。

面试里比较加分的一点是强调“竞态问题”。比如用户先请求 A，又很快请求 B，如果 A 比 B 晚回来，可能把新的页面状态覆盖掉。这个时候就要通过请求标识、取消机制或者只认最后一次结果的策略来保证正确性。

如果总结一下，减少无效请求要从触发层、请求层和数据层一起做：控制触发频率、去重与取消过期请求、缓存可复用数据，再加上后端的幂等和限流，整体体验才会稳定。

# 题目
前端性能优化里，图片通常是大头。请你讲一下图片优化有哪些常见手段，以及你会怎么按场景选择？

# 回答要点
- 图片优化核心目标：减少体积、减少请求、尽快显示、兼顾清晰度。
- 常见手段：
  - 选择合适格式（JPEG、PNG、WebP、AVIF、SVG）
  - 压缩图片体积
  - 按需裁剪尺寸
  - 响应式图片（`srcset`、`sizes`）
  - 懒加载
  - 关键图片预加载
  - 使用 CDN
  - 雪碧图（了解，现代场景减少）
- 格式选择：
  - 照片类常用 JPEG/WebP/AVIF
  - 透明背景可能用 PNG/WebP
  - 图标和简单矢量图适合 SVG
- 用户体验：
  - 首屏图优先加载
  - 列表图懒加载
  - 低质量占位图/骨架图
- 面试里可补充：
  - 不只是压缩，还要关注加载时机和展示尺寸
  - 现代优化更强调“按场景分发”

# 追问
- 为什么大图缩小显示仍然浪费性能？
- WebP 和 AVIF 各有什么优缺点？
- 为什么有了 HTTP/2，雪碧图的重要性下降了？
- 响应式图片怎么避免高分屏模糊？

# 参考答案
图片优化在前端性能里非常重要，因为很多页面的体积大头并不是 JS，而是图片资源。优化图片，核心不只是“压缩一下”，而是要同时考虑格式、尺寸、加载时机和分发方式。

第一步通常是选对格式。照片类图片适合 JPEG、WebP、AVIF；需要透明背景时可以考虑 PNG 或支持透明的 WebP；图标、Logo、简单图形更适合 SVG，因为它本身就是矢量格式，清晰且通常体积小。

第二步是控制尺寸和体积。很多页面会直接上传一张很大的原图，再在页面里缩小显示，这其实很浪费，因为浏览器仍然要下载完整大图。正确做法是根据实际展示尺寸生成合适规格的图片，再配合压缩，减少无意义的传输开销。

第三步是按设备和场景分发。比如使用 `srcset` 和 `sizes` 做响应式图片，让不同屏幕宽度或 DPR 的设备拿到合适分辨率的资源，既避免模糊，也避免无意义的大图下载。

第四步是控制加载时机。首屏关键图片应该优先加载，必要时可 preload；列表里的图片则更适合懒加载，避免一上来把带宽占满。还可以结合占位图、渐进加载，改善用户感知。

再往工程化一点，可以使用 CDN 做图片分发、压缩、格式转换和动态裁剪，这在真实项目里很常见。

如果总结成一句话，图片优化不是单一技巧，而是“选对格式 + 控制尺寸 + 按需加载 + 场景分发”。面试时能从这几个层次展开，通常会显得比较系统。

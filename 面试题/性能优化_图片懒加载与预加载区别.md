# 题目
请你讲一下图片懒加载和预加载的区别。它们分别适合什么场景？如果让你优化一个图片很多的页面，你会怎么组合使用这两种策略？

# 回答要点
- 懒加载和预加载的目标不同：
  - 懒加载：延后非当前必需资源的加载
  - 预加载：提前加载即将用到或关键资源
- 图片懒加载常见场景：
  - 长列表
  - 瀑布流
  - 电商商品列表
  - 图文内容流
- 图片预加载常见场景：
  - 首页首屏大图
  - 即将切换的轮播图
  - 用户下一步大概率访问的关键图片资源
- 常见实现方式：
  - 懒加载：`loading="lazy"`、`IntersectionObserver`
  - 预加载：`preload`、手动创建 `Image` 对象、资源提示
- 优化思路：
  - 首屏关键图优先加载
  - 屏外图片懒加载
  - 配合压缩格式、尺寸裁剪、CDN
  - 避免无脑预加载导致带宽竞争
- 关键认知：
  - 懒加载是减少首屏负担
  - 预加载是提高关键资源到达速度

# 追问
- `prefetch`、`preload`、懒加载三者有什么区别？
- 使用 `IntersectionObserver` 做懒加载相比滚动监听有什么优势？
- 为什么过度预加载可能反而让首屏更慢？
- 响应式图片和懒加载如何配合？

# 参考答案
图片懒加载和预加载虽然都和资源加载策略有关，但方向正好相反。

懒加载的核心思想是：当前不需要的图片先不加载，等它快进入视口或者真正需要显示时再加载。这样可以减少首屏资源数量，降低初始网络压力，提升页面首次渲染速度。它特别适合长列表、信息流、商品列表、相册这类图片很多的页面。

预加载的核心思想是：某些资源很快就要用到，而且对体验很关键，所以提前把它加载好。这样等真正需要显示时，就能更快呈现。它常见于首页首屏大图、下一张轮播图、切换页面时高概率会用到的关键资源。

从实现方式看，懒加载可以直接用浏览器原生的 `loading="lazy"`，或者使用 `IntersectionObserver` 监听元素是否进入可视区域；预加载则可以通过 `link rel="preload"`，或者在 JavaScript 里提前创建 `Image` 对象触发下载。

如果要优化一个图片很多的页面，我通常会这样组合：首屏关键图、LCP 相关图片优先加载，必要时配合 preload；首屏之外的图片统一做懒加载；同时搭配图片压缩、WebP/AVIF、响应式图片和 CDN，进一步减少传输成本。

要注意的是，预加载不是越多越好。如果把大量屏外图片都提前加载，反而会占用带宽，影响真正重要资源的下载，导致首屏变慢。所以可以把两者理解成：懒加载是“延后不重要的”，预加载是“提前重要的”。面试里把这个核心讲清楚就很好。

# 题目
如果后端接口响应很慢，前端除了“等后端优化”之外，还能做些什么来改善用户体验？

# 回答要点
- 前端不能解决后端慢的根因，但可以优化感知体验和请求策略。
- 常见做法：
  - loading / skeleton
  - 占位内容
  - 分片展示，优先加载关键内容
  - 缓存已请求结果
  - 请求去重
  - 预取可能会用到的数据
  - 超时提示与重试机制
- 体验层：
  - 明确状态反馈
  - 避免页面长时间纯白屏
  - 允许局部先渲染、局部后到
- 架构层：
  - 接口并行化
  - 避免不必要串行请求
  - 懒加载非关键数据
- 面试里可补充：
  - “体验优化”不等于“真实性能提升”
  - 需要区分根因优化和感知优化

# 追问
- 骨架屏为什么能改善体验但不一定提升真实性能？
- 什么情况下适合局部先渲染？
- 如果接口经常慢，前端缓存会有哪些风险？
- 超时后自动重试应该注意什么？

# 参考答案
如果后端接口本身很慢，前端确实没法从根本上把服务端处理时间变短，但前端仍然可以做很多事情，让用户感觉不那么“卡住”。

第一类是状态反馈。最怕的不是慢，而是用户不知道系统有没有在工作。所以 loading、骨架屏、局部占位内容这些都很重要。它们不一定让真实耗时变短，但能让用户更早看到界面有反应，减少焦虑感。

第二类是渲染策略优化。不是所有内容都必须等所有接口都回来再一起显示。很多页面可以先把框架、标题、基础信息、缓存数据先渲染出来，把非关键数据延后。这样用户至少先看到“有内容”的页面，而不是一直白屏。

第三类是请求层优化。比如原本几个接口写成串行，前端可以改成并行；相同参数的请求可以去重；已有数据可以缓存复用；用户大概率接下来会进入的页面，也可以提前预取部分数据。

第四类是失败与慢响应处理。比如给出超时提示、允许手动重试、在必要时做合理的自动重试。这样即使后端慢，用户也不会陷入完全无反馈的状态。

但面试里最好强调一点：这些多数属于“感知性能优化”，不是从根上提升后端接口速度。真正的瓶颈还是需要后端、数据库、缓存策略去解决。前端能做的是把等待过程设计得更合理、更可感知、更少无效请求。

面试里如果总结，可以说：接口慢时，前端主要从“状态反馈、渲染策略、请求优化和失败处理”四个层面改善体验，重点是减少白屏和无反馈感，而不是假装后端已经不慢了。

# 题目
请你讲一下虚拟列表（Virtual List）的核心原理是什么。它为什么能优化长列表性能？

# 回答要点
- 虚拟列表的核心思想：
  - 数据很多，但同一时刻真正可见的只有一小部分
  - 只渲染可视区域附近的元素
- 为什么能优化：
  - 大幅减少 DOM 数量
  - 降低布局、绘制和内存开销
  - 滚动时动态替换可见项
- 基本实现思路：
  - 计算可视区域高度
  - 根据滚动位置计算起始索引和结束索引
  - 只渲染这段区间的数据
  - 用占位容器模拟完整滚动高度
- 常见难点：
  - 动态高度项处理复杂
  - 滚动抖动
  - 快速滚动时的白屏与缓冲区控制
- 面试里可补充：
  - 固定高度虚拟列表更容易实现
  - 动态高度通常需要额外测量和缓存

# 追问
- 为什么虚拟列表通常还要加缓冲区（buffer）？
- 动态高度场景为什么更难？
- 虚拟列表会不会影响 SEO 和可访问性？
- React/Vue 里实现虚拟列表时最容易踩什么坑？

# 参考答案
虚拟列表的核心原理其实很简单，就是一句话：列表数据很多，但屏幕同一时刻能看到的元素其实只有一小段，所以没有必要把所有项都真实渲染到 DOM 里。

如果一个列表有几千条数据，而页面一次只能显示十几条，那么真正影响性能的不是“数据量大”，而是“一次性创建了几千个 DOM 节点”。DOM 太多会带来更高的内存占用，也会增加浏览器布局、绘制和更新成本。

虚拟列表就是通过“只渲染当前可视区域附近的项目”来解决这个问题。它会根据滚动位置，计算当前应该显示哪些数据项，比如从第 100 条到第 120 条，然后页面里只真正渲染这 20 条左右的节点。

为了让滚动条看起来仍然像完整列表一样长，虚拟列表通常还会有一个占位容器，用来撑起总高度。用户滚动时，程序不断重新计算可见区间，并替换当前渲染的数据。

这样做的结果就是：哪怕数据有一万条，页面里真正存在的 DOM 节点也可能始终只有几十个，所以性能会稳定很多。

固定高度的虚拟列表实现相对简单，因为只要知道每项高度和滚动距离，就能直接算出起止索引。动态高度会复杂很多，因为每一项高度不固定，需要额外测量、缓存高度，甚至在滚动中不断修正位置。

为了避免快速滚动时出现空白，很多虚拟列表还会额外渲染一点“缓冲区”，也就是把可视区域上下多渲染几项，保证滚动更平滑。

面试里如果总结，可以说：虚拟列表通过“占位总高度 + 只渲染可视区域少量节点”的方式，把长列表性能瓶颈从海量 DOM 降到很小的固定规模，因此能显著优化滚动和渲染性能。

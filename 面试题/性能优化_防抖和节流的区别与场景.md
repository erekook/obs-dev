# 题目
请你讲一下防抖（debounce）和节流（throttle）的区别。它们分别适合什么场景？

# 回答要点
- 两者都用于控制高频事件触发带来的性能问题。
- 防抖：
  - 高频触发时只保留最后一次
  - 触发结束后一段时间再执行
  - 强调“停止后再执行”
- 节流：
  - 在一定时间间隔内最多执行一次
  - 强调“持续触发时按频率执行”
- 防抖适用场景：
  - 搜索输入联想
  - 输入校验
  - 表单自动保存
  - resize 结束后重新计算布局
- 节流适用场景：
  - scroll
  - mousemove
  - 拖拽
  - 高频点击限制
- 关键区别：
  - 防抖更适合“结果只关心最终一次”
  - 节流更适合“过程需要持续反馈但不能太频繁”
- 进阶点：
  - 防抖和节流都可能有 leading / trailing 配置
  - 需要考虑 `this`、参数透传、取消执行

# 追问
- 如果搜索联想用节流会有什么问题？
- 如果滚动监听用防抖会有什么问题？
- `requestAnimationFrame` 能否替代节流？适合哪些场景？
- 如何实现一个同时支持 leading 和 trailing 的防抖函数？

# 参考答案
防抖和节流的本质，都是为了控制高频事件回调的执行次数，避免性能浪费或者请求过多。

防抖的意思是，只在最后一次触发后，等待一段时间再执行。如果这段时间内又触发了，就重新计时。所以在连续触发过程中，前面的执行机会都会被取消，只保留最后一次。

它适合那种“我只关心最终结果”的场景。比如搜索框输入联想、表单校验、自动保存、窗口 resize 结束后重新计算布局。这些场景里，如果每输入一个字符都立刻执行，往往没有必要。

节流的意思是，在固定时间间隔内最多执行一次。即使事件持续高频触发，函数也会按照设定频率执行。

它适合那种“过程需要持续反馈，但不必每次都执行”的场景。比如页面滚动监听、鼠标移动、拖拽、滚动加载等。如果完全不用节流，事件触发会非常频繁，性能压力很大。

所以两者最大的区别可以概括为：防抖强调“停下来之后执行一次”，节流强调“持续过程中按频率执行”。

举个简单例子，搜索联想更适合防抖，因为用户输入还没结束时频繁请求没有意义；而滚动监听更适合节流，因为用户在滚动过程中页面通常需要持续反馈，如果用防抖，只有滚动停止后才执行，体验会变差。

面试时如果能补充一句：两者都可能有 leading 和 trailing 等配置，实际实现时还要处理 `this`、参数和取消逻辑，就会显得更完整。

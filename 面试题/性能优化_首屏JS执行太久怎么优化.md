# 题目
如果页面首屏加载时，JavaScript 执行时间特别长，你会从哪些方向优化？

# 回答要点
- 首屏慢不只是下载问题，JS 解析和执行也可能是大头。
- 常见优化方向：
  - 减少首屏 bundle 体积
  - 路由级/组件级代码分割
  - 延后非关键逻辑执行
  - 按需加载大库
  - 避免首页做大量同步计算
- 执行层面：
  - 拆分长任务
  - 使用 `requestIdleCallback` / 延迟调度非关键逻辑
  - 把重计算移到 Web Worker（适合场景下）
- 代码层面：
  - 去掉无用依赖
  - 优化大对象初始化
  - 减少首屏不必要的副作用和渲染逻辑
- 面试里可补充：
  - 首屏性能既看网络，也看主线程是否被长任务占满
  - Lighthouse / Performance 面板可用于定位长任务来源

# 追问
- 什么是长任务（long task）？
- Web Worker 适合处理哪些计算，为什么不能直接操作 DOM？
- 为什么 bundle 变小不一定就代表执行时间一定短？
- 首屏哪些逻辑通常应该延后？

# 参考答案
页面首屏加载慢，很多人第一反应只盯着“资源下载太大”，但实际上 JavaScript 下载完之后，还要解析、编译、执行。如果执行时间很长，主线程就会被占住，页面即使资源都到齐了，用户也可能还是觉得卡。

优化这类问题，第一步通常还是先减少首屏必须执行的代码量。也就是说，不只是包体变小，更重要的是“首页真正需要跑的 JS 少一些”。这通常会涉及路由级代码分割、组件级按需加载、把重型第三方库延后到真正需要时再加载。

第二步是延后非关键逻辑。很多页面首屏一上来就跑埋点、复杂初始化、批量数据处理、非关键组件挂载，这些逻辑其实不一定非要和首屏主内容抢主线程。可以把它们拆出去，在页面初步可交互之后再执行。

第三步是处理长任务。如果某段 JS 本身计算量很大，比如复杂数据处理、图表转换、大量 JSON 处理，就需要考虑拆分执行，或者在适合的情况下放到 Web Worker 里做，避免长时间阻塞主线程。

另外，还要检查是不是有无意义的执行成本，比如首屏渲染根本没用到的大对象初始化、重复计算、过早注册一堆监听器、effect 里跑了不必要逻辑等。

所以这类问题的思路通常不是单纯一句“压缩包”，而是从“减少首屏必须执行的代码、延后非关键逻辑、拆分长任务、把重计算移出主线程”几个层次一起优化。

面试里如果总结，可以说：首屏 JS 执行太久，本质上是主线程被大量不必要或过重的任务占满，优化重点是减量、延后、拆分和迁移，而不是只盯着下载体积。

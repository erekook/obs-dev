# 题目
如果一个前端页面首屏加载很慢，你会从哪些方面做性能优化？请尽量从资源加载、渲染过程、网络传输和工程化策略几个维度来回答。

# 回答要点
- 先明确性能指标：FCP、LCP、TTI、CLS 等，不能脱离数据谈优化。
- 资源加载优化：
  - 代码分割、按需加载、路由懒加载
  - 图片压缩、WebP/AVIF、响应式图片
  - 字体子集化、减少大体积第三方库
  - `preload`、`prefetch`、`preconnect`
- 渲染优化：
  - 减少阻塞渲染的 CSS/JS
  - 合理使用 `defer`、`async`
  - SSR、SSG、流式渲染
  - 骨架屏、占位策略提升感知速度
- 网络优化：
  - CDN
  - HTTP 缓存
  - Gzip/Brotli
  - HTTP/2、HTTP/3
  - 减少请求数和请求体积
- 工程化与构建：
  - Tree Shaking
  - 压缩混淆
  - 产物分析，排查大包来源
  - 拆分 vendor，避免无意义重复依赖
- 运行时优化：
  - 避免首屏执行过多同步 JS
  - 延迟非关键逻辑
  - 减少回流重绘
- 回答时建议强调：
  - 先定位瓶颈，再做针对性优化
  - 区分“真实性能提升”和“感知性能提升”

# 追问
- 如果 LCP 很差，你会优先怀疑哪些资源？
- SSR 一定比 CSR 更快吗？为什么？
- `preload` 和 `prefetch` 的使用场景分别是什么？
- 如何定位到底是网络慢、JS 执行慢，还是渲染慢？

# 参考答案
如果一个页面首屏很慢，我不会一上来就堆优化手段，而是先看指标和性能分析结果，比如 Lighthouse、Performance 面板、Web Vitals、真实用户监控数据。因为首屏慢可能是网络、资源体积、JS 执行、接口阻塞、渲染链路等不同原因造成的，先定位瓶颈更重要。

从资源加载角度，最常见的做法是减少首屏必须下载的内容。比如做代码分割、路由懒加载，把非首屏模块延后；优化图片体积，使用 WebP 或 AVIF；避免首页就加载过大的图标库、编辑器、图表库；关键资源可以用 `preload` 提前加载。

从渲染过程看，要尽量减少阻塞。CSS 和同步 JS 都可能影响首屏渲染，所以可以把非关键脚本改成 `defer` 或 `async`，抽离关键 CSS，或者采用 SSR、SSG 来让用户更早看到可渲染内容。如果暂时做不到真实提速，也可以用骨架屏、占位图提升感知速度。

从网络层面，可以使用 CDN 提升静态资源分发效率，开启 Gzip 或 Brotli 压缩，合理利用浏览器缓存，减少重复请求，并借助 HTTP/2 的多路复用提升资源传输效率。

从工程化角度，需要分析打包产物，看看是不是引入了过大的依赖，能不能做 Tree Shaking、按需引入、压缩和拆包。有些项目首屏慢，本质上是首页 bundle 太大，导致下载和执行都很重。

最后，运行时也要注意，哪怕资源加载完了，如果首页执行了大量同步 JS、复杂计算或频繁 DOM 操作，用户还是会觉得卡。所以还要延迟非关键逻辑、拆分长任务、减少回流重绘。

总结来说，首屏优化不是单点技巧，而是“测量指标 → 定位瓶颈 → 按链路优化”的过程。回答时如果能按资源、网络、渲染、工程化几个维度展开，面试官通常会觉得你比较系统。

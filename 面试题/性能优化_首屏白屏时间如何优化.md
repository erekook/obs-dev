# 题目
如果一个前端页面在打开时白屏时间很长，你会从哪些方向排查和优化？

# 回答要点
- 白屏时间长通常与首屏 HTML 到可见内容之间的链路过长有关。
- 先定位问题：
  - 看网络请求瀑布图
  - 看 FCP、LCP 等指标
  - 看是资源慢、接口慢、JS 执行慢还是渲染阻塞
- 常见优化方向：
  - 减少首屏必须加载的资源
  - 路由级代码分割
  - 压缩 JS/CSS 体积
  - 使用 `defer` / `async`
  - 关键资源预加载
  - SSR / SSG / 骨架屏
- 网络层：
  - CDN
  - Gzip/Brotli
  - HTTP 缓存
  - 减少首屏接口串行依赖
- 渲染层：
  - 减少阻塞渲染的 CSS/JS
  - 优化首屏组件复杂度
  - 避免首屏执行大量同步逻辑
- 面试里可补充：
  - 白屏优化既包括真实性能优化，也包括感知性能优化
  - 骨架屏不是治本，但能改善用户感受

# 追问
- FCP、LCP、TTI 分别反映什么问题？
- 为什么首屏接口串行请求容易导致白屏时间变长？
- SSR 一定能解决白屏吗？
- 骨架屏和预渲染分别适合什么场景？

# 参考答案
页面白屏时间长，说明用户从打开页面到看到第一块有效内容之间花了太久。排查时我不会直接拍脑袋加优化，而是先看链路到底卡在哪。

第一步通常是看浏览器 Performance 和 Network 面板，判断问题主要在网络、资源体积、JS 执行，还是渲染阻塞。比如有些页面白屏，是因为首屏 bundle 太大；有些则是因为首屏数据接口串行请求太多；还有些是因为拿到资源后又执行了大量同步逻辑，导致页面迟迟不能渲染。

常见优化方向里，最优先的是减少首屏必须加载的东西。比如做路由级代码分割，把非首屏模块延后；压缩 JS 和 CSS 体积；把脚本改成 `defer` 或 `async`；对关键资源做预加载。

如果是网络问题，就要考虑 CDN、压缩、缓存、减少不必要请求，以及避免首屏接口层层串行依赖。很多项目白屏长，不是单个接口慢，而是先请求 A，再根据 A 请求 B，再根据 B 请求 C，这种链路会非常拖。

如果是渲染问题，就要减少阻塞渲染的逻辑。比如首页组件太重、同步计算太多、首屏直接渲染大量列表，这些都会让页面即使资源下载完了也迟迟出不来。

进一步的方案还有 SSR、SSG、预渲染和骨架屏。SSR/SSG 可以让用户更早看到 HTML 内容，骨架屏则能优化感知体验，避免纯白屏带来的焦虑感。不过骨架屏更多是改善感受，不是从根本上解决性能问题。

面试里如果总结一句话，可以说：白屏优化要先定位瓶颈，再从资源加载、网络传输、渲染阻塞和感知体验几个层次一起处理，目标是让用户尽快看到第一块有效内容。

# 题目
请手写一个简单的发布订阅（Pub/Sub）模式，并说明它和观察者模式有什么区别。

# 回答要点
- 发布订阅模式核心角色：
  - 发布者
  - 订阅者
  - 事件中心 / 调度中心
- 核心能力：
  - `on` 订阅
  - `emit` 发布
  - `off` 取消订阅
  - 可选 `once` 一次性订阅
- 手写实现关键：
  - 用对象或 Map 存储事件名和回调列表
  - 发布时遍历执行对应回调
  - 取消订阅时移除回调
- 区别点：
  - 发布订阅模式通过事件中心解耦
  - 观察者模式通常观察者直接订阅目标对象，耦合更直接
- 应用场景：
  - 事件总线
  - 组件通信
  - 自定义通知系统
- 面试里可补充：
  - 发布订阅模式容易滥用，要注意事件管理和内存泄漏

# 追问
- 为什么说事件中心让发布者和订阅者更解耦？
- `off` 方法怎么设计才能更稳妥？
- 如果同一个事件有很多订阅者，发布时某个回调报错怎么处理？
- 发布订阅模式在现代前端框架里为什么不再被鼓励到处乱用？

# 参考答案
发布订阅模式的核心思想是：发布者不直接调用订阅者，而是通过一个中间的事件中心来分发消息。这样发布者和订阅者之间就不需要彼此直接引用，耦合度更低。

一个简单实现可以这样写：

```js
class EventBus {
  constructor() {
    this.events = {};
  }

  on(eventName, fn) {
    if (!this.events[eventName]) {
      this.events[eventName] = [];
    }
    this.events[eventName].push(fn);
  }

  emit(eventName, ...args) {
    const handlers = this.events[eventName];
    if (!handlers) return;
    handlers.forEach(fn => fn(...args));
  }

  off(eventName, fn) {
    const handlers = this.events[eventName];
    if (!handlers) return;
    this.events[eventName] = handlers.filter(item => item !== fn);
  }

  once(eventName, fn) {
    const wrapper = (...args) => {
      fn(...args);
      this.off(eventName, wrapper);
    };
    this.on(eventName, wrapper);
  }
}
```

这个实现里，`on` 用来订阅事件，`emit` 用来发布事件，`off` 用来取消订阅，`once` 表示只执行一次。

它和观察者模式的区别，面试里经常会问。简单说，观察者模式通常是观察者直接依附在目标对象上，目标对象状态变化时主动通知观察者，双方关系更直接；而发布订阅模式多了一个事件中心，发布者和订阅者彼此不知道对方是谁，只通过事件名建立联系，所以解耦更强。

实际开发里，发布订阅模式适合做事件总线、全局通知、模块间解耦通信。但它也容易被滥用，因为事件来源和流向不够直观，项目大了以后会变得难追踪，还可能因为没有及时 `off` 导致内存泄漏。

如果面试里总结，可以说：发布订阅的关键不是“写个 on/emit”，而是理解它通过事件中心实现了解耦，同时也带来了管理复杂度。

# 题目
如果让你手写一个深拷贝函数，除了递归对象和数组之外，还需要考虑哪些边界情况？

# 回答要点
- 深拷贝不只是“递归复制对象”。
- 常见边界情况：
  - 循环引用
  - `Date`
  - `RegExp`
  - `Map`
  - `Set`
  - `Symbol`
  - `undefined`
  - 函数
  - `null`
  - 原型链
- 还要考虑：
  - 保持引用关系
  - 避免重复拷贝同一个对象
  - 区分普通对象和特殊对象类型
- 常见实现思路：
  - 使用 `WeakMap` 记录已拷贝对象，处理循环引用
  - 根据类型分支处理不同对象
- 面试里可补充：
  - 不是所有值都需要或适合被“深拷贝”
  - `structuredClone` 和 `lodash.cloneDeep` 是常见对照方案

# 追问
- 为什么要用 `WeakMap` 而不是普通对象记录已拷贝引用？
- 函数通常怎么处理，为什么？
- `Map` 和 `Set` 深拷贝时要怎么遍历？
- 如果对象有自定义原型，是否需要保留？

# 参考答案
手写深拷贝时，很多人第一反应是“对象递归一下，数组递归一下”，但真正的难点其实在各种边界情况上。

首先最经典的是循环引用。如果一个对象内部又引用回自己，单纯递归会无限调用直到栈溢出。所以通常要用 `WeakMap` 记录“这个对象我已经拷贝过了”，下次再遇到时直接返回之前的副本。

第二类是特殊对象类型。比如 `Date` 不能简单当普通对象递归，否则你会丢掉它的日期语义；`RegExp`、`Map`、`Set` 也是类似，处理方式都和普通对象不一样。

第三类是值类型和不可序列化值。像 `undefined`、`Symbol`、函数这些值，在不同深拷贝方案里行为都可能不同。比如函数通常不会真正“复制逻辑”，更多只是直接返回原函数引用。

第四类是原型链。如果你只是简单用 `{}` 或 `[]` 去装结果，那么原对象如果有自定义原型，拷贝后这个原型关系就丢了。所以更完整的实现可能会考虑通过 `Object.create(Object.getPrototypeOf(obj))` 保留原型。

另外还有一个容易忽略的问题是“相同引用关系”。如果原对象里有两个属性都指向同一个子对象，深拷贝后最好也保持这种共享关系，而不是拷贝成两个彼此独立的新对象。这个问题本质上也依赖已拷贝引用记录来解决。

所以面试里如果回答，不要只停留在“递归 + for...in”。更完整的思路应该是：先判断类型，再按类型分支处理，同时用 `WeakMap` 解决循环引用和共享引用问题，必要时还要保留原型链。

一句话总结就是：深拷贝真正难的不是“拷贝”，而是“正确处理各种对象类型、引用关系和边界语义”。

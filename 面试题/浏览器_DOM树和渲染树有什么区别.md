# 题目
请你讲一下 DOM 树和渲染树（Render Tree）有什么区别。为什么浏览器不直接拿 DOM 树去绘制页面？

# 回答要点
- DOM 树是 HTML 解析后的文档结构树。
- 渲染树是浏览器结合 DOM 和 CSSOM 后生成的用于渲染的树。
- DOM 树关注：
  - 文档结构
  - 标签层级
  - 节点关系
- 渲染树关注：
  - 最终需要显示的可视节点
  - 节点样式信息
  - 布局与绘制所需数据
- 为什么不直接用 DOM 树绘制：
  - DOM 没有完整样式信息
  - 不可见节点不一定需要渲染
  - 渲染还要结合 CSS 规则和布局信息
- 常见认知：
  - `display: none` 的节点通常不会进入渲染树
  - `visibility: hidden` 的节点通常还会在渲染树中，只是不可见
- 面试里可补充：
  - 后续布局和绘制阶段主要基于渲染树进行

# 追问
- CSSOM 是什么，和渲染树的关系是什么？
- 为什么 `display: none` 和 `visibility: hidden` 在渲染树表现不同？
- DOM 改变后一定会重新构建整棵渲染树吗？
- 渲染树和图层树是一回事吗？

# 参考答案
DOM 树和渲染树虽然都和页面渲染有关，但它们解决的问题不一样。

DOM 树是浏览器把 HTML 解析之后得到的文档结构树，它描述的是页面里有哪些元素、这些元素之间是什么父子兄弟关系。它更偏“结构层”。

但浏览器要真正把页面画出来，仅有结构还不够。因为一个元素到底长什么样、是否显示、宽高是多少、位置在哪，都还要结合 CSS 规则来确定。所以浏览器还会解析 CSS，生成 CSSOM，再把 DOM 和 CSSOM 结合起来，得到用于实际渲染的渲染树。

渲染树更偏“显示层”。它只保留那些真正需要参与渲染的节点，并且每个节点都带有最终样式信息。比如 `display: none` 的元素虽然在 DOM 树里还存在，但通常不会进入渲染树，因为它根本不参与显示；而 `visibility: hidden` 的元素仍然占据布局位置，所以通常还在渲染树里，只是最终不可见。

这也解释了为什么浏览器不能直接拿 DOM 树去绘制页面。因为 DOM 树只有结构，没有完整样式和显示信息；而绘制页面需要知道哪些节点可见、样式如何计算、尺寸和位置怎么排布。

后续浏览器在做布局、绘制时，主要就是基于渲染树来处理，而不是直接盯着 DOM 树本身。

面试里如果总结，可以说：DOM 树描述“页面结构”，渲染树描述“页面怎么显示”；浏览器之所以要构建渲染树，是因为真正绘制页面必须把 HTML 结构和 CSS 样式结合起来。

# 题目
请你讲一下 `<script>` 标签里的 `defer` 和 `async` 有什么区别。它们分别适合什么场景？

# 回答要点
- 两者都用于优化脚本加载，避免普通 `script` 阻塞 HTML 解析。
- 共同点：
  - 脚本下载过程一般不会阻塞 HTML 解析
- `defer`：
  - 下载与 HTML 解析并行
  - 等 HTML 解析完成后再按顺序执行
  - 会在 `DOMContentLoaded` 之前执行完
  - 适合依赖 DOM 结构或多个脚本有顺序要求的场景
- `async`：
  - 下载与 HTML 解析并行
  - 下载完成后立即执行
  - 执行时可能打断 HTML 解析
  - 多个 async 脚本执行顺序不保证
  - 适合独立脚本，如统计、广告、埋点
- 面试里可补充：
  - 模块脚本 `type="module"` 默认行为更接近 defer

# 追问
- 为什么普通 `script` 会阻塞 HTML 解析？
- `defer` 为什么适合主业务脚本？
- 如果多个 `async` 脚本有依赖关系，会出什么问题？
- `defer` 和把脚本放到 `body` 底部相比，有什么异同？

# 参考答案
`defer` 和 `async` 都是为了解决普通 `script` 阻塞 HTML 解析的问题，但它们在“什么时候执行脚本”这件事上不一样。

普通的 `script` 标签在浏览器解析 HTML 时，一旦遇到它，通常会先停下 HTML 解析，去下载并执行脚本，执行完再继续往下解析。因为脚本可能会修改 DOM 结构，所以浏览器默认要这样做。这也是为什么普通脚本容易拖慢页面首屏。

`defer` 的特点是：脚本下载和 HTML 解析并行进行，但不会立刻执行，而是等整个 HTML 解析完成后，再按照它们在页面里出现的顺序依次执行。并且这些脚本会在 `DOMContentLoaded` 事件之前执行完。

所以 `defer` 很适合主业务脚本，尤其是那些依赖 DOM 已经解析完成、或者多个脚本之间有先后依赖关系的场景。

`async` 则更激进一些。它也是边解析 HTML 边下载脚本，但一旦脚本下载完成，就会立刻执行，不会等 HTML 全部解析完。这意味着它执行时可能打断当前 HTML 解析流程。另外，如果页面里有多个 `async` 脚本，它们谁先下完谁先执行，顺序无法保证。

因此 `async` 更适合那种完全独立、彼此没有依赖，也不强依赖 DOM 完整结构的脚本，比如统计分析、广告脚本、埋点脚本等。

如果面试里总结，可以说：`defer` 是“并行下载，解析完再按顺序执行”，适合主业务脚本；`async` 是“并行下载，下载完就立即执行”，适合独立的第三方脚本。

# 题目
请你讲一下浏览器里的 `input` 事件和 `change` 事件有什么区别。它们分别适合什么场景？

# 回答要点
- 两者都和表单值变化有关，但触发时机不同。
- `input`：
  - 值发生变化时通常立即触发
  - 更适合实时响应输入
- `change`：
  - 通常在值变更并“提交”后触发
  - 文本输入框里常见为失焦后触发
  - 对 select、checkbox、radio 等控件表现更直接
- 常见场景：
  - `input`：实时搜索、联动校验、字数统计
  - `change`：提交前确认、下拉选择变化处理、表单完成后处理
- 面试里可补充：
  - 不同表单元素的 `change` 触发时机会略有差异
  - React 的 `onChange` 对文本输入行为更接近原生 `input`

# 追问
- 为什么 React 里的 `onChange` 看起来不像原生 `change`？
- `input` 事件在输入法组合输入时要注意什么？
- checkbox 的 `change` 和文本框的 `change` 为什么体验不同？
- 实时校验为什么通常更适合绑定 `input`？

# 参考答案
`input` 和 `change` 都和表单值变化有关，但最核心的区别在于触发时机不同。

`input` 更偏“实时变化”。只要输入框内容发生变化，通常就会立即触发。比如用户每输入一个字符、删除一个字符，监听 `input` 的回调就会执行一次。所以它很适合做实时搜索联想、字数统计、输入联动、即时提示这类场景。

`change` 更偏“变化完成后通知”。对于文本输入框来说，通常是用户修改了值，并且在失焦后才触发；对于 `select`、`checkbox`、`radio` 这类控件，它的触发会更直接，比如用户选中新的下拉项、勾选状态变化时就会触发。

所以可以简单理解为：`input` 更关注“过程中的每次变化”，`change` 更关注“本次编辑结果被确认”。

这也是为什么实时反馈类功能更常用 `input`，而一些只关心最终结果的表单处理，更适合用 `change`。

面试里还有一个常见延伸点，就是 React。React 中的 `onChange` 在文本输入场景下，行为其实更接近原生 `input`，也就是说输入时基本会实时触发，这和浏览器原生 `change` 的直觉不完全一样，所以很多人第一次会觉得命名有点迷惑。

如果总结成一句话，可以说：`input` 适合“值一变就响应”，`change` 适合“值改完后再处理”；前者更实时，后者更偏确认后的通知。

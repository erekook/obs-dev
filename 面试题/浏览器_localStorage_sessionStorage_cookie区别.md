# 题目
请你讲一下 `localStorage`、`sessionStorage` 和 `cookie` 的区别。它们分别适合存什么，不适合存什么？

# 回答要点
- 三者都能在前端存储数据，但机制和适用场景不同。
- `localStorage`：
  - 持久化存储
  - 除非主动清除，否则数据一直存在
  - 同源下共享
  - 容量通常比 cookie 大
- `sessionStorage`：
  - 会话级存储
  - 标签页关闭后通常失效
  - 不同标签页之间一般不共享
- `cookie`：
  - 可由服务端设置，也可前端设置
  - 会随同域请求自动发送给服务器
  - 容量较小
  - 可设置过期时间、路径、域、HttpOnly、Secure、SameSite
- 安全和使用建议：
  - 不要随便在前端明文存敏感信息
  - `localStorage` / `sessionStorage` 容易受 XSS 影响
  - cookie 自动携带，需注意 CSRF 风险
- 面试里可补充：
  - cookie 更偏客户端与服务端之间状态传递
  - Storage 更偏前端本地数据保存

# 追问
- 为什么 cookie 会影响请求体积？
- `HttpOnly` 对前端和安全意味着什么？
- 为什么把 token 放在 localStorage 会有 XSS 风险？
- `sessionStorage` 为什么常用于一次性页面状态保存？

# 参考答案
`localStorage`、`sessionStorage` 和 `cookie` 都可以存数据，但它们的定位和使用场景并不一样。

先说 `localStorage`。它是浏览器提供的持久化本地存储，同源页面下通常都可以访问，只要用户不主动清理，数据会一直保留。所以它适合存一些不敏感、需要长期保留的前端状态，比如主题设置、语言偏好、最近访问记录等。

`sessionStorage` 和它很像，但生命周期更短。它通常跟当前标签页会话绑定，页面关闭后数据就失效，而且不同标签页之间一般不共享。它更适合存一些只在当前会话或当前页面流程中需要的数据，比如临时表单状态、某一步操作上下文等。

`cookie` 则更偏向浏览器和服务器之间的状态传递。它最大的特点是会随着同域请求自动带给服务端，这也是为什么它常被用于登录态、会话标识等场景。它支持设置过期时间、作用域、`HttpOnly`、`Secure`、`SameSite` 等属性，安全控制能力更丰富。

但 cookie 也有明显限制，比如容量更小，而且每次请求都自动携带，会增加请求开销。所以它不适合存大量数据。

安全方面，`localStorage` 和 `sessionStorage` 都能被前端脚本直接读取，所以如果页面发生 XSS，里面的数据就可能被窃取。cookie 如果设置了 `HttpOnly`，前端 JS 就读不到，这对防止脚本窃取有帮助，但 cookie 自动携带又会带来 CSRF 风险，所以通常还要结合 `SameSite` 和服务端校验一起使用。

如果面试里总结，可以说：Storage 更适合前端本地状态存储，cookie 更适合客户端和服务端之间传递会话状态；选择时要同时考虑生命周期、容量、是否自动携带请求以及安全风险。

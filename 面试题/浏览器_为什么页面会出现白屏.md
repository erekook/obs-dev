# 题目
前端页面为什么会出现白屏？如果线上用户反馈“打开就是一片白”，你会优先怎么排查？

# 回答要点
- 白屏不一定只有一种原因，可能发生在网络、资源、执行、渲染多个阶段。
- 常见原因：
  - HTML / 入口资源请求失败
  - JS 包加载失败或路径错误
  - 运行时报错导致应用没挂起来
  - 首屏接口阻塞关键渲染
  - 渲染逻辑本身没有输出内容
  - CSS / 容器尺寸异常导致内容不可见
- 排查思路：
  - 看 Network 是否有资源 404 / 500 / 超时
  - 看 Console 是否有运行时报错
  - 检查静态资源路径、环境变量、部署配置
  - 看根节点是否成功挂载内容
  - 检查首屏是否被接口串行或异常状态卡住
- 面试里可补充：
  - 真白屏和“页面有结构但内容不可见”要区分
  - 监控里可结合资源错误、JS 报错、FCP/LCP、白屏探针一起判断

# 追问
- 为什么资源路径配置错误很容易导致线上白屏？
- 首屏接口为什么不应成为唯一阻塞条件？
- 白屏监控通常怎么做？
- 如何从“白屏”进一步定位是网络问题还是代码问题？

# 参考答案
页面出现白屏，说明用户在打开页面后，没有及时看到任何有效内容。但“白屏”本身只是现象，它背后可能卡在不同阶段：资源没到、代码没跑、应用没挂载、内容没渲染、或者内容其实渲染了但不可见。

排查时，我通常会先按链路拆开看。

第一步看 Network。确认 HTML、主 JS、CSS、图片、接口这些关键资源是否请求成功，有没有 404、500、超时、跨域失败或路径错误。很多线上白屏，其实根本不是业务逻辑问题，而是打包后的静态资源 URL 配错了，导致主脚本压根没加载到。

第二步看 Console。应用主包如果执行时报错，比如初始化异常、运行时 undefined、环境变量缺失，框架应用可能还没来得及挂载页面就中断了，这种也会表现成白屏。

第三步看 DOM 挂载情况。要判断是真正“什么都没渲染”，还是页面其实已经挂载了容器，只是因为样式、尺寸、条件判断、异常分支等原因，最终展示区域为空。

第四步看首屏逻辑设计。有些页面把首屏完全绑定在一个接口返回上，一旦接口慢或异常，就整页不出内容。这种不是“资源失败”，但同样会造成白屏体验。

所以白屏排查最重要的是先区分阶段：是资源没到、代码没跑、应用没挂、还是内容没显示。每种问题的解决办法都不一样。

面试里如果总结，可以说：白屏不是单点问题，而是前端加载链路任一关键环节失败后的共同表现；排查要先按资源加载、脚本执行、应用挂载和内容渲染这几个层次逐步缩小范围。

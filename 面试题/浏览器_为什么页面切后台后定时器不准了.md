# 题目
为什么浏览器页面切到后台后，`setTimeout` / `setInterval` 往往就不准了？这背后和浏览器的什么机制有关？

# 回答要点
- 定时器“不准”通常不是 bug，而是浏览器的节流与调度策略。
- 常见原因：
  - 后台标签页会被降频
  - 浏览器优先把资源留给前台活跃页面
  - 为了省电、省 CPU、提升整体性能
- 结果表现：
  - 回调触发延后
  - `setInterval` 间隔变长
  - 多次定时器可能被合并调度
- 关键理解：
  - 定时器本来就不是精确定时器
  - 页面后台后这种误差会更明显
- 面试里可补充：
  - 前端不要把普通浏览器定时器当实时系统用
  - 可结合 `visibilitychange`、时间差校正等方式处理

# 追问
- 为什么 `setTimeout(fn, 0)` 也不会立刻执行？
- 浏览器为什么要主动限制后台页面？
- 如果做倒计时，为什么不能只靠 `setInterval` 每秒减一？
- 后台页面恢复前台时，前端通常应该怎么补偿时间差？

# 参考答案
页面切到后台后，`setTimeout` 和 `setInterval` 变得不准，通常不是代码写错了，而是浏览器有意这么做的。

浏览器不会把每个标签页都当成同等优先级持续全速运行。当前台页面是用户正在交互的主要对象时，后台页面通常会被降频处理，包括定时器触发频率、动画帧执行、资源调度等，都会受到限制。

这样做的目的很现实：节省 CPU、降低能耗、减少电池消耗、提升整机和前台页面的体验。如果后台页面的定时器还像前台一样高频执行，那浏览器整体性能会受到很大影响。

所以当页面进入后台后，你会看到：原本设定每 1 秒触发一次的 `setInterval`，实际可能几秒才触发一次；多个定时器也可能被浏览器合并调度；`setTimeout` 也会明显延后。

另外要注意，定时器本来就不是精确定时器。即使在前台，`setTimeout(fn, 1000)` 也不代表 1000ms 一到就立刻执行，它只是“最早不早于这个时间进入可执行队列”。后台模式只是把这种不确定性放大了。

所以实际开发里，如果你在做倒计时、轮询、刷新策略这类依赖时间的逻辑，不应该简单假设“每次 interval 都准时触发”。更稳妥的做法是记录真实时间戳，用当前时间减起始时间来计算结果；页面切回前台时，也可以结合 `visibilitychange` 做一次校正。

面试里如果总结，可以说：后台页面定时器不准，是浏览器为了节能和性能对后台任务做的主动节流；前端应把定时器当成近似调度工具，而不是精确计时器。

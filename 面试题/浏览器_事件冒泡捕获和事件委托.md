# 题目
请你讲一下浏览器事件机制里的事件捕获、事件冒泡和事件委托。事件委托为什么常用于列表场景？

# 回答要点
- DOM 事件传播通常分三个阶段：
  - 捕获阶段
  - 目标阶段
  - 冒泡阶段
- 事件捕获：
  - 事件从 window/document 向目标元素逐层传递
- 事件冒泡：
  - 事件从目标元素再向外层祖先元素逐层冒泡
- 默认常见监听更多是在冒泡阶段触发
- 事件委托：
  - 不给每个子元素单独绑定事件
  - 而是把事件绑定到公共父元素上
  - 利用冒泡机制统一处理
- 事件委托优点：
  - 减少事件监听数量
  - 降低内存占用
  - 动态新增子元素也能响应
- 常见实现：
  - 结合 `event.target`
  - 配合 `matches` / `closest` 判断真实触发元素
- 面试里可补充：
  - 并非所有事件都适合委托，比如不冒泡事件需特殊处理

# 追问
- `addEventListener` 第三个参数控制的是什么？
- 为什么 `focus` 和 `blur` 做事件委托时要特别注意？
- `event.target` 和 `event.currentTarget` 的区别是什么？
- 事件委托在 React/Vue 这类框架里还常见吗？

# 参考答案
浏览器的事件机制里，一个事件触发后，通常会经历三个阶段：捕获阶段、目标阶段和冒泡阶段。

捕获阶段是事件从最外层开始，一级一级往目标元素方向传递；目标阶段是事件真正到达触发的那个元素；冒泡阶段则是事件从目标元素再一层层往外返回到父元素、祖先元素，直到最外层。

在原生开发里，我们平时最常见的事件监听，默认大多是在冒泡阶段触发的。也就是说，点击一个子元素后，父元素如果也监听了同类事件，通常也能收到这个事件。

事件委托就是利用这个机制。它的思路不是给每一个子元素都单独绑定事件，而是把事件统一绑定到它们共同的父元素上。当某个子元素触发事件后，事件冒泡到父元素，由父元素统一判断到底是哪个子元素触发了事件。

比如一个很长的列表，如果每个列表项都绑定一个点击事件，节点很多时会带来额外开销；而如果只给列表容器绑定一次点击事件，再通过 `event.target` 或 `closest` 判断用户点的是哪一项，就能减少监听器数量，而且后续动态新增的列表项也不需要重新绑定事件。

这也是为什么事件委托特别适合列表、表格、菜单这类重复元素很多的场景。

不过它也不是万能的。因为有些事件默认不冒泡，比如 `focus` 和 `blur`，这时候要么换成可冒泡的替代事件，比如 `focusin`、`focusout`，要么用捕获阶段处理。

总结来说，事件捕获是“从外到内”，事件冒泡是“从内到外”，事件委托则是利用冒泡机制，把对子元素的大量事件绑定收敛到父元素上统一处理。面试里把这三者关系讲顺，一般就比较完整了。

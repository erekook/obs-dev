# 题目
如果只看“从输入 URL 回车，到浏览器收到第一个字节响应（TTFB）”这一段，中间大致经历了哪些过程？

# 回答要点
- TTFB（Time To First Byte）关注的是首字节到达时间，不包含完整渲染全过程。
- 主要链路：
  - 解析 URL
  - 查缓存 / HSTS / DNS 缓存
  - DNS 解析
  - 建立连接（TCP 三次握手）
  - HTTPS 场景下 TLS 握手
  - 浏览器发送 HTTP 请求
  - 服务端接收请求并处理
  - 服务端返回响应的第一个字节
- 影响 TTFB 的常见因素：
  - DNS 慢
  - 握手耗时
  - 网络 RTT 高
  - 服务端处理慢
  - 反向代理 / CDN 回源耗时
- 面试里可补充：
  - TTFB 是网络与服务端共同作用的指标
  - 它不是前端页面渲染性能的全部，但很关键

# 追问
- TTFB 和 FCP、LCP 的区别是什么？
- CDN 为什么可能显著改善 TTFB？
- HTTPS 为什么通常比 HTTP 多一些握手开销？
- 如果 TTFB 高，如何判断是网络问题还是后端处理慢？

# 参考答案
从输入 URL 回车，到浏览器收到响应的第一个字节，这一段通常就是 TTFB 关注的范围。它的重点不是页面什么时候渲染出来，而是请求什么时候真正开始收到服务端返回的数据。

这个过程大致可以拆成几步。首先浏览器会解析 URL，确认协议、域名、端口、路径等信息。然后会先检查一些本地缓存，比如 DNS 缓存、连接复用、HSTS 规则等，如果命中了，这一部分时间可以缩短。

如果没有现成结果，就要做 DNS 解析，把域名转换成 IP 地址。拿到 IP 后，浏览器需要和服务器建立连接。如果是 HTTP，一般是 TCP 三次握手；如果是 HTTPS，还要继续做 TLS 握手和证书校验。

连接建立好之后，浏览器才会真正发出 HTTP 请求。服务端收到请求后，还要经过网关、负载均衡、应用服务、数据库或缓存等处理流程。最终服务端把响应开始返回，浏览器收到第一个字节，这时 TTFB 才算结束。

所以 TTFB 本质上是多个环节叠加的结果，包括 DNS、网络往返时间、连接建立时间以及服务端处理时间。它并不是单纯的“前端性能指标”，而是客户端网络链路和后端响应速度共同决定的。

如果面试里总结一句话，可以说：TTFB 关注的是“请求从发起到收到首字节”的时间，核心经过 URL 解析、DNS、建连、握手、请求发送和服务端处理几个阶段。

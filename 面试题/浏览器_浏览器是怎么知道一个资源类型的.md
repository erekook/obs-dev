# 题目
浏览器是怎么判断一个资源到底是 HTML、CSS、JS、图片还是视频的？它主要依据什么？

# 回答要点
- 浏览器不会只看文件名后缀。
- 常见判断依据：
  - HTTP 响应头里的 `Content-Type`
  - HTML 标签上下文（如 `<script>`、`<img>`、`<link>`）
  - 某些情况下的内容嗅探（sniffing）
- `Content-Type` 的作用：
  - 告诉浏览器资源的 MIME 类型
  - 比如 `text/html`、`text/css`、`application/javascript`、`image/png`
- 标签上下文的影响：
  - 同一个 URL 放在不同标签里，请求目的不同
  - 浏览器会按该资源用途去处理
- 面试里可补充：
  - 错误的 `Content-Type` 可能导致资源无法正确执行或解析
  - 安全上通常会配合 `X-Content-Type-Options: nosniff`

# 追问
- 为什么浏览器不能完全相信文件扩展名？
- `nosniff` 的作用是什么？
- 如果 JS 文件被返回成 `text/plain`，会发生什么？
- HTML 里 `<img>` 标签请求到一段 JS 代码，浏览器会怎么处理？

# 参考答案
浏览器判断资源类型，最核心的依据通常不是文件后缀，而是服务器返回的响应头，尤其是 `Content-Type`。

`Content-Type` 会告诉浏览器，这个响应体的 MIME 类型是什么。比如：

- `text/html` 通常表示 HTML
- `text/css` 表示样式表
- `application/javascript` 表示 JavaScript
- `image/png`、`image/jpeg` 表示图片
- `video/mp4` 表示视频

浏览器收到响应后，会根据这个类型决定后续怎么解析和处理资源。

不过浏览器也不是只看响应头。它还会结合“当前这个资源是通过什么标签或上下文请求的”来理解用途。比如一个通过 `<img>` 发起的请求，浏览器期望它最终是可显示的图片资源；一个通过 `<script>` 加载的资源，浏览器期望它能作为脚本执行。

如果服务端返回的类型和浏览器上下文预期不一致，就可能出问题。比如脚本资源被返回成错误的 `Content-Type`，浏览器可能拒绝执行，或者按普通文本处理。

在某些情况下，浏览器还会做内容嗅探，也就是根据实际内容猜测资源类型。但这既不稳定，也可能带来安全风险，所以现代安全实践通常会配合 `X-Content-Type-Options: nosniff`，明确告诉浏览器不要乱猜，只按声明类型处理。

所以面试里可以总结成一句话：浏览器判断资源类型，主要靠响应头里的 `Content-Type`，再结合请求上下文和少量内容嗅探逻辑；真正可靠的做法是服务端正确返回 MIME 类型，而不是只依赖文件后缀。

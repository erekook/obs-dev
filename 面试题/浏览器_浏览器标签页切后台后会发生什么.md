# 题目
浏览器标签页切到后台之后，前端页面通常会发生什么变化？为什么有些定时器、动画和请求行为会变得不一样？

# 回答要点
- 浏览器切后台后，会主动做资源调度优化。
- 常见变化：
  - `requestAnimationFrame` 降频或暂停
  - `setTimeout` / `setInterval` 可能被强制延迟
  - 页面渲染暂停
  - 某些音视频、网络、CPU 资源策略变化
- 原因：
  - 节省电量和 CPU
  - 减少后台标签占用资源
  - 提升前台页面体验
- 与前端相关的影响：
  - 动画时间轴不再稳定
  - 定时逻辑不能假设严格按毫秒执行
  - 后台恢复时可能出现“时间跳跃”
- 常见补充：
  - 可通过 `visibilitychange` 感知页面可见性变化
  - 某些业务需要在切后台/切前台时做状态同步
- 面试里可补充：
  - 浏览器行为存在差异，但总体思路一致：后台标签会被限流或降频

# 追问
- 为什么 `requestAnimationFrame` 在后台页通常会暂停？
- `visibilitychange` 常用来解决哪些问题？
- 如果页面依赖定时器轮询，切后台后会有什么坑？
- 前端如何在恢复前台时校正倒计时和动画状态？

# 参考答案
浏览器标签页切到后台后，页面并不是“完全停止”，但浏览器通常会主动对它做降频和限流处理，目的是节省资源，把更多 CPU、绘制和电量留给当前前台页面。

最常见的变化之一是动画相关行为。像 `requestAnimationFrame` 这类和页面重绘绑定的调度机制，在后台页通常会被暂停或大幅降频，因为后台页本来也不需要持续重绘。

定时器也会受到影响。`setTimeout` 和 `setInterval` 并不会保证后台页里还按照你写的毫秒数精确执行，浏览器往往会把它们延后、合并，甚至做更强的节流。这也是为什么某些倒计时、轮询逻辑在切后台再回来时，会突然“跳了一大截”。

除此之外，页面渲染相关任务通常也会暂停，因为用户根本看不到后台页。某些媒体播放、网络调度、功耗策略，也可能根据浏览器和系统不同而发生变化。

所以前端开发里，一个很重要的认知是：不要把后台标签页当作“正常实时运行环境”。如果业务依赖时间推进，比如倒计时、轮询、动画、心跳同步，就要考虑页面切后台再回来时如何校正状态。

常见做法是监听 `visibilitychange` 事件。当页面重新回到前台时，可以重新计算当前时间差、刷新关键数据、恢复动画状态，而不是盲目相信定时器一直在稳定运行。

面试里如果总结，可以说：浏览器后台标签页通常会被降频、限流甚至暂停部分任务，这是浏览器的资源优化策略；前端如果有动画、计时、轮询等逻辑，需要显式处理前后台切换带来的时序变化。

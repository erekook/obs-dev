# 题目
为什么浏览器里执行 JavaScript 时，常常会阻塞页面渲染或 HTML 解析？它背后的原因是什么？

# 回答要点
- 浏览器主线程通常要负责：
  - JS 执行
  - HTML 解析
  - 样式计算
  - 布局和绘制等工作
- JS 为什么会阻塞：
  - JS 和渲染共享主线程
  - 同一时刻主线程只能做一件事
- 普通 `script` 会阻塞 HTML 解析：
  - 浏览器担心脚本修改当前 DOM
  - 所以通常先停下解析，等脚本下载并执行完再继续
- 长时间运行的 JS 会阻塞渲染：
  - 主线程忙于执行脚本
  - 没空去布局、绘制、响应交互
- 面试里可补充：
  - `defer` / `async` 是对脚本加载执行时机的优化
  - “阻塞”本质是主线程调度竞争问题

# 追问
- 为什么浏览器不能一边执行 JS 一边安全地继续解析？
- CSS 为什么有时也会间接影响脚本执行时机？
- 长任务（long task）会给用户体验带来什么问题？
- 如何优化阻塞渲染的 JS？

# 参考答案
浏览器里 JavaScript 会阻塞页面渲染或 HTML 解析，核心原因是：浏览器主线程是共享的，而 JS 执行和页面渲染很多关键工作都依赖这条主线程。

浏览器主线程通常既要解析 HTML、计算样式、做布局和绘制，也要执行 JavaScript。因为同一时刻主线程只能做一件事，所以当它在忙着执行 JS 时，就没法同时去处理渲染更新。

对于普通 `script` 标签来说，浏览器在解析 HTML 时一旦遇到它，通常会先暂停 HTML 解析，去下载并执行脚本。为什么要这样做？因为脚本可能会直接修改当前 DOM 结构，比如插入新节点、删掉节点、改写文档内容。如果浏览器一边继续往下解析 HTML，一边又让脚本改 DOM，就会让当前文档状态变得非常难保证一致性。

所以默认策略是更保守的：先把脚本相关事情处理完，再继续解析后面的 HTML。这就是为什么普通脚本会阻塞解析。

另外，如果一段 JavaScript 执行时间很长，即使它不是在 HTML 解析阶段触发，也会阻塞后续渲染。因为浏览器要等这段脚本执行完，才能继续做布局、绘制和交互响应。用户此时就可能感受到页面卡顿、点击没反应、动画掉帧。

从本质上说，这种“阻塞”并不是浏览器故意为难前端，而是单线程主线程在多个关键任务之间做调度时的结果。

面试里如果总结，可以说：JavaScript 会阻塞页面，是因为 JS 执行和渲染解析共享主线程，而脚本又可能修改 DOM，所以浏览器默认会优先保证执行一致性，代价就是在某些时机会暂停解析和渲染。

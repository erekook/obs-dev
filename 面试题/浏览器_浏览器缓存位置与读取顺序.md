# 题目
请你讲一下浏览器缓存一般有哪些位置，比如内存缓存、磁盘缓存、Service Worker 缓存？它们大致的读取顺序和适用场景是什么？

# 回答要点
- 浏览器缓存不只有 HTTP 强缓存/协商缓存，还包括不同存储层级。
- 常见缓存位置：
  - Memory Cache（内存缓存）
  - Disk Cache（磁盘缓存）
  - Service Worker Cache / Cache Storage
  - Push Cache（了解即可，不同浏览器实现差异较大）
- 一般理解：
  - 内存缓存读取快，但生命周期短，页面关闭或进程回收后可能失效
  - 磁盘缓存持久性更强，但读取相对慢于内存
  - Service Worker 可由开发者主动控制缓存策略
- 缓存命中受多种因素影响：
  - 资源类型
  - 缓存头
  - 浏览器策略
  - 页面是否刷新
- 常见认知：
  - 内存缓存适合短期高频访问资源
  - 磁盘缓存适合跨页面、跨会话复用
  - Service Worker 适合离线能力、精细化缓存控制
- 面试里可以说明：
  - 实际读取顺序不是死规则，但通常会先看更快、更近的缓存层
  - 最终仍要结合缓存策略头和浏览器实现理解

# 追问
- 为什么强制刷新时很多缓存会失效？
- Service Worker 缓存和 HTTP 缓存有什么关系？
- 哪些资源更容易进内存缓存，哪些更可能落到磁盘？
- `from memory cache` 和 `from disk cache` 在 DevTools 里分别说明了什么？

# 参考答案
浏览器缓存可以从两个角度理解。一个角度是 HTTP 层的强缓存、协商缓存；另一个角度是资源到底被放在了什么位置，比如内存、磁盘，或者由 Service Worker 控制的缓存空间。

常见的缓存位置里，最容易被提到的是内存缓存和磁盘缓存。内存缓存读取速度很快，适合当前页面或短时间内重复访问的资源，但它通常不够持久，浏览器标签页关闭、进程回收或者内存压力变化后，缓存可能就没了。磁盘缓存则更持久一些，浏览器重启后仍可能命中，适合跨页面、跨会话复用资源，但读取速度通常慢于内存缓存。

除了浏览器自动管理的缓存，前端还可以通过 Service Worker 使用 Cache Storage 主动缓存资源。它和普通 HTTP 缓存不同，开发者可以自己决定缓存哪些请求、什么时候更新、离线时返回什么内容，所以常用于离线页面、PWA 和更可控的资源缓存策略。

至于读取顺序，面试里不要把它说成绝对固定，因为不同浏览器实现会有差异。但一般可以这样理解：浏览器会优先检查更近、更快的缓存层，比如内存缓存；如果没有命中，再看磁盘缓存；如果有 Service Worker 参与，请求还可能先被它拦截，再决定是走缓存还是网络。

另外要注意，资源能不能命中缓存，不只是看它存在哪，还要看缓存头和请求方式。比如用户普通刷新、强制刷新、禁用缓存调试，结果都可能不同。DevTools 里常看到的 `from memory cache`、`from disk cache`，本质上就是告诉你资源是从哪一层缓存里拿到的。

总结来说，内存缓存更快但更短命，磁盘缓存更持久，Service Worker 缓存最灵活。面试时如果能把“缓存策略”和“缓存位置”这两个维度分开讲，会更清楚。
